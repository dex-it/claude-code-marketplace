# Специализация агентов: общий vs специальный под стек

Правило, по которому решается, нужен ли отдельный агент под конкретный стек
(.NET / TS / Python / …) или достаточно одного языко-агностичного агента с
условной загрузкой skills под стек.

> Это решение о **дублировании vs покрытии**. Плодить агента на каждый язык там,
> где процесс одинаков — дублирование. Не дать специального там, где исполнение
> идиоматично по стеку — потеря качества. Правило ниже разводит эти два случая.

## Критерий

**Специальный агент под стек оправдан тогда и только тогда, когда стек-специфика
живёт В ТЕЛЕ АРТЕФАКТА, который агент производит, а не в условной загрузке skills.**

Спроси про роль один вопрос:

> Меняет ли стек **то, что агент выдаёт** (содержание артефакта), или только
> **какие чек-листы он сверяет** (знание ловушек)?

- **Меняет содержание выхода** → стек в теле артефакта → **специальный оправдан**.
  Выход на C# и на TS — это *разный артефакт*, не один и тот же с разным набором
  проверок. Идиомы языка / библиотеки экосистемы — часть результата.
- **Меняет только набор проверок** → стек в загрузке skills → **общий достаточен**.
  Процесс и форма выхода стек-нейтральны; различия выносятся в условную загрузку
  skills (`Если <стек> → грузи X`). Специальный агент здесь скопировал бы процесс
  и обернул skill-блок в `если <стек>` — это дубль, а не специализация.

## Признак дубля (red flag)

Специальный агент — замаскированный дубль общего, если при сравнении видно:

- фазы / scan recipes / severity-таблица / output-формат **скопированы** дословно;
- единственное отличие — блок загрузки skills обёрнут в `Если <стек>`;
- выход обоих агентов на одном баге/ревью **идентичен** по форме.

Тогда правильная форма — **один общий агент**, стек-различия в условных skills.

## Признак оправданной специализации (green flag)

Специальный агент несёт реальную ценность, если:

- фазы **наполнены** стек-специфичным содержанием (не «сверь по skill», а
  конкретные решения/идиомы в теле фазы);
- выход выражен **в терминах экосистемы** (конкретные библиотеки, идиомы, API);
- пара общий↔специальный поддерживается **дисциплинированно**: Sync note о
  намеренно идентичной структуре + двусторонняя маршрутизация в Boundaries
  («если не мой стек → делегируй парному агенту»).

## Применение критерия к ролям (разобрано на существующих агентах)

| Роль | Стек в теле артефакта? | Вердикт | Пример |
|------|------------------------|---------|--------|
| **coder** | да — идиомы языка в коде | специальный | dex-dotnet-coder, dex-ts-fullstack-coder |
| **tester** | да — API моков/раннера (Moq/xUnit vs vi.mock/Vitest) | специальный | dex-dotnet-tester, dex-ts-tester |
| **performance** | да — конкретика диагностики (GC/alloc/ValueTask) | специальный | dex-dotnet-performance |
| **architect** | да — alternatives в терминах библиотек экосистемы | специальный | dex-architect / dex-architect-dotnet |
| **reviewer** | нет — процесс ревью нейтрален, .NET в skills | общий | mr-reviewer / self-reviewer |
| **debugger** | нет — root-cause процесс нейтрален, .NET в skills | общий | dex-debugger |
| **security** | нет — категории OWASP нейтральны, .NET в skills | общий (+ усиления) | dex-security-reviewer |
| **researcher** | нет — сбор контекста нейтрален | общий | general-purpose / Explore |

### Почему architect — специальный, а debugger — общий (ключевой контраст)

- `architect-dotnet` **наполняет** Phase 4/6: «Modular monolith → MediatR»,
  «Microservices → MassTransit + RabbitMQ», EF Core Fluent API, Polly через
  `IHttpClientFactory`. Стек — в **теле** каждого предложения архитектуры.
  → специализация оправдана.
- общий `debugger` и .NET `bug-hunter` имеют **идентичные** фазы/scan/severity/
  output; .NET-агент лишь оборачивает тот же skill-блок в `если .NET`. Выход на
  .NET-баге идентичен. → дубль, нужен один общий.

## Граница применимости: «оправдан» ≠ «создавать сейчас»

Специальный агент оправдан критерием, но создавать его стоит, только если есть
**чем наполнить тело** — достаточная глубина стек-специфичных skills под роль.
Если skills под стек тонкие, даже «оправданный по критерию» специальный агент
выродится в дубль общего. До набора критмассы — общий агент + те skills, что есть.

> Пример: JS/TS-архитектор оправдан тем же критерием, что architect-dotnet, но под
> него пока мало архитектурных JS-skills → его не создают, JS-фичу ведёт общий
> dex-architect. Это отложенная специализация, не отказ.
