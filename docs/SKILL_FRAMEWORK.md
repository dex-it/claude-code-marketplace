# Skill Framework

Методичка для создания skills в Claude Code marketplace. Используется человеком или Claude при работе над плагинами - в runtime при обычной работе skill не загружается.

Параллель к [AGENT_FRAMEWORK.md](AGENT_FRAMEWORK.md): там - как собирать агентов из фаз, здесь - как писать skills, которые эти агенты загружают.

**Skill** - markdown-файл с frontmatter в `plugins/skills/dex-skill-X/skills/X/SKILL.md`. **Ловушка** - неочевидное поведение или типичная ошибка; основная единица содержания trap-skill, формат «Плохо / Правильно / Почему». **Активация** - semantic matching: Claude Code сопоставляет контекст разговора с `description` каждого skill и подгружает те, чьё описание совпало по смыслу.

## Два типа skill

Тип skill - **валидируемая ось**, не косметика. От типа зависит, как `tools/validate-skill.js` проверяет файл.

| | **Trap-skill** (ловушка) | **Process-skill** (оркестрация) |
|---|---|---|
| Отвечает на | «какие грабли в области X» | «как действовать / что грузить / в каком порядке» |
| Единица содержания | ловушка: Плохо / Правильно / Почему | правило, развилка решения, реестр |
| Структура тела | H2-категории, >= 5 H3-ловушек (цель 10-15) | таблица-реестр или >= 2 H2-раздела с правилами |
| Триада обязательна | да (каждая H3) | нет - разрешена, но не требуется |
| Регистрация | дефолт, нигде явно | allowlist `PROCESS_SKILLS` в валидаторе + `<!-- skill-type: process -->` в теле |
| Примеры | `dotnet-ef-core`, `owasp-security`, `react`, `docker` | `stack-registry`, `dotnet-project-baseline` |

**Trap-skill** - дефолт. Каталог граблей конкретной технологии или темы. Этому посвящён весь остальной документ, если не сказано иное.

**Process-skill** - тонкое правило-оркестрация: реестр («стек -> какие skills грузить»), развилка («новый solution с нуля vs добавка в существующий»). Не каталог API-граблей, поэтому триады не обязан содержать.

### Калибровка валидатора по типу

| Проверка | trap-skill | process-skill |
|---|---|---|
| `too-few-traps` (< 5 H3) | error | **N/A** - не применяется |
| `trap-missing-triad` (H3 без триады) | error | **N/A** - H3 в process это развилка/правило, не ловушка |
| `process-empty` (нужна таблица или >= 2 H2) | - | **error**, если не выполнено - защита от пустого process |
| keyword-минимум (10) / наличие «Активируется при» | error | **warning** - грузится детерминированно (pre-load/по имени), авто-активация вторична |
| размер | > 250 строк error | > 500 строк error (костяк process не дробится по размеру) |
| description-лимиты, code-fence, doc-titles | как все | как все |

Process-skill освобождён от trap-эвристик, но не от содержания: вместо них действует `process-empty` - нужна либо таблица-реестр, либо >= 2 H2-разделов с правилами. Иначе это не process-skill, а заглушка, прикрытая исключением.

**Почему активаторы у process-skill - warning, не error:** process-skill грузится **детерминированно**, не по семантическому матчу `description`. Два пути, оба не семантические: безусловный (node-contract узлу) - **pre-load через `skills:`** во frontmatter (см. AGENT_FRAMEWORK «Подключение skills»); условный (completeness-mapping «при изменении контракта») - **Skill tool по имени** `{plugin}:{skill}` в фазе. Семантическая авто-активация для process-skill вторичный путь (для pre-load она вообще не задействуется), поэтому жёсткий порог в 10 активаторов неуместен. Полностью снять активаторы нельзя: для случаев, где skill всё же может авто-активироваться, `description` остаётся рычагом; поле, отключающее ТОЛЬКО авто-активацию, - `disable-model-invocation`, но при нём skill **нельзя pre-load** (сверено по code.claude.com: pre-load черпает из того же набора, что Claude может вызвать) - ставить его на pre-loaded node-contract нельзя. Поэтому `description` сохраняется, смягчается лишь уровень (error -> warning). Trap/leaf-skill порог держит на error: для них авто-активация - основной путь.

### Регистрация process-skill

Источник истины - **allowlist `PROCESS_SKILLS`** по имени в `tools/validate-skill.js`. Добавить process-skill = осознанная правка валидатора с ревью; самодекларацией в теле тип не получить (иначе послаблением можно протащить недоделанный skill). Маркер `<!-- skill-type: process -->` в теле - **для читателя**, валидатор его не парсит.

### Process-skill != агент

Граница тонкая: process-skill говорит «как действовать», и агент через фазы - тоже. Разница механическая и содержательная.

| | Process-skill | Агент (specialist) |
|---|---|---|
| Физически | текст, попадает в чужой контекст | отдельный субагент со своим контекстом |
| Свой контекст / tools | нет | да (`tools:` в frontmatter) |
| Грузит skills / спавнит агентов | нет | да |
| Отвечает на | «в этой точке поступай так» | «доведи задачу от и до» |

**Критерий:** «это могло бы быть одной фазой внутри агента?» -> process-skill (фаза-знание, вкладывается в носителя). «Это набор фаз с гейтами от старта до коммита, со своей ролью и инструментами?» -> агент.

> ⚠️ Если «процесс» разрастается в полный жизненный цикл задачи (декомпозиция -> дизайн -> правка -> верификация -> коммит) с делегированием и самопроверкой - это **агент**, а не process-skill. Признак: файл переваливает за 250 строк и описывает роль и фазы. В superpowers такие движки оформляют как `SKILL.md` (`autonomous-task` - 529 строк), потому что там нет отдельного уровня specialist. У нас есть - см. [AGENT_FRAMEWORK.md](AGENT_FRAMEWORK.md). Исключение - главный цикл-оркестратор (`autonomous-task`): он держит канал к юзеру и сам спавнит агентов-узлы, субагентом-специалистом быть не может, поэтому оформляется process-skill'ом несмотря на размер и роль (потолок размера для него - 500, см. «Размер skill»).

## Ключевой принцип: skill - это ловушки, не документация

Claude **знает** как работает EF Core, как выглядит PyTorch training loop, какой синтаксис у Dockerfile. Всё это в training data. Объяснять базовые концепции или синтаксис в skill - дублировать доступное и тратить контекст.

Skill нужен для того, что Claude **не знает или часто забывает**:

- **Ловушки**, срабатывающие незаметно (N+1 без ошибки, только медленно)
- **Неочевидное поведение** API против интуиции (`AddAsync` делает лишний roundtrip в БД)
- **Стандартные ошибки** из проекта в проект (captive dependency, async void)
- **Trade-off'ы**, где выбор зависит от контекста (`AsSplitQuery` эффективен для списка, но не для single entity)
- **Критические правила**, нарушение которых = баг в проде (soft-delete + cascade FK = потеря данных)

Та же логика, что в [AGENT_FRAMEWORK.md](AGENT_FRAMEWORK.md): агент задаёт цель, Claude решает как; skill указывает грабли, Claude знает как их обойти.

### Документация vs ловушка

**Документация (нельзя):** раздел «Как настроить DbContext» с полным классом `AppDbContext`, `UseSqlServer`, `AddDbContext` - первая страница официальных docs, 30 строк, 0 новой информации.

**Ловушка (нужно):**

```markdown
### DbContext как Singleton

Плохо: `services.AddSingleton<AppDbContext>()` или DbContext в статическом поле
Правильно: `AddDbContext<AppDbContext>()` (Scoped по умолчанию)
Почему: Change Tracker растёт бесконечно (memory leak), stale данные, DbContext не thread-safe - concurrent access = random exceptions
```

4 строки. Claude получает то, что мог забыть: конкретный failure mode + правильный вариант + причинно-следственная связь.

## Структура skill

### Путь в репозитории

```text
plugins/skills/dex-skill-<name>/
├── .claude-plugin/
│   └── plugin.json
└── skills/
    └── <name>/
        └── SKILL.md
```

Имя плагина: `dex-skill-<name>`. Имя skill внутри: `<name>`. Ссылка для императивной загрузки из агента: `dex-skill-<name>:<name>`.

**Конвенция стекового префикса.** Для **стек-специфичных** skills `<name>` = `<стек>-<тема>`: `dotnet-ef-core`, `ts-patterns`, `python-pytorch`. Стек в префиксе - не косметика: это **единственный признак**, по которому общий агент группирует skills одного стека при by-stack loading (метаданных-фильтра по стеку в Claude Code нет). Все skills одного стека обязаны иметь единый префикс `dex-skill-<стек>-*`, иначе фильтрация по стеку даст ложно-пустой результат. Реестр стек->префикс и правило отбора «стек × тема» - в process-skill `dex-skill-stack-registry`. **Язык-нейтральные** skills (docker, kafka, solid, owasp-security, ddd) префикса стека не несут - грузятся по теме.

### Frontmatter

```yaml
---
name: <name>
description: <краткое назначение>. Активируется при keyword1, keyword2, ...
---
```

Опциональные поля (по [официальной документации](https://code.claude.com/docs/en/skills)): `disable-model-invocation`, `user-invocable`, `argument-hint`, `allowed-tools`, `model`, `effort`, `context`, `agent`, `hooks`, `paths`, `shell`. Любые поля вне этого списка - запрещены (валидатор упадёт).

### Тело skill

Trap-skill: markdown с ловушками по категориям. Каждая категория - H2, каждая ловушка - H3 с триадой «Плохо / Правильно / Почему». Process-skill: таблица-реестр или H2-разделы с правилами (триады необязательны).

## Правила для description (семантическая активация)

Description - **единственный механизм** автоматической активации. Плохой description = skill не активируется, его содержание не работает.

### Аспект vs точка входа

Description - это **название технологии + ключевые аспекты**, не список ловушек и не перечень API.

- **Аспект** ~ раздел верхнего уровня в SKILL.md: тема, область, паттерн (`## Aggregate`, `## Mapping`, `## Concurrency`). Подцепляет ЛЮБОЙ запрос внутри своей темы.
- **Точка входа** != аспект. Конкретные API / симптомы / термины внутри одной ловушки (`OwnsOne`, `ExecuteDelete`, `cache miss`). Подцепляет только буквальное совпадение и забивает шумом релевантность скилла. В description им не место.

### Формат

```text
description: <Технология> - <ключевые аспекты в 1 предложении>. Активируется при <тех-якорь>, <аспект 1>, <аспект 2>, ...
```

- **Только аспекты** в активаторах - каждый соответствует H2-разделу или законченной теме. Активатор под одну H3-ловушку - точка входа, удалить.
- **Якорь технологии обязателен** - `ef core`, `DDD`, `Polly`, `csproj`, `RabbitMQ`.
- **Не сужать тему до одного инструмента, если skill шире** - для язык-нейтральных тем якорь по теме, не по вендору: «Logging», а не «Serilog logging» (применимо к NLog, winston, log4j). Для стек-специфичных skills якорь стека, наоборот, обязателен - см. «Конвенция стекового префикса».
- **Без лишних слов** - «Активируется при», без «при упоминании» / «когда контекст содержит».

### Лимиты (валидатор)

| Условие | Реакция валидатора |
|---|---|
| длина > 500 символов | warning (PR не блокируется) - компактный description триггерится надёжнее |
| длина > 750 символов | error (проектный потолок) |
| длина > 1536 символов | error - хард-лимит Claude Code на `description` + `when_to_use`; свыше текст обрезается из listing |
| < 10 keywords после «Активируется при» | error (trap/leaf); **warning** для process-skill - см. «Калибровка валидатора по типу» |

При нехватке места режь **точки входа**, аспекты остаются.

### Когда менять description при добавлении ловушки

| Ловушка вводит... | Действие |
|---|---|
| **Новый аспект** (раздел H2 или новая тема) | добавить аспект в активаторы |
| Частный кейс уже покрытого аспекта | **не трогать** |
| Точку входа (конкретный API / симптом) в уже покрытой теме | **не трогать** |

Перед bump'ом спроси: «эта ловушка про новую тему или про уже описанную?». Про уже описанную - description остаётся.

### Хороший пример

```text
description: Entity Framework Core - ловушки запросов, миграций, concurrency, mapping.
Активируется при ef core, dbcontext, migration, N+1, AsNoTracking, Include,
AsSplitQuery, IQueryable, ExecuteUpdate, cartesian explosion, GroupBy, owned types
```

Якорь (`ef core`) + аспекты-разделы. Каждый активатор - область, не точка входа.

### Плохие примеры

- `EF Core skill` - нет аспектов, semantic matching не сработает.
- `... N+1 lazy load, AsNoTracking forgot, ToList instead of projection, ExecuteDelete batch, OwnsOne, OwnsMany, Add vs AddAsync, soft-delete cascade ...` - список ловушек/точек входа. Раздут, ломает лимит при росте, релевантность размыта.
- `... ubiquitous language, persisted поле, вычислимое поле, денормализация` - три формы одной идеи (один аспект `persisted state`). Правильно - один аспект.

### Анти-паттерн: «вернуть в description, потому что есть в теле»

Соблазн: «ловушка популярная, верну активаторы». Проверка: это аспект (раздел/тема) или точка входа (API внутри одной ловушки)?

| Кандидат | Класс | Возвращать? |
|---|---|---|
| `cartesian explosion` (раздел `## Split Queries`) | аспект | да |
| `specification` (паттерн, раздел) | аспект | да |
| `LINQ to Entities` (термин внутри ловушки) | точка входа | нет |
| `ExecuteDelete` (один API, одна ловушка) | точка входа | нет |

## Правила для ловушек

### Формат

```markdown
### <Название ловушки>

Плохо: `<короткий код или описание>`
Правильно: `<короткий код или описание>`
Почему: <объяснение в 1-2 предложения>
```

### Принцип «указываем путь, не прокладываем дорогу»

В строке «Правильно» - **имя API, условие, флаг (pointer)**, а не развёрнутый snippet.

**Плохо (дорога):** 10 строк `var orders = await context.Orders.Include(...).Where(...).OrderByDescending(...).Take(50).ToListAsync();` - boilerplate, Claude сам напишет идеально.

**Правильно (путь):** `Include(o => o.Customer)` или projection `.Select(...)`. Одна строка - указатель на нужный инструмент.

Жёсткий лимит валидатора - 12 строк на code-fence, но это **потолок**, не цель: типичная ловушка - 1-3 строки на «Плохо» и «Правильно».

### Обязательная часть «Почему»

Без «почему» ловушка - запрет без обоснования, карго-культ. Отвечает на два вопроса: **что именно происходит плохого** при «плохом» варианте и **как альтернатива это решает**.

```markdown
### AsNoTracking забыт для read-only

Плохо: `context.Products.Where(p => p.IsActive).ToListAsync()` - все entities в Change Tracker
Правильно: `.AsNoTracking()` для данных, которые не будут изменяться
Почему: Change Tracker хранит копию каждой entity + сравнивает при DetectChanges. На 10000 записей - ощутимый overhead
```

«Почему» объясняет механизм (копия + DetectChanges) и где проявляется (10000 записей).

### Категории

Ловушки группируются в H2-категории по общему признаку (область API, тип проблемы, этап работы). Ориентир - 5-10 категорий (валидатор число не проверяет): меньше - плоско, труднее навигация; больше - дробление, skill становится каталогом мелочей.

## Размер skill

- **Цель** 80-120 строк - ориентир автора, валидатор не проверяет.
- **> 250 строк** - error валидатора (проектный потолок для trap/leaf-skill).
- **Process-skill**: потолок поднят до 500 - движок/правило-оркестрация не дробится на каталог по размеру; деталь по требованию выносится в смежные файлы (mandatory-Read), не режется костяк.
- **> 500 строк** - error валидатора (хард-лимит Claude Code; для process-skill это и проектный потолок).

Каждая ловушка: 3-5 строк, 10-15 ловушек достаточно. Skill > 250 строк - сигнал, что либо смешаны несколько подтем (разделить на два skill), либо есть процедурные куски / документация (вырезать).

## Дублирование между skills

Skill **не может загрузить другой skill** - API нет (`skill -> skill` ✗). Ссылка `> см. dex-skill-X` в теле - только сноска для человека.

Если две ловушки из разных skills говорят об одном:

1. **Вынести общее** в более базовый skill (LINQ-ловушки из `dotnet-ef-core` -> в `dotnet-linq-optimization`).
2. **Оставить в одном месте**, из другого - сноска:
   ```markdown
   > Общие LINQ ловушки (Count vs Any, фильтрация) - см. `dex-skill-dotnet-linq-optimization`
   ```
3. **Не копировать** - разные копии расходятся со временем.

Skill не знает, какой агент его загрузит, не ссылается на агентов, не содержит фаз. Граница skill<->агент - в [AGENT_FRAMEWORK.md](AGENT_FRAMEWORK.md#агент-и-skills).

## Библиотека типовых категорий ловушек

Референс для структуры нового trap-skill. Не исчерпывающе.

- **Performance** - N+1, материализация коллекций, blocking calls в async, лишние roundtrip'ы, чтение всех колонок вместо проекции, отсутствие пагинации.
- **Concurrency** - race conditions, deadlock, lost updates, thread-unsafe shared state, missing locks, async void, fire-and-forget.
- **Resource** - утечки памяти, unbounded growth коллекций, незакрытые handles/connections, captive dependency, singleton-with-state, event handlers без отписки.
- **Security** - injection (SQL/XSS/command), auth bypass, IDOR, hardcoded secrets, CSRF, weak crypto, missing validation на границах.
- **Correctness** - граничные случаи, null handling, off-by-one, integer overflow, timezone, float-сравнения, equality checks, silent failures.
- **API usage** - deprecated methods, неправильные флаги, несовместимые версии, метод X выглядит как Y но работает иначе, скрытые side effects.
- **Workflow** - неправильная последовательность шагов, пропуск обязательных фаз, commit без review, deploy без verify, migration без backup.
- **Testing** - тесты зависят друг от друга, mocking реальных зависимостей, shared state, flaky от времени/сети, тестирование деталей реализации вместо поведения.
- **Build artifacts** - явная `PackageReference` поверх транзитивной, дубли версий при CPM, нет `PrivateAssets="all"` для analyzers, `ProjectReference` без использования типов. Стек-агностично: `package.json`+lockfile, `pyproject.toml`, `go.mod`, `Cargo.toml`.
- **Resilience** - resilience-policy на листовом клиенте вместо корневого flaky-источника, retry на неидемпотентных операциях без `Idempotency-Key`, только per-attempt timeout без total, нет circuit breaker, silent swallow в fallback. Polly / retry-axios / tenacity / resilience4j.
- **Domain naming** - один термин = разные сущности в одном bounded context, generic-имена (`Id`, `Name`, `Value`) без префикса контекста, заимствование имён из внешних API без переосмысления, доменно-путающий идентификатор (имя одной сущности, тип другой).

## Рецепты trap-skill по доменам

Типовые скелеты под частые задачи. Это **рецепты**, не жёсткие контракты. (Process-skill сюда не входит - это отдельный тип, см. «Два типа skill».)

- **Framework skill** - ловушки инструмента (EF Core, PyTorch, React, Docker). Категории: основные API + неочевидное поведение, lifecycle/scoping, performance traps, версионные отличия, интеграция с экосистемой.
- **Pattern skill** - ловушки паттерна без привязки к инструменту (async/await, LINQ, logging, observability). Категории: типичные неправильные применения, пересечение с другими паттернами, performance/correctness trade-off'ы.
- **Security skill** - категория уязвимостей (`owasp-security`). Категории: классы уязвимостей с примерами атак, как код становится уязвимым, trade-off безопасность<->удобство.
- **Workflow skill** - процессные грабли (git-workflow, code review, release). Категории: этапы и их нарушения, coordination/handoff проблемы, что легко пропустить или сделать не в том порядке.

## Анти-паттерны

1. **Документация API вместо ловушек.** «EF Core - это ORM. Основные классы: DbContext, DbSet<T>...» - первая страница docs. Claude знает.
2. **Развёрнутый код вместо pointer'а.** Класс/метод/try-catch целиком. Claude сам напишет boilerplate под контекст. Лимит валидатора 12 строк - потолок, не цель.
3. **Объяснения базовых концепций.** «async/await - синтаксический сахар, компилятор генерирует state machine...» - концепция, не ловушка.
4. **Шаги туториала.** «Шаг 1: установите пакет. Шаг 2: создайте DbContext...» - getting started, есть в docs.
5. **Дублирование с другим skill.** Одна и та же ловушка в двух местах - удалить одну, оставить ссылку.
6. **Привязка к одному стеку, когда skill универсальный.** «Logging с Serilog» с log levels - они применимы к NLog, winston, pino, log4j. Skill называется «logging», не «serilog».
7. **Description без ключевых слов.** `Skill про базы данных` - semantic activation не сработает. Нужны якорь + аспекты.
8. **Слишком большой skill.** 400 строк - смешаны подтемы или есть документация. Разделить или сократить.
9. **Ловушка без «почему».** «Плохо: X. Правильно: Y» без объяснения - карго-культ, разработчик не поймёт границ применимости.

### 10. Конкретика одного проекта вместо обобщённого правила

**Плохо:** ловушка `ProjectId vs ProjectVcsId` с обоснованием «в UpCore Project - бизнес-сущность, а репозиторий в GitLab - ProjectVcs». У других команд другие сущности, правило для них мёртвый вес.

**Правильно - обобщить:**

```markdown
### Коллизия доменного термина

Плохо: один термин обозначает разные сущности в одном bounded context
Правильно: каждая сущность получает уникальное имя с контекстным префиксом
Почему: смешение одноимённых сущностей ведёт к тихим багам на стыках слоёв - читаешь имя, предполагаешь одну сущность, в рантайме работает другая
```

Проверка: **убери имена проекта - правило всё ещё читается?** Если убрать разрушает ловушку - она про конкретный проект. Конкретика допустима как короткая иллюстрация после обобщённой формулировки, не как сама ловушка.

### 11. Непроверенное правило про поведение библиотеки

**Плохо (и звучит правдоподобно):** «`IsInEnum()` нельзя для `[Flags]`-enum - отвергает легальные комбинации». В актуальной FluentValidation (8.0+) `IsInEnum()` как раз корректно обрабатывает `[Flags]` через `IsFlagsEnumDefined`. Совет основан на «общеизвестном», а не на исходнике - и **слабее** того, что критикует. Такая ловушка опаснее отсутствия: разработчик ей доверяет и ухудшает код.

То же с дефолтами: «System.Text.Json пропустит `NaN`» неверно - дефолтный `JsonNumberHandling.Strict` режет это на десериализации.

Проверка: **правило завязано на поведение API / дефолт фреймворка? Сверься с текущей версией (исходник, тесты, доки).** Особенно когда звучит как «общеизвестное» - оно часто устарело на мажорные версии. Зависит от версии/настройки - указать это в ловушке («при Newtonsoft / при `AllowNamedFloatingPointLiterals`»), а не давать как абсолют.
