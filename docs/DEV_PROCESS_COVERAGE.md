# Покрытие процесса разработки агентами

Карта «слот процесса разработки -> агент, который его закрывает». Процесс
стек-нейтральный; для слотов со специальными агентами показаны варианты под стек.

> Назначение: при автономной или ручной разработке видно, какой профильный
> исполнитель отвечает за каждый шаг. Подбор исполнителя - по типу работы + стеку
> (см. [AGENT_SPECIALIZATION.md](AGENT_SPECIALIZATION.md) - общий vs специальный).

## Треки автономного движка -> исполнители

Автономный движок (`autonomous-task`) ведёт цель по одному из треков; трек
определяется по цели. Ниже - какой профильный агент маркетплейса закрывает каждый
трек. Детальная слот-карта дальше относится к треку **«Разработка»** (самый
составной); остальные треки - это в основном один-два агента.

| Трек движка | Исполнитель (.NET / общий) | Артефакт |
|-------------|----------------------------|----------|
| **Разработка** (фича / баг-фикс / рефакторинг) | детальная слот-карта ниже | MR/коммиты |
| **Ревью входящего MR** | dex-mr-reviewer -> dex-mr-check-reviewer | inline-треды |
| **Приёмка слитой фичи на стенде** (post-merge) | dex-stand-reviewer -> dex-bug-fixer | отчёт-приёмка + follow-up коммиты |
| **Аналитика/ресёрч** | general-purpose / Explore (+ deep-research) | бриф/доклад |
| **Тест-инжиниринг** | dex-dotnet-tester / dex-ts-tester | тесты + отчёт о покрытии |
| **Диагностика/инцидент** | dex-dotnet-runtime-diagnostician (+ debug/perf/tracing skills, netcoredbg-cli) | root-cause-отчёт |
| **Документирование** | dex-doc-writer (+ dex-adr-writer / dex-diagram-creator / dex-api-designer по жанру) | нормативный документ |
| **Арх-дизайн** (самоцель) | dex-architect-dotnet / dex-architect | design-doc / ADR / диаграммы |

Под-виды «Разработки» используют ту же слот-карту: **баг-фикс** активирует слот
**Debug**, **рефакторинг** идёт через **Код** с инвариантом «поведение не меняется».
**Диагностика** по живому процессу/дампу - отдельный трек (объект - запущенная
система, не исходник); правка найденного бага возвращается в «Разработку».
**Арх-дизайн как самоцель** - отдельный трек; как **шаг** проектирования перед кодом
он остаётся слотом «План/спека» детальной карты ниже.

## Процесс разработки (слоты)

```
Контекст -> План -> [TDD-тесты -> Код] -> Прогон+Саморевью -> Security -> [Debug | Perf] -> Сдача
```

## Карта слот -> агент (трек «Разработка»)

| Слот | Что делается | .NET | TS/JS | Общий fallback |
|------|--------------|------|-------|----------------|
| **Контекст** | прицельный сбор: код вокруг фичи, контракты, потребители | - | - | general-purpose / Explore |
| **План / спека** | дизайн нетривиальной фичи (требования -> capacity -> alternatives -> plan) | dex-architect-dotnet | dex-architect¹ | dex-architect |
| └ ADR | если решение значимо | - | - | dex-adr-writer |
| └ диаграмма | если нужна C4/sequence | - | - | dex-diagram-creator |
| └ публичный API | если проектируется контракт | - | - | dex-api-designer |
| **TDD-тесты** | тесты ожидаемого поведения до кода | dex-dotnet-tester | dex-ts-tester | - (идиоматично, fallback слабый) |
| **Код** | реализация в стиле окружения | dex-dotnet-coder | dex-ts-fullstack-coder | - |
| **Прогон + саморевью** | build/test/lint зелены + ревью своего кода (вкл. арх) | self-reviewer² | self-reviewer² | self-reviewer |
| **Security** | отдельный обязательный security-проход | dex-security-reviewer³ | dex-security-reviewer³ | dex-security-reviewer |
| **Debug** | root-cause бага по коду (при красном прогоне) | dex-debugger⁴ | dex-debugger⁴ | dex-debugger |
| **Perf** | оптимизация (N+1, alloc, hot path) - по нужде | dex-dotnet-performance | - ⁵ | - |
| **Сдача** | коммит/MR по конвенции, трекер | - | - | сам (git) |

### Сноски

1. **JS/TS-архитектора нет** - JS-фичу ведёт общий `dex-architect` + условные
   JS-skills. Специализация отложена (мало архитектурных JS-skills), см.
   AGENT_SPECIALIZATION.md.
2. **self-reviewer закрывает сразу три слота**: прогон (Local verification -
   реально гоняет build/test/lint), ревью кода и архитектурное ревью (7 фокусов:
   security/architecture/language/business/regressions/loose-ends/local-verification).
   Языко-агностичный. Выделенный арх-ревьюер не нужен.
3. **Security - отдельный проход**, не растворённый в общем ревью: даже если общее
   ревью «чисто», security проходится отдельно. `dex-security-reviewer` -
   языко-агностичный агент глубокого анализа (threat model -> attack paths ->
   цепочки эксплойтов), усиливается профильными skills по стеку (см. «Принцип
   загрузки skills» ниже).
4. **Один debugger** на все стеки (root-cause процесс нейтрален), глубину под стек
   даёт загрузка профильных skills, не отдельный агент. Для runtime по живому
   процессу/дампу - `dex-dotnet-runtime-diagnostician`.
5. **Perf под TS** не покрыт специальным агентом (вне текущего скоупа).

> **Принцип загрузки skills общими агентами** - by-stack loading: стек по манифесту ->
> фильтр available-skills по префиксу `dex-skill-<стек>-*` через реестр
> `dex-skill-stack-registry`. Новый стек = новые skills + строка реестра; агенты не
> правятся. Полное описание и обоснование - [AGENT_FRAMEWORK.md](AGENT_FRAMEWORK.md),
> раздел «By-stack loading».

## Чужой MR - отдельный процесс (не часть разработки)

Детализация трека **«Ревью входящего MR»** из верхней таблицы. Ревью **входящего**
MR - отдельный процесс (read-only, выход = комментарии, код не трогаем):

| Слот | Агент |
|------|-------|
| первичное ревью MR | dex-mr-reviewer (языко-агностичный, фокусы + публикация) |
| ре-ревью дельты | dex-mr-check-reviewer |
| breadth-first аудит существующего кода | dex-code-discovery |

## Слитая фича на стенде - отдельный процесс (post-merge)

Детализация трека **«Приёмка слитой фичи на стенде»**. Фича уже в основной ветке, с
неё развёрнут стенд; процесс начинается с приёмки (read-only) и продолжается
ремедиацией найденного через follow-up, не правкой общей ветки.

| Слот | Агент |
|------|-------|
| ревью слитой фичи на стенде против ТЗ (read-only) | dex-stand-reviewer (языко-агностичный, привязка образа к merge-коммиту, требование -> проба на стенде) |
| пакетная ремедиация найденных багов (follow-up MR, фикс на источнике) | dex-bug-fixer |

Границы. **Debug**-слот (`dex-debugger`) чинит один баг по коду в dev-time при красном
прогоне; `dex-bug-fixer` берёт пачку уже найденных багов post-merge и доставляет фикс
через follow-up MR с верификацией на стенде. `dex-incident-investigator` ведёт один
инцидент от симптома со своей RCA на стенде; `dex-stand-reviewer` верифицирует слитую
фичу против ТЗ, а не расследует поломку. Ревью **входящего** MR (`dex-mr-reviewer`) идёт
до мерджа по диффу, здесь - после мерджа по поведению на стенде.
