# Design: process-skill `pipeline-handoff` — контракт передачи между узлами трека «Разработка»

**Дата:** 2026-06-27
**Статус:** утверждён (brainstorming), готов к плану реализации
**Трек:** инфраструктура агентов маркетплейса (claude-code-marketplace)

## Проблема

Трек «Разработка» в `docs/DEV_PROCESS_COVERAGE.md` описан как линия слотов:

```
Контекст -> План -> [TDD-тесты -> Код] -> Прогон+Саморевью -> Security -> [Debug|Perf] -> Сдача
```

Карта нормирует **топологию** (какой агент закрывает какой слот), но **не контракт ребра** — что именно один узел обязан передать соседу. Каждый агент описывает свой вход/выход сам по себе; стык «выход architect = вход coder» нигде не зафиксирован. Следствия:

- Узел может отдать неполный handoff («сам решил, что нужно»), потребитель не знает, чего не хватает.
- Возврат при находке (reviewer/tester -> coder) ненормирован: находка может прийти «голой», без якоря и критерия закрытия — потребитель не знает, что чинить и когда закрыто.
- Движок (внешний `autonomous-task`) ведёт по карте, но узлы стыкуются вслепую.

Оркестратор НЕ нужен — он существует (внешний `autonomous-task`, superpowers), и карта на него опирается. Не хватает **звена между движком и узлами**: реестра сигнатур узлов (goal + вход + выход) + формы стыка, из которых оркестратор собирает цепочку под задачу.

## Решение

Один process-skill `dex-skill-pipeline-handoff` — словарь полей + контракт стыка. Единый skill: как агент декларирует сигнатуру через словарь (раздел A) + универсальное правило стыка (0) + каталог типовых рёбер (1-6). Сигнатура узла живёт в самом агенте (`description` для обнаружения оркестратором + тело для исполнения), не в таблице skill.

### Разделение домов

| Дом | Что описывает | Трогаем |
|---|---|---|
| `DEV_PROCESS_COVERAGE.md` | топология: слот -> агент | нет (по сути) |
| `pipeline-handoff` (новый skill) | контракт ребра: что течёт по стрелке | создаём |
| `bug-finder -> bug-fixer` handoff | частный случай (баги) | не переписываем, сноска как образец |

Карта = топология. Skill = контракт ребра. Не дублируются.

### Форма контракта (решения brainstorming)

- **Единый process-skill**, не раздел карты (агенты грузят skill в фазах; docs в контекст агента не попадают).
- **Узлы как взаимозаменяемые компоненты; сигнатуру (goal + вход + выход) декларирует сам агент, сборку строит оркестратор.** skill даёт словарь полей + форму стыка; агент несёт сигнатуру в `description` (обнаружение оркестратором до спавна) + в теле через Input/Output (исполнение). Последовательность собирает оркестратор, читая `description` агентов + требования. Рёбра — каталог типовых стыков, не жёсткий маршрут. Это растворяет «особые случаи» (рефакторинг, миграция) в значения полей + выбор узла. Отдельного реестра-таблицы нет — дом сигнатуры — агент.
- **Именованные обязательные поля + свободный формат изложения.** Полнота гарантирована перечнем полей; как излагать — на усмотрение узла. Не строгий машинный шаблон (хрупок), не «только принципы» (оставляет лазейку неполноты).
- **Прямые рёбра + нормированный возврат + резолюция неясности.** Контракт покрывает handoff вперёд (N -> N+1), возврат назад при находке, и поведение при неясности (входной пробел / mid-flight вопрос) — mode-aware.
- **Охват — ядро:** Intake -> План -> Код -> Тест -> Ревью. Входной узел (0', постановка на стороне основного агента) + 3 прямых ребра + 2 возврата (reviewer/tester -> coder, coder -> architect) + mid-flight clarification (6). Контекст/Security/Debug/Perf-рёбра — следующая итерация.
- **Состав полей — v1.** Перечни обязательных полей каждого ребра (секции 1-5) — первая версия, фиксируется как есть; добор полей (напр. приоритет/порядок задач в плане architect -> coder) — по итогам обкатки, не в этой итерации.
- **Критерий полноты требования — не наш дом.** «Полно ли R» определяет `dex-skill-requirement-quality` (оракул дефектов); раздел 0' даёт только механику стыка (основной агент сверяет, спрашивает юзера, оформляет handoff). Сноска, не копия.

## Структура skill

Process-skill, маркер `<!-- skill-type: process -->`, регистрация в `PROCESS_SKILLS` (`tools/validate-skill.js:111`). Разделы:

### Назначение: недостающее звено сборки

`pipeline-handoff` — то, что превращает набор разрозненных агентов в цепочку, **собираемую оркестратором под любую задачу**. Аналог `dex-skill-stack-registry`, но ось — узлы процесса, не стеки. Разделение ответственности:

| Слой | Дом | Что даёт |
|---|---|---|
| Движок прогона | внешний `autonomous-task` | ведёт прогон, **решает последовательность** узлов |
| **Сигнатура узла** (goal + вход + выход) | **сам агент** (`description` + фазы Input/Output) | что узел принимает и что отдаёт |
| Словарь полей + правило стыка | этот skill | определения полей (R/I, criteria, diff-scope, якорь), форма стыка, реакция на нехватку |

**Сигнатуру декларирует агент, не реестр-таблица.** `description` несёт сжатую сигнатуру для обнаружения оркестратором; фазы Input/Output — полную форму для исполнения (детали — раздел A). skill — **общий словарь полей**; агент ссылается на него в своей сигнатуре. Дом сигнатуры = агент (он автор своего контракта) -> нет рассинхрона «таблица vs агент», новый агент приносит сигнатуру с собой. Это «один факт — один дом»; отдельного реестра-таблицы НЕТ.

**Последовательность строит оркестратор**, читая `description` агентов (единственное видимое до спавна, см. раздел A) + требования (проработанное R). Контракт НЕ зашивает маршрут «architect->coder->tester» — даёт словарь полей и форму стыка, из которых оркестратор собирает цепочку. Рёбра 1-6 — **каталог типовых стыков** (примеры комбинаций словаря), не жёсткий путь.

### A. Как агент декларирует сигнатуру

Сигнатура живёт в **двух носителях с разными читателями** — это не дубль, а разрешение видимости (платформа подтверждена, см. ниже):

| Носитель | Что несёт | Кто читает / когда |
|---|---|---|
| **`description` агента** (frontmatter) | сжатая сигнатура: «принимает R/I + criteria, отдаёт diff-scope» | оркестратор — **до** спавна (выбрать узел + чем снабдить) |
| **тело** (фазы Input/Output) | полная форма полей + проверка приёмником на входе | сам субагент — **после** спавна |

- **фаза входа** (первая рабочая): `**Input (handoff):**` — какие поля словаря узел принимает (его половина контракта приёмника, см. раздел 0 «реакция на нехватку»);
- **фаза выхода** (последняя): `**Output (handoff):**` — какие поля словаря узел отдаёт.

**Сигнатура ОБЯЗАНА быть в `description`, не только в теле.** Платформенный факт (sub-agents.md, сверено): оркестратор до спавна видит **только `description`** из frontmatter — реестр агентов статичен, формируется при старте сессии, тело субагента грузится только в его контекст после спавна и оркестратору невидимо. Та же механика, что у skill (`description` всегда, тело по вызову) и by-stack loading (отбор по видимому списку + `description`). Поэтому без сигнатуры в `description` оркестратор не знает, чем снабдить узел -> цепочка не собирается. `description` = контракт обнаружения, не реклама.

Узел выбирается по goal (из `description`), не по имени: «код в готовой архитектуре» -> coder; «пачка багов» -> bug-fixer. Fan-out (1->N->1) — **внутренняя кухня узла**, не новый тип стыка: снаружи узел принимает один handoff и отдаёт один (так разведка/декомпозиция внутри узла не пересекает границу контракта).

> **feature-implementer вне охвата этой итерации.** Он агрегат — внутри сам себе coder+tester+интегратор (спавнит суб-агентов только для разведки, не вызывает наши узлы). Встраивать его рядом с атомарным coder дало бы неоднозначность маршрута. Контракт ядра отрабатываем на атомарных узлах; feature-implementer — отдельный крупноблочный вход, подключение позже.

**success criteria выражает и «новое поведение», и «инвариант».** Для фичи criteria = «поведение X появилось»; для рефакторинга/миграции criteria = «поведение сохранено, существующие тесты зелёные до и после». tester получает criteria и работает под них — генерирует новые ИЛИ верифицирует эквивалентность. Класс задачи (новое vs инвариант) — это **значение поля criteria**, не отдельное ребро.

### 0. Общее правило стыка (применимо ко всем рёбрам)

- Выход узла = именованные обязательные поля раздела ребра.
- Отсутствующее поле фиксируется явным статусом (`n/a` + причина / `unverified`), не молчанием.
- Формат изложения свободный; обязателен состав, не верстка.
- Потребитель на входе проверяет полноту полей; недостающее поле — повод вернуть запрос источнику, не домыслить.
- **`[default-ok]` поле `mode: interactive | autonomous`** в каждом прямом handoff. Источник проставляет; дефолт `interactive`. Субагент режим не угадывает — читает из handoff. **Режим меняет адресата возврата, НЕ право пропустить бизнес-решение** (см. ниже).

**Реакция приёмника — критерий «природа недостающего решения», не имя поля и не режим.** (Уточнено по тесту сценариев: агент в `autonomous` корректно сделал halt при нехватке success criteria — старая модель «autonomous = всегда допущение» была слишком груба.)

| Род нехватки | Что это | Реакция (оба режима) |
|---|---|---|
| **Инженерное решение** | агент компетентен выбрать: округление, тип возврата, паттерн, имя, структура | `autonomous` -> явное допущение + громкая пометка; `interactive` -> можно вернуть оркестратору ИЛИ принять с пометкой |
| **Бизнес-решение** | агент НЕ может знать правильно: success criteria («готово»), бизнес-правило, приёмка, намерение | **Halt + возврат оркестратору в ОБОИХ режимах.** Отсутствие = неполная постановка; задачу надо пополнить, не угадать |
| **`[default-ok]`** | non-goals, ADR, остатки | дефолт + пометка, оба режима |

`success criteria` и `requirements` — бизнес-ось: halt даже в `autonomous`. `diff-scope`, контракты — блокеры, но без них работать тоже нельзя -> halt. Сомнение «инженерное/бизнес» -> считать бизнес. `autonomous` НЕ значит «эскалировать некому, поэтому угадай»: для бизнес-нехватки -> «верни оркестратору, пусть пополнит постановку». **Исключение — self-reviewer:** намерение для него не конститутивно (diff есть всегда), его отсутствие -> `intent: n/a`, ревью идёт, не halt.

### 0'. Requirement Intake (вход цепочки — уточнение грубой постановки)

Источник вызова цепочки часто даёт **грубое R** («экспорт заказов в CSV») без обязательных полей. Доработать грубое R в полное = интервью с юзером. **Интервью физически может вести только основной агент** (главный цикл): у него есть `AskUserQuestion`; спавненный субагент канала к юзеру не имеет и спросить не может.

Поэтому вход цепочки нормируется как поведение основного агента, не субагента:

- Основной агент, получив грубое R, грузит `pipeline-handoff` и сверяет R с обязательными полями ребра 1 (R/I, success criteria — `[blocking]`).
- **Проверка R — на полноту И на дефектность, не только полноту.** Оракул — `dex-skill-requirement-quality` (двусторонний, активируется на «постановка задачи / критерии приёмки / definition of done»). Проверяются все классы дефекта: неполнота happy-path, **взаимное противоречие пунктов**, неоднозначность без измеримого критерия, конфликт с инвариантом/ADR, невыполнимость. Критерий дефекта не дублируется в 0' — это дом `requirement-quality`; раздел 0' даёт только механику стыка.
- **Подробное ТЗ не освобождает от проверки на дефекты.** Полнота != корректность: ТЗ может быть исчерпывающим и при этом противоречивым. Intake не пропускает полное R мимо `requirement-quality` — прогоняет на противоречие/конфликт с ADR до спавна (кейс «подробное описание»).
- Любой дефект (пробел ИЛИ противоречие) -> **разрешить с юзером до спавна цепочки**, не домыслить и не передать дефектное R дальше молча.
- Когда `[blocking]` поля собраны -> отдать coder'у (или architect'у) полный handoff ребра 1.

**Источник ребра 1 — проработчик: architect ИЛИ основной агент.** architect-субагент проектирует на собранных ответах — интервью вёл основной агент (главный цикл) до спавна. Сам architect не интервьюирует (см. «Все агенты не интерактивны»). Основной агент сам становится источником ребра 1, когда проработал R интервью без architect'а. Coder на входе источник не различает — сверяет полноту полей одинаково.

**Маршрутизация architect vs coder-напрямую — по наличию архитектурного измерения** (критерий из Boundaries самого `architect`):

| Признак задачи | Маршрут |
|---|---|
| Новая система/сервис; storage/sharding/cache; нагрузка/масштаб; распределённость; CAP/trade-offs; «спроектировать» | основной агент -> **architect** (субагент: capacity + reference как данные) -> coder |
| Код в **существующей** архитектуре: метод/класс/эндпоинт/фича по готовым контрактам; «реализуй/добавь/напиши» | основной агент -> **coder напрямую** (architect избыточен) |
| Неясно, есть ли арх-измерение | по умолчанию **coder**; его Phase 2 (Study Project Context) выявит потребность в арх-решении -> **возврат секции 5** «нужен этап арх-проработки», не домысливать архитектуру |

Третья строка закрывает лазейку без угадывания: неверная маршрутизация ловится возвратом coder'а, а не тихим домысливанием.

Этот раздел закрывает слот «Контекст» трека (DEV_PROCESS_COVERAGE) на стороне основного агента; своего субагента слот не получает (интервью субагенту недоступно).

### 1. architect/основной агент -> coder (План -> Код)

Обязательные поля выхода плана (проработчик = architect или основной агент после интервью):
- `[blocking]` требования R (явные) и I (неявные, из проекта);
- `[blocking]` success criteria — проверяемые, измеримые;
- `[default-ok]` non-goals (что НЕ делать);
- `[default-ok]` ключевые решения и релевантные `Accepted` ADR;
- `[default-ok]` известные ограничения / риски.

**Контракт ребра 1 — минимум по смыслу, не по именам полей.** Инвентаризация: architect выдаёт план как **инкременты** с per-инкремент `Scope / Dependencies / Risks / DoD / Success metric` + Deep Dive (storage/API/cache/...), а не дословно «R + success criteria». Это покрывает ребро 1 другими именами: `DoD`/`Success metric` = success criteria; `Scope` (что входит/не входит) = требования R + non-goals; `Deep Dive` = ключевые решения; `Risks` = ограничения. Словарь полей skill признаёт эти соответствия как **синонимы источника** — coder на входе сверяет наличие *смысла* (есть ли проверяемый критерий готовности, очерчен ли scope), не буквальной метки «success criteria». Иначе coder отверг бы валидный план architect из-за разницы в именах. architect/architect-dotnet выдают эту форму идентично (подтверждено) — ребро 1 единое для обоих.

### 2. coder -> tester (Код -> Тест)

Обязательные поля выхода кодера для тестописателя:
- `[blocking]` **пути** изменённых/созданных файлов (`diff-scope`), не тела;
- `[blocking]` публичные контракты под тест (сигнатуры, эндпоинты, типы);
- `[blocking]` **success criteria из плана как оракул тестов** — тесты держатся за критерии приёмки, НЕ за план реализации (иначе тесты цементируют ошибки реализации или расходятся с кодом);
- `[default-ok]` что НЕ покрыто кодером (явные пробелы).

**Тела функций в handoff не передаются** — tester читает реализацию с диска по путям (как self-reviewer берёт diff из git). Транспорт кода = диск, handoff несёт пути + сигнатуры. Дублировать тело в текст handoff избыточно и провоцирует тест-зеркало реализации.

### 3. coder/bug-fixer -> self-reviewer (-> Ревью)

**self-reviewer захватывает изменения сам — два режима.** Дефолт: из git (Phase 0 Capture: committed+staged+worktree) — транспорт через git/worktree (как у mr-reviewer), не текстовый handoff diff'а. Fallback: git-scope пуст (нет коммитов/ветки) ИЛИ handoff явно перечислил `diff-scope` пути -> ревьюит перечисленные файлы с диска. Готовый diff-текст в handoff не принимает в любом режиме — только пути. От контракта ему нужно **намерение** — иначе intent-gate не с чем сверять:
- `[blocking]` **success criteria / источник намерения** (план, задача) — для intent-gate (сверка кода с требованием). Для bug-fixer намерение = **исходные баги** (red->green как оракул), не план фичи;
- `[default-ok]` результат прогона build/test/lint от источника — статус (зелёный / красный + что);
- `[default-ok]` `diff-scope` пути — для fallback-режима захвата;
- `[default-ok]` известные остатки / loose ends.

Готовый diff-текст в этом ребре НЕ передаётся (избыточно — self-reviewer берёт изменения из git, fallback читает по путям). Это сужает Input против первоначального наброска, где ошибочно стоял `[blocking]` diff-скоуп.

**Источник ребра 3 — наш coder ИЛИ `bug-fixer`.** bug-fixer завершает баговый трек на локальных коммитах и сам указывает «следующий шаг — `dex-self-reviewer`» (Phase 6); его выход несёт намерение (баг->закрыт/отложен, исходные баги). bug-fixer **не дорабатываем** (своя 7-фазная канва, правка = major-риск) — контракт лишь признаёт его легальным источником; self-reviewer сверяет намерение одинаково, кто бы ни был источник. Это смыкание багового трека (`bug-finder -> bug-fixer`, свой handoff) с ядровым ребром 3.

> **«Следующий узел» — не нормативное поле контракта.** Какие-то агенты называют в теле «следующий шаг X» (bug-fixer -> self-reviewer, architect -> coder) — это подсказка человеку, не обязательство. Маршрут (кто за кем) решает **оркестратор**, не агент. Узел обязан только принять свой Input и отдать свой Output; куда Output пойдёт — дело сборки. В карте сигнатур «след. узел» — информативно («типично передаёт X»), не часть контракта.

### 4. Возврат reviewer/tester -> coder (обратное ребро)

Каждая находка возврата = обязательные поля:
- якорь (file:line);
- критерий закрытия (когда находка считается исправленной);
- severity.

Находка без якоря или критерия закрытия не передаётся как возврат — это закрывает дыру «голой находки». Образец живого контракта возврата уже в репо: `bug-finder` Phase 5 handoff-карточка (сноска).

### 5. Возврат coder -> architect (обратное ребро плана)

Кодер обнаружил, что план невыполним или противоречив (требования взаимоисключают, контракт нереализуем на стеке, не хватает решения для развилки). Возврат architect'у = обязательные поля:
- что именно невыполнимо/противоречиво (конкретный пункт плана, не «план плохой»);
- почему (на чём именно ломается: контракт, ограничение стека, недостающее решение);
- чего не хватает в плане, чтобы стало выполнимо.

Кодер не домысливает план сам и не реализует «как понял» — возвращает находку. Симметрично возврату reviewer/tester -> coder.

### 6. Mid-flight clarification (вопрос, возникший в ходе работы)

Вход был полон, но **по ходу** работы всплыла развилка, не покрытая success criteria («экспортировать заказы удалённых юзеров?», «дата ISO или локаль?»). Это не нехватка входного поля и не «план невыполним» — отдельный класс. Субагент **не может спросить юзера** (канала нет), физический выбор: домыслить или вернуть наверх.

Резолюция — по природе развилки (та же ось «инженерное / бизнес», что в разделе 0):

| Класс развилки | Реакция (оба режима) |
|---|---|
| **Бизнес-развилка** (меняет контракт/поведение: права эндпоинта, удалять vs архивировать) | **Halt + возврат оркестратору в обоих режимах**: развилка, варианты, почему не решает сам. Не угадывать -- это неполная постановка. |
| **Инженерная развилка** (агент компетентен: паттерн, форма) | `autonomous` -> допущение + громкая пометка; `interactive` -> можно вернуть оркестратору. |
| **Некритичная** (любой разумный выбор приемлем: формат даты, имя приватной переменной) | Дефолт + пометка в выводе, оба режима. |

«Вернуть наверх» != «спросить юзера»: субагент возвращает вопрос **оркестратору/вызывающему** (всегда, в обоих режимах). Оркестратор сам решает — `AskUserQuestion` юзеру, пополнить из другого узла или по своей политике. Гарантия контракта: по бизнес-развилке субагент не угадывает молча, юзера не адресует.

## Доработка агентов (прозрачные стыки)

Каждый узел ядра **декларирует сигнатуру в `description`** (сжато, для обнаружения оркестратором — раздел A), грузит `dex-skill-pipeline-handoff` императивно в двух точках и декларирует полную форму в теле:
- **`description` агента** — дописать сигнатуру «принимает X, отдаёт Y» к существующим триггерам; без неё оркестратор не знает, чем снабдить узел (тело до спавна невидимо);
- **фаза входа** (Understand Requirements / Intake) — `**Input (handoff):**` объявляет принимаемые поля; узел **проверяет полноту входящих требований** и при нехватке обязательного поля **возвращает вопрос на доработку** источнику (по правилу раздела 0, mode-aware), не домысливает;
- **фаза выхода** (Validate / Report) — `**Output (handoff):**` объявляет отдаваемые поля; узел отдаёт handoff по контракту.

> Сигнатура в `description` + проверка входа + возврат вопроса на доработку — общее свойство **всех** агентов-узлов, не только ядра. Нормативный дом — `AGENT_FRAMEWORK.md` (раздел «Inter-agent handoff»): фреймворк требует сигнатуру в `description`, Input/Output в теле и поведение приёмника на неполный вход. spec фиксирует применение к ядру; правило — во фреймворке.

> **Риск формата `description`:** сигнатура «принимает X, отдаёт Y» соседствует с триггерами активации в одном поле. Триггеры — ось обнаружения по запросу юзера, сигнатура — ось снабжения оркестратором; не смешивать в одну фразу. Бюджет `description` (<=750 симв., проектный потолок) проверить после дописывания. Точная форма (отдельная фраза vs префикс) — решить на пилоте; см. «Открытые риски».

**Приёмная сторона грузит контракт безусловно** — не только когда источник тоже наш агент. Источником вызова часто будет не-наш основной агент (главный цикл) или внешний `autonomous-task`, которые контракта не знают и могут передать неполный промпт. Платформа полей не валидирует (handoff идёт текстом промпта `Agent` tool, типизированного интерфейса нет), поэтому приёмник — последняя линия: он сверяет пришедшее с перечнем обязательных полей и реагирует на нехватку по правилу ниже. Это превращает неполный handoff из тихой дыры в явный статус в выводе, даже если вызывающий «наивный».

Затрагиваемые агенты:

| Узел | Агенты | Источник входа | Канва |
|---|---|---|---|
| План | `architect`, `architect-dotnet` | основной агент (после интервью) | данные на допущениях, не интервьюирует (см. «Все агенты не интерактивны»; перепись в скилл — отдельная задача) |
| Код | `dotnet-coder`, `ts-fullstack-coder` | ребро 1 (проработчик) | канва кодеров единая — правки дословно симметричны |
| Тест | `dotnet-test-writer`, `ts-test-writer` | ребро 2 (coder) | симметрично |
| Ревью | `self-reviewer` | ребро 3 (свой coder) | принимает наш handoff |

**`mr-reviewer` / `mr-check-reviewer` — НЕ на нашем ребре.** Они ревьюят чужой PR; их источник входа — **git** (PR, треды, range-diff), не handoff от нашего coder. Транспорт состояния — гит-платформа, не текст промпта. Контракт ребра 3 к ним неприменим — убраны из доработки (см. «Два транспорта handoff»).

Правка одной оси (Skill-ссылка в фазе) применяется ко всем агентам узла единообразно — иначе рассинхрон канвы.

## Все агенты не интерактивны: диалог — только главный цикл

**Ни один наш агент-плагин не ведёт диалог с юзером.** Канал к юзеру (`AskUserQuestion`) есть только у главного цикла (встроенный Claude Code, которого запустил человек). Спавненный субагент канала не имеет физически — «интервью» в субагентной позиции либо виснет, либо превращается в молчаливые допущения. Поэтому интерактивность — **не свойство агента**; агент всегда отдаёт данные и возвращает открытые вопросы наверх, не спрашивает сам.

| Кто | Канал к юзеру | Роль в диалоге |
|---|---|---|
| Главный цикл (человек запустил напрямую) | есть | **единственный интервьюер** — уточнение постановки (0'), арх-согласование, обсуждение находок |
| Любой наш агент-плагин (всегда субагент) | нет | данные + возврат вопросов наверх; интервью/gate «одобрено юзером» не ведёт |

Следствия:

- **architect не согласовывает и не интервьюирует** — считает и предлагает варианты (capacity, reference architecture) как данные, открытые вопросы возвращает наверх; согласование с юзером ведёт главный цикл.
- **reviewer не обсуждает находки** — отдаёт наверх; «какую чинить первой» решает юзер через главный цикл.
- Связь с `mode`: агент-субагент = `mode: autonomous` (раздел 0/6) — тот же переключатель, не новый механизм. `interactive` относится к поведению главного цикла, не агента.

**Следствие для architect / architect-dotnet (отдельная задача, не эта итерация):** сейчас они насквозь интерактивны — `description` «Интерактивный architect — интервью», в теле gate «решение показано пользователю и одобрено», «пользователь подтвердил». Это обещает то, чего субагент не может. Развязка: **методология system-design (RESHADED, reference designs, capacity-формулы) -> скилл**, который грузит главный цикл при арх-согласовании; агент-architect оставляет расчётное ядро (считает варианты как данные, без интервью/gate). Аналог выноса методологии в `stack-registry`. В этой итерации — только фиксируем направление; перепись architect — следующей задачей.

Это граница контракта: `pipeline-handoff` нормирует **субагентную** передачу (текст промпта). Диалог человек <-> главный цикл — вне контракта (живой разговор, не handoff-поля).

## Два транспорта handoff

Передача между узлами идёт двумя несводимыми каналами. Контракт `pipeline-handoff` нормирует только первый.

| Транспорт | Где состояние | Применим контракт? | Узлы |
|---|---|---|---|
| **Внутрицепочечный** (субагент -> субагент) | текст промпта `Agent` tool + финальное сообщение | **да** — наш контракт | architect/coder/tester/self-reviewer ядра |
| **Через гит** (PR, треды, range-diff) | git-платформа (gh/glab) | **нет** — состояние в гите, не в передаче | `mr-reviewer`, `mr-check-reviewer`, `stand-reviewer` |

Гит-транспорт самодостаточен: ре-ревью читает дельту и прежние треды из PR, а не из handoff-полей. Натягивать `pipeline-handoff` на него — ошибка (дублировал бы то, что уже несёт гит). Поэтому mr-ревьюеры из доработки убраны.

## AGENT_FRAMEWORK под потоки

Добавить раздел «Inter-agent handoff» рядом с «Parallel Reviewer Orchestration». Сейчас фреймворк знает:
- фазы внутри агента (контракт фазы);
- параллельную оркестрацию (Parallel Reviewer Orchestration — fan-out + Calibration Pass).

Не знает: **последовательный контракт ребра** (handoff N -> N+1 и возврат) и **сигнатуру агента как узла**. Раздел нормирует три свойства любого агента-узла (дом контракта-словаря — skill, фреймворк требует поведение):

1. **Декларация сигнатуры в двух носителях.** В `description` — сжатая сигнатура (принимает X, отдаёт Y): **единственное, что оркестратор видит до спавна** (тело субагента ему невидимо — sub-agents.md), по ней он выбирает узел и снабжает входом. В теле — `**Input (handoff):**` в фазе входа и `**Output (handoff):**` в фазе выхода, ссылаясь на словарь `pipeline-handoff`: полная форма для самого субагента + проверка приёмником. Сигнатура — свойство агента, не внешней таблицы.
2. **Проверка входящих требований.** Приёмник на входе сверяет пришедшее с объявленным Input; недостающее обязательное поле — не домысливается.
3. **Возврат вопроса на доработку.** При нехватке обязательного поля реакция по природе решения: **бизнес-нехватка** (success criteria, намерение, бизнес-правило) -> halt + возврат оркестратору в обоих режимах; **инженерная** -> `autonomous` допущение + громкая пометка, `interactive` можно вернуть. «Вернуть наверх != спросить юзера» — субагент канала к юзеру не имеет, возврат всегда оркестратору (см. «Все агенты не интерактивны»).

Раздел ссылается на skill как на дом контракта-словаря (не копия). Свойства 1-3 — у **всех** агентов-узлов, не только ядра «Разработка»; ядро — первое применение.

## Scope guard (что НЕ делаем)

- НЕ строим оркестратор (движок внешний — `autonomous-task`); **последовательность узлов и координацию нескольких цепочек** (мульти-фича, проверка независимости задач по общим файлам) решает оркестратор, не контракт. Контракт даёт словарь полей + форму стыка; сигнатуры узлов оркестратор читает из `description` агентов (тело до спавна невидимо).
- НЕ трогаем рёбра Контекст / Security / Debug / Perf (следующая итерация).
- НЕ вводим строгий машинный шаблон (выбрали именованные поля + свободный формат).
- НЕ переписываем bug-handoff (`bug-finder -> bug-fixer`) — сноска как образец.
- НЕ меняем имена/порядок фаз агентов (нормативная ось) — только добавляем Skill-ссылку в существующие фазы.
- НЕ дублируем критерий полноты/дефектности требования — дом `dex-skill-requirement-quality`, раздел 0' ссылается.
- НЕ строим узел продуктового анализа (BRD, epics) — это `dex-business-analyst` / `dex-requirements-analyst`, отдельный уровень, не вход кодовой цепочки.
- НЕ даём субагенту спрашивать юзера (платформа не позволяет) — вопросы юзеру только через основной агент; субагент возвращает наверх.
- НЕ нормируем гит-транспорт ревью (`mr-reviewer`/`mr-check-reviewer`/`stand-reviewer`) — состояние в PR, контракт неприменим.
- НЕ делаем агент интерактивным — ни один наш агент не ведёт диалог; интервью/обсуждение только у главного цикла. Это абсолют, не свойство позиции.
- НЕ переписываем architect / architect-dotnet в этой итерации — вынос методологии в скилл + снятие интервью/gate зафиксированы как направление (см. «Все агенты не интерактивны»), исполнение — отдельной задачей.
- НЕ включаем feature-implementer в охват — агрегат (сам себе coder+tester+интегратор), рядом с атомарным coder даёт неоднозначность маршрута; отдельный крупноблочный вход, подключение позже.

## Открытые риски

- **Размер skill.** Ядро = словарь полей + раздел A (декларация сигнатуры) + общее правило стыка + раздел 0' (intake) + 3 прямых ребра + 2 возврата + раздел 6 (mid-flight) + неинтерактивность + два транспорта. Заметно больше типового skill — process-skill освобождён от trap-эвристик, но не от лаконичности; гнать через `optimize-for-llm`, при перерастании порога рассмотреть вынос «неинтерактивность / транспорты» в отдельный материал one-level-deep.
- **Канва self-reviewer.** Фазы не совпадают с кодерами; точку загрузки контракта подобрать по факту фаз (вход — Capture/Context, выход — Report), не навязывать чужую канву. (mr-ревьюеры исключены — гит-транспорт.)
- **Форма сигнатуры в `description`.** Сигнатура «принимает X, отдаёт Y» добавляется в `description`, где уже живут триггеры активации. Риск: раздувание поля (потолок 750 симв.) и смешение двух осей (триггеры обнаружения по запросу vs сигнатура снабжения оркестратором). На пилоте решить форму: отдельное предложение в конце `description` vs структурированный префикс; проверить, что триггеры активации не деградировали (trigger-check на реалистичных запросах) и бюджет не превышен. Правило репо «description = технология + аспекты, не список точек входа» — сигнатура это не точки входа, а контракт ввода/вывода; держать одной короткой фразой.
- **Дубль с bug-handoff.** Возврат reviewer->coder концептуально = bug-finder->bug-fixer. Держать сноской, не копией; при пересечении — вынести общее в `pipeline-handoff`, из bug-агентов сослаться (skill->skill загрузки нет, только сноска для человека). Стык `bug-fixer -> self-reviewer` решён в ребре 3 (bug-fixer как легальный источник, не дорабатывается) — не риск, но при реализации проверить, что Phase 6 Output даёт diff-скоуп по форме ребра 3.
