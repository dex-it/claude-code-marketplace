# Agent Framework

Конструктор агентов Claude Code marketplace: автор собирает агента из фаз под конкретную задачу. Читается человеком или Claude при работе над плагинами, в runtime обычного агента не грузится. Не каталог готовых шаблонов - набор принципов + словари фаз и ролей.

## Quick Start

1. Определи тип: Analyst / Diagnostician / Creator / Designer / Operator / Planner / Reviewer.
2. Возьми рецепт из [Рецептов](#рецепты-ролей) - стартовый набор фаз.
3. На каждую фазу: **Goal** (1 фраза) + **Output** (артефакт) + **Exit criteria** (когда готово). У **фазы входа** добавь **Input** - что агент требует на вход (половина сигнатуры границы).
4. Mandatory-фазы помечай с обоснованием «зачем».
5. Hard gate'ы - между критичными переходами (Isolate -> Fix).
6. В `tools:` добавь `Skill`; в теле фаз - какие skills грузить условно.
7. `node tools/validate-agent.js <path>` -> 0 ошибок.

## Глоссарий

Термины определены здесь один раз, дальше применяются без повторных объяснений.

| Термин | Определение |
| --- | --- |
| **Агент** | Специализированный .md-файл в `plugins/specialists/*/agents/`, грузится Claude Code как sub-agent. Описывает роль, контекст, workflow под класс задач. |
| **Фаза** | Самодостаточный этап работы с собственной целью, наблюдаемым результатом, критерием завершения. Базовый блок фреймворка. |
| **Goal** | Одна фраза: зачем фаза существует. Не «что делать», а «что должно быть достигнуто». |
| **Input** | Что фаза требует на вход (декларация, не цель). Обязателен у **фазы входа** агента - половина сигнатуры границы, основа сверки пришедшего. У внутренних фаз Input = Output предыдущей, явно не пишется. |
| **Output** | Наблюдаемый артефакт фазы: файл, тест, отчёт, гипотеза в явной форме. Observable, не внутреннее состояние Claude. |
| **Exit criteria** | Условие завершения фазы. Проверяемо извне. |
| **Gate** | Условие перехода к следующей фазе сверх простого завершения предыдущей. Защита от перескакивания. |
| **Hard gate** | Gate, нарушение которого блокирует переход полностью. Агент возвращается или эскалирует. |
| **Soft gate** | Gate, нарушение которого допускается с явной пометкой в отчёте. |
| **Explicit confirmation** | Gate, требующий явного подтверждения пользователя. Для необратимых действий. |
| **Mandatory** | Фаза, которую нельзя пропустить. Помечается явно, требует обоснования «зачем». |
| **Optional** | Фаза, пропускаемая по решению агента без объяснений. |
| **Conditional** | Фаза, пропуск которой зависит от явного `skip_if`. |
| **Skip_if** | Условие пропуска conditional-фазы. Проверяемое утверждение. |
| **Fallback** | Что делать, если exit criteria не достигнуты. Обычно эскалация или возврат в предыдущую фазу. |
| **Рецепт** | Типовая композиция фаз под частую роль. Стартовая точка, не контракт. |
| **Императивная загрузка skill** | Вызов Skill tool в runtime из фазы (в отличие от pre-loaded через frontmatter). Тянет только релевантные контексту skills. |
| **Observable** | Наблюдаемое извне свойство: артефакт создан, текст записан, утверждение зафиксировано. Противоположность - внутреннее состояние Claude. |

# Принципы построения

Два конститутивных принципа. Агент проектируется на них сразу, не дооснащается потом.

## Принцип 1: фаза = контракт, не процедура

Claude **знает, как** делать большинство вещей: читать stack trace, писать LINQ, анализировать Dockerfile, проектировать REST API - это в training data. Описывать процедуру = дублировать то, что он умеет, и делать агента негибким.

Фаза описывает:

- **Goal** - зачем фаза существует
- **Input** (у фазы входа) - что требует на вход; у внутренних фаз вход = Output предыдущей, не дублируется
- **Output** - какой наблюдаемый артефакт производит
- **Exit criteria** - когда завершена
- **Gate** (опц.) - условия перехода

Фаза **не описывает**: пошаговые инструкции, примеры кода, синтаксис команд, «как именно анализировать/писать/проверять». Это та же логика, что для skills («ловушки, не документация API»): **фаза - контракт, не инструкция**.

**Анти-паттерн (процедура):**

```markdown
### Phase: Reproduce
1. Прочитай stack trace из логов
2. Найди первое место в нашем коде
3. Открой файл через Read tool
4. Проверь null safety вокруг строки
```

Claude и так это умеет; пошаговый скрипт ломается при отклонениях, дублирует training data, делает агента негибким.

**Правильно (контракт):**

```markdown
### Phase: Reproduce
**Goal:** Убедиться, что баг воспроизводим - понятны шаги, входные данные, environment.
**Output:** Воспроизводящий тест ИЛИ фиксированная последовательность + минимальные входные данные + environment. ИЛИ явное «не воспроизводится, нужна доп. информация».
**Exit criteria:** Баг воспроизведён (есть артефакт) либо однозначно квалифицирован как невоспроизводимый.
**Mandatory:** без reproduce диагностика превращается в угадывание.
**Fallback:** не удалось воспроизвести -> остановиться, запросить логи/env/последовательность.
```

Ничего о «как» - только «что, зачем, когда готово».

## Принцип 2: у агента стандартизованный вход и выход

Принцип 1 нормирует контракт **внутри** агента (фаза). Принцип 2 - **границу** агента: что он принимает и что отдаёт, объявлено явно и единообразно. Это **главный принцип** - он не зависит от того, в цепочке агент или нет.

Агент-одиночка обязан иметь стандартный I/O ровно так же, как узел конвейера: вызывающий (человек через главный цикл, оркестратор, другой агент) должен знать, что подать и что получит, **до** вызова. Цепочка - лишь следствие стандартного I/O: когда входы/выходы единообразны, оркестратор стыкует выход одного со входом другого. Первично - контракт границы, не стыковка.

Каждый агент:

1. **объявляет сигнатуру** (что принимает, что отдаёт) - в `description` (вызывающий видит до спавна) и в теле (Input/Output фаз);
2. **проверяет входящее** - сверяет пришедшее с объявленным входом, недостающее обязательное поле не домысливает;
3. **возвращает нехватку наверх** - не угадывает молча и не спрашивает юзера (канала к юзеру у субагента нет);
4. **объявляет принятые решения в выходе** - всё, что узел решил сам, идёт в Output явно, не молча в коде.

Развёртка - [Inter-agent handoff](#inter-agent-handoff-последовательный-стык). Дом рантайм-контракта узла (режим `interactive`/`autonomous`, реакция на неполный вход, возврат наверх, словарь полей, graceful degradation) - skill `dex-skill-node-contract`. Конститутивно как «фаза = контракт»: агент проектируется со стандартным I/O сразу.

# Построение агента (развёртка принципа 1)

## Атрибуты фазы

Фаза - декларативные атрибуты. Минимум - `name`, `goal`, `exit criteria`. Остальное по необходимости.

| Атрибут | Обязательность | Описание |
| --- | --- | --- |
| `name` | обязательно | Короткое имя (Reproduce, Isolate, Generate, Verify) |
| `goal` | обязательно | Одна фраза: зачем фаза существует |
| `output` | рекомендуется | Какой наблюдаемый артефакт производится |
| `exit criteria` | обязательно | Когда завершена |
| `mandatory` / `optional` / `conditional` | рекомендуется | Можно ли пропустить |
| `skip_if` | опционально | Условие пропуска |
| `gate_to_next` | опционально | Hard/soft условие перехода |
| `fallback` | опционально | Что делать, если exit criteria не достигнуты |

**Output должен быть observable.** Хорошо: «`bug-repro-test.cs` создан и падает», «отчёт содержит список гипотез с файлами и строками», «решение зафиксировано в ADR-042». Плохо: «агент понял задачу», «анализ завершён», «гипотеза в голове».

**Output одной фазы - вход следующей.** Output, который дальше не используется, = фаза лишняя либо output сформулирован неверно.

## Композиция фаз

**Последовательность (базовая)** - фазы по порядку, каждая после exit criteria предыдущей. Дефолт для большинства агентов.

```text
Phase 1 -> Phase 2 -> Phase 3 -> ...
```

**Условный пропуск** - фаза пропущена по `skip_if`. Не «выполнилась», а «не применима в контексте». Для Context Gathering: контекст уже в промпте -> сбор не нужен.

```text
Phase 1 -> [Phase 2 skip_if context_already_known] -> Phase 3
```

**Ветвление** - после фазы выбор из нескольких следующих; выбор делается в самой фазе (её output включает решение какую ветку брать). Для Operator-агентов (docker: Diagnose -> Troubleshoot | Optimize | Setup).

```text
Phase 1 -> Phase 2 (Branch) -> {Phase 3a | Phase 3b | Phase 3c} -> Phase 4
```

**Не поддержано:** циклы (LLM плохо держит состояние, риск бесконечного повтора) и параллельные фазы (LLM выполняет последовательно, параллельность - сложность без пользы).

## Gate'ы

Gate - критерий перехода между фазами, **защита от перескакивания** (LLM охотно прыгает от гипотезы к фиксу без проверки).

- **Hard gate** - переход невозможен, пока условие не выполнено; иначе возврат или эскалация. Пример (Diagnostician, Isolate -> Fix): «root cause подтверждён - файл, строка, причинно-следственная связь». Без этого Fix запрещён.
- **Soft gate** - предупреждение, переход возможен с пометкой в отчёте. Когда экспертиза Claude оправдывает пропуск. Пример (Analyst, перед Skill-Based Scan): «Direct Analysis не нашёл проблем -> Scan можно пропустить, указав в отчёте».
- **Explicit confirmation** - явное подтверждение пользователя перед необратимым действием. Пример (Operator, перед Execute с правкой инфраструктуры).

Формат:

```markdown
### Phase 3: Fix
**Goal:** Минимальное изменение, закрывающее root cause.
**Gate from Phase 2 (hard):** root cause подтверждён - файл, строка, причинно-следственная связь. Без этого - вернуться в Phase 2.
**Gate (explicit confirmation):** пользователь одобрил план изменения.
**Output:** Изменённый код + failing test, который теперь проходит.
**Exit criteria:** Test, воспроизводящий баг, перешёл из failing в passing.
```

**Gate без условия - не gate.** «Gate from Phase 1: Phase 1 завершена» = тривиальное следование. Gate - **дополнительное** условие сверх завершения предыдущей.

## Правила пропуска

| Тип | Смысл | Когда |
| --- | --- | --- |
| `mandatory` | Нельзя пропустить никогда | Критичные для корректности: Reproduce в Diagnostician, Verify в Operator |
| `optional` | Можно пропустить по решению агента | Добавляют ценность, но не критичны: Document в Designer |
| `conditional` | Пропуск по `skip_if` | Context Gathering - если контекст уже в промпте |

Mandatory-фазы - **защита от failure mode'ов**. Помечаются явно; нарушение должно быть заметно. **Mandatory без обоснования** = скорее всего фаза не mandatory: после `**Mandatory:** yes` обязателен «зачем».

## Тело фазы: что писать и что нельзя

Разметка фаз - **в теле агента**, не во frontmatter (нестандартные YAML-поля рискуют сломать загрузку; LLM лучше следует инструкциям в теле markdown). Это про *разметку фаз* - стандартные поля `name`/`description`/`tools`/`model`/`skills` валидны и живут во frontmatter.

**Писать можно:**

- Граничные условия и запреты («не переходи к Fix без подтверждённого reproduce»)
- Ссылки на skills для этой фазы (императивно через Skill tool)
- Специфику предметной области, важную именно здесь
- Формат output, если нестандартный
- Что считается «достаточным» для exit criteria

**Писать нельзя:**

- Пошаговые процедуры («сначала прочитай файл, потом grep, потом...») - Claude сам выберет стратегию (это процедурное описание фазы, см. принцип 1)
- Объяснения, как работает технология, и документацию API/синтаксиса - Claude знает; неочевидная грабля - материал для skill, не агента
- Примеры кода - Claude сгенерирует под контекст
- **Дублирование содержимого skill** - копия раздела/чек-листа разойдётся с оригиналом; оставить ссылку. Skill переиспользуется разными агентами, актуализируется отдельно
- **Безусловную загрузку всех skills** - то же, что pre-load, но дороже по токенам. Условия конкретны: «если стек X», «если баг класса Y», «всегда для этой категории»
- **Skill вместо базовых знаний Claude** - SOLID/базовые концепции Claude знает из training data; skill только там, где грабля не очевидна

Появились нумерованные шаги «сделай то, потом то» - фаза описана как процедура, переписать в контракт.

# Взаимодействие агентов (развёртка принципа 2)

Два вида межагентного взаимодействия: **последовательный стык** (выход N -> вход N+1) и **параллельный fan-out** (несколько агентов на разных фокусах -> свод оркестратором).

## Inter-agent handoff (последовательный стык)

Стандартный I/O нужен каждому агенту - и одиночке, и узлу. Когда агенты стыкуются, выход N становится входом N+1 (и возврат N+1 -> N). Дом рантайм-контракта узла (словарь полей, правило стыка, режим `interactive`/`autonomous`, реакция на неполный вход, возврат наверх, graceful degradation) - skill `dex-skill-node-contract`; фреймворк требует **стандартный I/O и поведение на неполный вход**, не копирует словарь.

Узлу `node-contract` нужен **всегда** - это безусловный process-skill, поэтому подключается через **`skills: dex-skill-node-contract`** во frontmatter (pre-load, см. [«Подключение skills»](#подключение-skills-pre-load-безусловного-императив-условного)), не ручным Skill-вызовом в Input-фазе. Pre-load кладёт словарь стыка в контекст до первой фазы и не зависит от того, что агент не забудет его вызвать.

Четыре свойства - у **каждого** агента (трек «Разработка» - первое применение словаря стыка, не исключение):

### 1. Сигнатура в двух носителях

Сигнатуру декларирует **сам агент**, не внешний skill или реестр. Два носителя, два адресата:

- **`description`** - сжатая сигнатура (что принимает, что отдаёт), **контракт для вызывающего**. Вызывающий видит **только `description`** до спавна, тело субагента ему невидимо ([sub-agents.md](https://code.claude.com/docs/en/sub-agents)). Нет сигнатуры в `description` -> вызывающий не знает вход, оркестратор не видит узел для сборки. Поэтому обязательна.
- **Тело** - `**Input (handoff):**` в фазе входа, `**Output (handoff):**` в фазе выхода, **для самого агента** после спавна: полная форма + основа проверки пришедшего (свойство 2). Ссылка на словарь `node-contract` объясняет *что значат* поля; *какие* поля требует агент - пишет сам агент. Дом сигнатуры - агент, словарь - общий язык.

Расхождение между носителями - дефект (вызывающий подаст не то, что агент валидирует).

### 2. Проверка входящих требований

Приёмник сверяет пришедшее с объявленным Input. Недостающее обязательное поле **не домысливается** - реакция по свойству 3. Сверка по смыслу, не по имени (синоним засчитывается).

### 3. Возврат вопроса на доработку

Реакция на нехватку - по **природе недостающего решения**, не по режиму:

- **Бизнес-нехватка** (success criteria, намерение, бизнес-правило - агент не может знать правильно) -> halt + возврат оркестратору в **обоих** режимах. Отсутствие = неполная постановка; пополняют, не угадывают.
- **Инженерная** (агент компетентен: округление, тип, паттерн) -> `autonomous`: явное допущение + громкая пометка; `interactive`: можно вернуть. Сомнение «бизнес или инженерное» -> считать бизнес.

**«Вернуть наверх» != «спросить юзера».** Субагент канала к юзеру не имеет; возврат **всегда** оркестратору/вызывающему. Оркестратор решает - эскалировать юзеру, пополнить из другого узла или сам.

### 4. Принятые решения - в выход

Узел решил сам (восполнил инженерную нехватку, истолковал неоднозначность, выбрал паттерн/имя, применил дефолт) - решение **обязано** попасть в Output явным пунктом, не остаться молча в коде. Вызывающий должен видеть, на каких допущениях стоит результат. Решение без следа = тихое допущение, запрещено в **обоих** режимах (`autonomous` даёт право не возвращать, не право молчать). Симметрично свойству 3: что не вернул наверх - объяви в выходе.

### Интерактивность - свойство позиции вызова, не агента

Канал к юзеру (`AskUserQuestion`, любой вопрос-ожидание) есть **только у главного цикла** - его запустил человек.

- **Спавн через `Agent`** - канала к юзеру НЕТ физически. «Интервью» здесь виснет или вырождается в молчаливые допущения. Узел отдаёт данные, открытые вопросы возвращает **наверх**. Это дефолт (`mode: autonomous`).
- **Исполнение телом в главном цикле (через команду-обёртку)** - канал ЕСТЬ. Тело, подхваченное главным циклом по команде, может вести диалог и ждать апрува. Это `mode: interactive`, включается **только** явной передачей во входе; по обстановке не «детектируется».

Один агент неинтерактивен при спавне и может быть интерактивным при исполнении телом через команду. Агент с интерактивными шагами **обязан** развести их по `mode` (дефолт `autonomous`); интерактивный шаг без явного `interactive` во входе - дефект (в субагентной позиции зависнет). Контракт handoff нормирует только субагентную передачу; режим и его дефолт - дом `dex-skill-node-contract`.

### Охват: новый агент несёт handoff сразу

Свойства 1-4 - требование к **новому** агенту с создания (узел проектируется как узел). Существующие дооснащаются по трекам. Агент без объявленной сигнатуры не ошибочен - он ещё не подключён к сборке; создавать новый без неё нельзя.

## Parallel Reviewer Orchestration (fan-out)

Несколько агентов параллельно (разные фокусы - code quality / security / performance / architecture) -> результаты сводятся через **Calibration Pass** оркестратора, не конкатенацией:

```text
[Agent A] ─┐
[Agent B] ─┤-> Cross-Agent Deduplication -> Calibration Pass -> Final Report
[Agent C] ─┤
[Agent D] ─┘
```

**Cross-Agent Deduplication** - одну находку от нескольких агентов объединить в один пункт.

**Calibration Pass** - для каждой `CRITICAL`/`HIGH` находки три вопроса: (1) actual blocker в этом проекте или универсальный best practice? (2) локальные причины (pre-auth, временная интеграция, MVP)? (3) минимальный фикс vs идеальный? **Зачем отдельный pass:** параллельные агенты создают **видимость консенсуса** (3 из 4 подтвердили -> кажется важным), но все 4 могут иметь общий слепой зон (все предполагают production-стадию).

**Negative claim суб-агента не закрывает фокус на слово.** Вердикт зонда «в фокусе чисто» - claim (см. [«Ответ второй стороны - claim»](#fact-verification-и-ответ-второй-стороны)), не доказанное отсутствие. Принимается, только если видно *что* и *где* искал (перечень мест/паттернов); голое «чисто» без следов охвата = фокус не отработан. Оркестратор перезапускает фокус или проверяет сам. Зеркально находке: её фальсифицируют, а ненайденность фальсификации проверяют через охват.

# Frontmatter и оформление

## Frontmatter и структура агента

Frontmatter - стандартные поля: `name`, `description`, `tools`, `model`, опц. `skills:` (pre-load безусловного process-skill, см. [«Подключение skills»](#подключение-skills-pre-load-безусловного-императив-условного)). `permissionMode: default` не пишем (и так дефолт). **Нет** `allowed-tools:` (у агентов `tools:` / `disallowedTools:`).

```markdown
---
name: agent-name
description: ...
tools: Read, Edit, Bash, Grep, Glob, Skill
model: sonnet
skills: dex-skill-node-contract   # pre-load безусловного process-skill (узлу нужен всегда); опускается у не-узлов
---

# Agent Name

Краткая роль и область применения. 1-2 абзаца.

## Phases

Перечисление фаз с goal - обзор workflow.

## Phase 1: <Имя>
**Goal:** ...
**Output:** ...
**Exit criteria:** ...
**Mandatory:** yes / optional / conditional (skip_if: ...)

[тело фазы: граничные условия, запреты, ссылки на skills, специфика домена]

## Phase 2: <Имя>
**Goal:** ...
**Gate from Phase 1 (hard):** ...
**Output:** ...
**Exit criteria:** ...

## Boundaries

Глобальные границы: что НЕ делает, когда эскалирует, что требует подтверждения.
```

### Выбор модели (`model`)

`model` - **жёсткий потолок**, не минимум. Значения: `opus`, `sonnet`, `haiku`, полный ID, `inherit`. Дефолт Claude Code - `inherit`.

**Почему не `inherit`:** при нём агент на Opus-сессии делает дешёвую работу тоже на Opus - оверкилл по стоимости. Каждому агенту - явный `model` **по характеру суждения**, не по теме. Это потолок сверху: даже на Opus-сессии типовой агент идёт на Sonnet/Haiku.

| Тир | Когда | Сигналы |
| --- | --- | --- |
| `opus` | Высокая цена ошибки, открытое суждение | Оркестрация (спавн через `Agent`), cross-linking, калибровка severity, фальсификация гипотез, open-ended дизайн с trade-off, многошаговый root-cause |
| `sonnet` | Дефолт-лошадь | Кодогенерация по правилам, diagnostician по чек-листу, структурированная генерация, troubleshooting со skill-ловушками |
| `haiku` | Узкое read-only / шаблонное | Заполнение по фиксированной форме, «снять статус / прочитать логи» без правок кода и нетривиального суждения |

- Критерий - **тип когнитивной работы**, не предметная область. «Тяжёлую» тему (k8s, EF) на Sonnet нормально, если работа сводится к правилам.
- Размер файла - слабый прокси: агент это тонкий workflow-контракт, работу несут модель + skills.
- `opus` - скупо (оркестраторы, ревью, архитектура, root-cause). `haiku` - только если агент не выносит сложного суждения; грузит skills с ловушками и имеет `Write/Edit` -> уже `sonnet`.

## description (семантическая активация)

`description` - якорь активации: Claude матчит контекст разговора с keywords, чтобы выбрать субагента под `Agent` tool. Описывать **роль + области ответственности + симптомы пользователя**, не названия фаз и не имена skills/инструментов.

Формула: `<Роль / специализация> - <области ответственности>. Триггеры - <симптомы>, <названия областей>`.

| Понятие | Что это | Пример (bug-hunter) |
| --- | --- | --- |
| **Роль** | Кем выступает агент | «Поиск и исправление багов в .NET» |
| **Область ответственности** | Класс решаемых задач | `root cause analysis`, `отладка exceptions`, `deadlock`, `memory leak` |
| **Симптом пользователя** | Как пользователь описывает проблему до диагностики | `find bug`, `не работает`, `NullReferenceException`, `crash` |

Симптом - то, **с чем приходит пользователь** до понимания причины: написал «у меня NRE» -> агент подцепляется, даже если слова `bug-hunter` в чате нет.

**В description НЕ пишем:**

| Анти-паттерн | Почему |
| --- | --- |
| Названия фаз (`Reproduce`, `Verify`) | Внутренний workflow, не точка входа - пользователь не пишет «нужен Reproduce» |
| Имена skills (`dex-skill-dotnet-async-patterns`) | Грузятся императивно в фазах; в description забивают лимит |
| Имена инструментов (`Read`, `Grep`) | Это `tools:`, отдельное поле |
| Перечень всех ловушек агента | Размывает description до индекса skill, ломает лимит |
| Generic без домена («помогает с кодом») | Не активируется ни на что - мёртв для match |

**Когда менять description при расширении агента:**

| Изменение | Менять? |
| --- | --- |
| Новая область ответственности (новый класс задач) | Да - область + симптомы |
| Новая фаза в существующей области | Нет - фазы не triggers |
| Уточнил фазу, добавил skill в Pass 2 | Нет - деталь реализации |
| Новый стек (был .NET -> стал ещё Node) | Да - тех-якорь нового стека |
| Переименовал агента / сменил роль | Да - роль + области |

Проверка: «изменилось ли то, **с чем придёт пользователь**, чтобы запустить агента?» Нет - не трогать.

**Лимиты.** Платформенный лимит на description агента не задокументирован - обрезку не утверждать. Проектные пороги (`npm run validate:agents`): > 500 символов - warning, > 750 - error (хард-кап). До 750 валиден, PR не блокируется; > 500 - сигнал «стоит оптимизировать» (description каждого установленного агента в системном промпте сессии целиком - длинный налог на контекст). Якорь технологии обязателен при привязке к стеку. Симптомы - на языке пользователя, дублировать ru/en, если ожидаются оба.

**Приоритет бюджета при нехватке места** (description несёт три блока: роль/области + сжатая сигнатура handoff + активаторы). Упёрся в потолок -> **сжимай сигнатуру, активаторы держи**. Активаторы - причина существования поля (без них агент не выбирается, мёртв для матча); полная сигнатура и так в теле (`Input`/`Output` фаз). Оба блока обязаны остаться: сигнатура без активаторов = агент не находится, активаторы без сигнатуры = вызывающий не знает вход. Рез = сжатие формулировки, не удаление блока.

**Симметрия со skill.** В SKILL_FRAMEWORK: description skill = `технология + ключевые аспекты`. Для агента - `роль + области + симптомы`. Логика та же (точка входа, не содержимое), природа разная: skill активируется на тему, агент - на роль / запрос.

## Агент и skills

Агент и skill - разные сущности с разной ответственностью, не путать и не дублировать.

**Агент:** последовательность фаз (что, в каком порядке), цели и gate'ы, специфика роли, координация (когда какой skill грузить).
**Skill:** специализированные знания (ловушки, anti-patterns), стек-специфичные чек-листы (EF Core, Redis, OWASP), переиспользуемая детализация.

**Границы:** skill не знает, в какой фазе используется (самодостаточен); агент не дублирует содержимое skill (ссылается); один skill - разным агентам (`dex-skill-owasp-security` нужен code-reviewer и api-designer).

### Подключение skills: pre-load безусловного, императив условного

Claude Code даёт два механизма (оба официальные, [sub-agents.md](https://code.claude.com/docs/en/sub-agents)):

- **A - pre-load через `skills:` во frontmatter.** Полное содержимое SKILL.md детерминированно инъектируется в контекст субагента при старте. `skills:` - **штатное поддержанное поле** субагента (не кастомное, загрузку не ломает). Ограничение: нельзя pre-load skill с `disable-model-invocation: true`.
- **B - императивная загрузка через Skill tool в runtime.** Skill не в контексте до вызова; агент сам зовёт по точному `dex-skill-X:X` (нужен `Skill` в `tools:`).

**Семантическая авто-активация по `description` для субагентов не задокументирована - на неё не полагаемся ни для A, ни для B** (доку проверял claude-code-guide: механизм описан для главного цикла, к субагентам явно не отнесён). Выбор A vs B решается по критерию **«всегда нужен» vs «условно нужен»**, не по умолчанию:

| Skill | Когда нужен | Механизм | Почему |
| --- | --- | --- | --- |
| **Безусловный** (агент грузит ВСЕГДА: `node-contract` у узла) | каждый запуск | **A - `skills:` pre-load** | гарантия загрузки + не тратит ход на Skill-вызов; «всегда нужен» = ровно случай pre-load. Императив здесь - ручная эмуляция pre-load, зависящая от того, что агент не забудет вызвать |
| **Условный** (по стеку/diff/ситуации: trap-skill `ef-core`/`owasp`; условный process `completeness-mapping` «при изменении контракта») | часть запусков | **B - Skill tool** | pre-load затащит все в контекст впустую; условность по стеку при pre-load невозможна (либо всё, либо ничего) |

Почему B для условных (а не «всё pre-load»):

1. **Контекст под задачу** - агент связан с 6 trap-skill, задача требует 2; pre-load тащит все 6 впустую.
2. **Условная загрузка по стеку** - реляционка/EF -> `dex-skill-dotnet-ef-core`, Mongo -> `dex-skill-mongodb`; с pre-load невозможно.
3. **Разделение контекста между фазами** - Phase 1 без skill-контента, Phase 2 грузит только нужное.
4. **Согласованность с принципом 1** - фаза заявляет цель «проверь по чек-листу» и какой именно, декларативно.

Безусловный process-skill - во frontmatter:

```yaml
skills: dex-skill-node-contract
```

Условные skills указываются в теле фазы после атрибутов:

```markdown
## Phase 2: Isolate
**Goal:** ...
**Exit criteria:** ...

Загружай релевантные skills императивно через Skill tool:
- Всегда `dex-skill-dotnet-di:dotnet-di` - captive dependency, Service Locator
- Если deadlock/fire-and-forget - `dex-skill-dotnet-async-patterns:dotnet-async-patterns`
- Если баг с данными/EF - `dex-skill-dotnet-ef-core:dotnet-ef-core`
```

Правила: точное имя `{plugin}:{skill}`; условия декларативны («всегда» / «если X»); одно-двухсловная характеристика зачем; без копии содержимого.

**Skill недоступен:**

- **Skill tool недоступен в subagent** (маловероятно, подтверждено что доступен) - fallback: «пропусти загрузку, укажи в отчёте, что фаза без skill-проверок».
- **Skill установлен, но контекст не требует** - нормальная работа условной загрузки, пропуск без ошибки.
- **Skill упомянут, но не установлен** - ошибка автора, ловит валидатор (имена сверяются с `marketplace.json`).

### By-stack loading: общий агент грузит профильные skills по реестру

**Проблема.** Языко-агностичный агент (`debugger`, `feature-implementer`, `security-reviewer`, `mr-reviewer`) работает на любом стеке. Зашить в фазу матрицу «если .NET -> X; если TS -> Y» = править агента при каждом стеке.

**Решение - двухосевой отбор через реестр.** Агент вычисляет список, не хранит:

1. Грузит `dex-skill-stack-registry:stack-registry` - реестр стек->манифест->префикс.
2. Определяет стек по манифесту (`*.csproj` -> dotnet, `package.json` -> ts, `pyproject.toml` -> python).
3. Берёт префикс `dex-skill-<стек>-*` и **фильтрует видимый available-skills** по нему.
4. Сужает по теме задачи (симптом, ось ревью, поверхность атаки).

**Две оси, не дубль:** префикс имени = **жёсткая ось стека** (детерминированный матч); `description` skill = **мягкая ось темы**. Оба поля видны агенту до загрузки тела - отбирает явным чтением списка. Только по description ненадёжно (формулировки разнятся, агент колеблется на границе стека).

**Почему префикс (эмпирически).** Субагент получает блок available-skills и фильтрует по префиксу имени без чтения диска. Метаданных-фильтра по стеку в Claude Code нет - префикс **единственный** признак группировки. Все skills стека обязаны нести единый `dex-skill-<стек>-*`; разнобой = ложно-пустая фильтрация. Skill tool на несуществующее имя возвращает мягкую `Unknown skill` - фаза не падает (новый стек без skills = безопасный no-op).

**Новый стек = правка реестра, не агентов:** `dex-skill-<стек>-*` + строка в `stack-registry`; перечни .NET/TS в фазах - примеры под общим правилом, не закрытый список.

# Словари

## Библиотека типовых фаз

Референсный список частых фаз. Брать как есть или формулировать свои. Описания декларативные (цель/результат), без процедур.

### Context Gathering
**Goal:** Достаточно контекста для решений: стек, scope, constraints.
**Output:** Зафиксированные факты о среде, влияющие на последующие фазы.
**Typical skip_if:** контекст явно в промпте.

### Reproduce
**Goal:** Подтвердить воспроизводимость, чтобы диагностика опиралась на факты.
**Output:** Воспроизводящий тест/последовательность + env + ожидаемое vs фактическое.
**Typical mandatory:** yes для Diagnostician.

### Classify
**Goal:** Определить категорию проблемы, от которой зависит стратегия (какие signals, skills, ветку брать).
**Output:** Явная классификация в одну из предопределённых категорий + обоснование данными предыдущей фазы.
**Typical position:** после Reproduce/Diagnose, до Isolate/Execute.
**Когда нужна:** агент покрывает несколько принципиально разных сценариев (ml-debugger: loss / overfitting / memory).

### Project Bootstrap
**Goal:** Заложить технический baseline нового проекта с нуля: структура, базовые зависимости, warning-профиль, анализаторы - до бизнес-логики.
**Output:** Собирающийся скелет с активным baseline (стиль для последующих фаз задаёт он, не соседский код).
**Typical position:** стартовая фаза Creator-рецепта, до Understand Requirements.
**Typical skip_if:** агент всегда дописывает в существующий проект.
**Когда нужна:** только если агент создаёт проект с нуля.

### Study Project Context
**Goal:** Понять стиль, паттерны, соглашения существующего проекта-приёмника. Частный случай Context Gathering для Creator'ов.
**Output:** Архитектурный стиль, паттерны, naming conventions, DI registration, компоненты-образцы.
**Отличие от Context Gathering:** общий собирает стек/constraints (что снаружи), Study - стилевые соглашения (что внутри кода).
**Typical skip_if:** standalone-утилита, пустой проект (стиль задаёт Bootstrap), явное разрешение писать в своём стиле.

### Direct Analysis
**Goal:** Применить общие знания Claude для первичных гипотез/находок.
**Output:** Список находок/гипотез с привязкой к коду (файл:строка).

### Skill-Based Deep Scan
**Goal:** Проверить код против специализированных чек-листов из skills.
**Output:** Доп. находки + дедуп с Direct Analysis.
**Typical gate:** after Direct Analysis.

### Domain Priming
**Goal:** Понять словарь и контекст проекта до оценок, чтобы найти доменные конфликты (`Project` как Entity vs репозиторий) и опираться на гайдлайны проекта.
**Output:** Ключевые сущности домена (имена + смысл), термины bounded context, релевантные гайдлайны - любые документы проекта с конвенциями (CLAUDE.md, README, `*guidelines*.md`, glossary, ADR по модели домена).
**Typical position:** перед Direct Analysis в Reviewer'ах. Для Diagnostician - опц., если баг связан с доменной семантикой.
**Typical mandatory:** yes для Reviewer - иначе пропускаются ubiquitous language нарушения.
**Когда нужна:** агент даёт content-level оценки (нейминг, абстракции, соответствие домену), не только structure-level.

### Non-Code Artifacts Audit
**Goal:** Проверить изменения в non-code файлах: манифесты сборки (`.csproj`, `package.json`, `pyproject.toml`, `go.mod`), CPM (`Directory.Packages.props`, lockfiles), MSBuild (`.props`, `.targets`), конфиги (`appsettings*.json`), CI/CD.
**Output:** Находки по non-code артефактам отдельной категорией, с привязкой файл:строка.
**Typical position:** параллельно/после Skill-Based Deep Scan, до Cross-Linking.
**Typical mandatory:** yes для Reviewer - иначе пропускается класс ляпов (транзитивные зависимости, нарушение CPM, дубли инфраструктуры).

### Cross-Linking
**Goal:** Связать изолированные находки, чтобы вместо `N` мелких пунктов выдать дерево «root cause -> symptoms» с единым решением.
**Output:** Граф связанных находок: на группу - общий root cause + единое решение; stand-alone - явная пометка «не связана».
**Typical position:** после всех scan-фаз, до Synthesis/Report.
**Typical mandatory:** yes для Reviewer - иначе расфокусированный список из 20 пунктов вместо 5 проблем.
**Когда нужна:** scan-фазы дают >5 находок с возможными связями.

### Content-Level Pass
**Goal:** Проверить уместность и семантику, не только формат: для `LogInformation` - обоснование уровня; для имени - соответствие bounded context; для `try/catch` - оправданность типа исключения.
**Output:** Content-level находки с обоснованием «почему смущает» (не «формат нарушен», а «семантика не та»).
**Typical position:** после Skill-Based Deep Scan, с Non-Code Artifacts Audit.
**Typical mandatory:** обычно yes для Reviewer; optional при узко-формальном ревью.
**Отличие от Skill-Based Deep Scan:** scan ищет ловушки по чек-листу (formal pattern matching), content-level задаёт качественные вопросы (semantic judgement).

### Severity Calibration
**Goal:** Подстроить severity и формулировку под контекст проекта (stage, audience, чувствительность данных, compliance) - одна находка имеет разную severity в разных контекстах.
**Output:** Каждая находка с калиброванной severity + minimum fix и ideal fix. Калибровка меняет уровень/формулировку, **не вычёркивает** находку.
**Typical position:** после всех scan-фаз и Cross-Linking.
**Typical mandatory:** yes для Reviewer - иначе отчёт ложно паникует или ложно расслабляет.
**Fallback:** контекст неизвестен -> explicit assumption с пометкой.

### Tech Debt Classification
**Goal:** Для каждой находки-отклонения - статус `documented-tech-debt` / `silent-tech-debt` / `error`; от него зависит рекомендация (TODO+тикет / тикет в backlog / фикс сейчас).
**Output:** Категория + обоснование (какие маркеры accepted tech debt найдены/нет). Дефолт - подсвечивать: без явных маркеров находка идёт в `silent-tech-debt`/`error`, не в «возможно так задумано».
**Typical position:** после Severity Calibration.
**Typical mandatory:** yes для Reviewer.
Полный перечень маркеров - в референс-агенте mr-reviewer (рецепт Reviewer).

### Systemic vs Specific Triage
**Goal:** Отделить находки конкретного MR от системных проблем команды/проекта - системную бесполезно подавать как блокер MR.
**Output:** Каждая находка с меткой `mr-specific` / `systemic`; для systemic - рекомендация уровня процесса (DoD, CI-gate, ADR), не блокер MR.
**Typical position:** после Tech Debt Classification.
**Typical mandatory:** обычно yes для Reviewer; optional, если контекст прошлых MR недоступен.

### Output Labeling
**Goal:** Каждой находке - явная метка действия, иначе отчёт это плоский список без решения о мёрдже.
**Output:** Метка по шкале риска 🟢 accepted -> 🟡 TODO in code -> 🟠 follow-up ticket -> 🔴 block merge; 🟣 needs discussion - вне шкалы. 🟢 - только при явных маркерах accepted, иначе минимум 🟡.
**Typical position:** последняя фаза перед Report.
**Typical mandatory:** yes для Reviewer.
Полные правила - в референс-агенте mr-reviewer (рецепт Reviewer).

### Hypothesize
**Goal:** Сформулировать проверяемое утверждение о причине/следствии.
**Output:** Гипотеза в явной форме: «если X, то Y, потому что Z».

### Test Hypothesis
**Goal:** Проверить гипотезу экспериментом (тест, запрос, инспекция).
**Output:** Подтверждение/опровержение + доказательство.

### Understand Requirements
**Goal:** Убедиться, что требования понятны однозначно.
**Output:** Переформулированные требования + неоднозначности/пропуски.

### Generate
**Goal:** Создать запрошенный артефакт (код, тест, документ, схему).
**Output:** Артефакт.

### Validate
**Goal:** Проверить корректность артефакта (компилируется, тесты проходят, соответствует требованиям).
**Output:** Подтверждение корректности или список проблем.
**Typical mandatory:** yes для Creator.

### Propose Alternatives
**Goal:** Представить несколько вариантов для выбора.
**Output:** N альтернатив с trade-off'ами.

### Decide
**Goal:** Выбрать вариант с обоснованием.
**Output:** Решение + обоснование + отвергнутые альтернативы.

### Document
**Goal:** Зафиксировать решение/артефакт для долговременного хранения (ADR, спецификация, readme).
**Output:** Документ.

### Diagnose (Operator)
**Goal:** Понять текущее состояние инфраструктуры/среды перед действиями.
**Output:** Снимок состояния, релевантный задаче.

### Branch
**Goal:** На основе диагностики выбрать сценарий действий.
**Output:** Выбранный сценарий + обоснование.

### Execute
**Goal:** Применить изменения к системе.
**Output:** Результат команд + возврат значений/статусов.
**Typical gate:** explicit confirmation.

### Verify
**Goal:** Подтвердить ожидаемый эффект изменений без регрессий.
**Output:** Подтверждение работоспособности или список новых проблем.
**Typical mandatory:** yes для Operator и Diagnostician.

### Fix
**Goal:** Минимальное изменение, закрывающее установленный root cause.
**Output:** Диф + failing test -> passing test.
**Typical gate from Isolate:** hard - root cause подтверждён.

### Prioritize
**Goal:** Упорядочить элементы (задачи, варианты, находки) по явному критерию.
**Output:** Упорядоченный список + критерий сортировки.

### Report / Present
**Goal:** Представить результаты в удобной для пользователя форме.
**Output:** Финальный отчёт в заданном формате.

### Gap Analysis
**Goal:** Выявить, что покрыто и не покрыто относительно ожидаемого набора.
**Output:** Таблица покрытия + список пробелов.

## Рецепты ролей

Типы из CLAUDE.md (Analyst, Diagnostician, Creator, Designer, Operator, Planner, Reviewer) - **рецепты композиции**, не контракты. Стартовая точка.

### Analyst
```text
[Context Gathering?] -> Direct Analysis -> Skill-Based Deep Scan -> Report
```
Context Gathering часто опционален. Skill-Based Scan обычно mandatory для полноты.

### Diagnostician
```text
Reproduce -> [Classify?] -> Direct Analysis -> Skill-Based Deep Scan -> Hypothesize -> Fix -> Verify
```
Reproduce и Verify - mandatory. Между Isolate (= Direct + Skill) и Fix - hard gate. Classify - если несколько разных категорий проблем (ML debugger).

### Creator
```text
[Project Bootstrap?] -> Understand Requirements -> [Study Project Context?] -> Generate -> Validate
```
Validate - mandatory (иначе непроверенный артефакт). **Study Project Context** - если результат попадает в существующий проект; пропуск для standalone и для свежезаложенного скелета. **Project Bootstrap** - опц. стартовая, только если агент создаёт проект с нуля.

Имена фаз **нормативные**: стек-специфичные кодеры на одной канве, расходится только тело (идиомы языка). Имена и назначение совпадают дословно - страховка от пропуска при кросс-правке (правка одной оси применяется ко всем кодерам единообразно).

**Study Project Context** грузит `dex-skill-codebase-conventions` (как читать конвенции соседей + ось ADR). Профильные по стеку skills - по реестру `dex-skill-stack-registry` в Generate, без зашитого списка.

**Референс:** `dex-dotnet-coder` (с Bootstrap), `dex-ts-fullstack-coder` (без Bootstrap). Не Creator-рецепт: `dex-feature-implementer` (отдельный pipeline с Design и Falsify), `dex-ef-specialist` (Operator под данные).

### Designer
```text
[Context Gathering?] -> Analyze Constraints -> Propose Alternatives -> Decide -> [Document?]
```
Document часто опционален - ADR для значимых решений.

### Operator
```text
Diagnose -> Branch -> Execute -> Verify
```
Diagnose и Verify - mandatory. Execute обычно требует explicit confirmation для destructive (read-only без подтверждения). Branch почти всегда есть - Operator покрывает несколько сценариев; без ветвления агент путается. Однозадачный Operator - редкий случай, Branch сворачивается.

### Planner
```text
Gather -> Analyze -> Prioritize -> Present
```
Минимум ветвлений, линейный pipeline.

### Reviewer
```text
Domain Priming -> Direct Analysis -> Skill-Based Deep Scan -> Fact Verification -> Non-Code Artifacts Audit
  -> Content-Level Pass -> Cross-Linking -> Severity Calibration
  -> Tech Debt Classification -> Systemic vs Specific Triage -> Output Labeling -> Report
```
Reviewer - частный случай Analyst под code review. Три блока: **сбор находок** (Domain Priming -> ... -> Content-Level Pass), **структурирование** (Cross-Linking), **калибровка и оформление** (Severity Calibration -> ... -> Output Labeling).

**Mandatory:** Domain Priming, Non-Code Artifacts Audit, Fact Verification, Cross-Linking, Severity Calibration, Tech Debt Classification, Output Labeling. Без них агент пропускает класс находок или даёт расфокусированный отчёт. Content-Level Pass и Systemic vs Specific Triage обычно mandatory, опц. при узко-формальном ревью или недоступном контексте прошлых MR.

**Референс:** `dex-mr-reviewer` (полные контракты фаз, шкала меток, маркеры tech debt). Родственные: `self-reviewer`, `mr-check-reviewer`, `discover-reviewer`. Фреймворк держит состав и обоснование рецепта; полные правила фаз - в агентах, не дублировать сюда (копии расходятся).

**Skill-Based Deep Scan** грузит skills по стеку условно (EF Core, async, OWASP, resilience, csproj-hygiene), не все подряд.

<a id="fact-verification-и-ответ-второй-стороны"></a>
**Fact Verification** - каждое техутверждение находки (имя API, дефолт фреймворка, поведение ошибки, сложность, причинность в «Почему») сверяется с источником истины: context7 по библиотеке версии проекта, при недоступности или для общеязыкового факта - WebSearch. Уход от сверки - явный статус (`unverifiable` / `contradicted`), не молчание. Дисциплина в `dex-skill-fact-verification`; генеративная модель закономерно выдумывает имена API и переворачивает причинность, поэтому шаг отдельный и mandatory.

> **Tools для каскада:** агент с `dex-skill-fact-verification` обязан иметь в `tools` `WebSearch, WebFetch` (fallback) **и** `ToolSearch` (приоритет - context7). Эмпирически: при `tools`-allowlist context7 экспонируется как deferred MCP-инструмент, достижим только через `ToolSearch`; перечисление `mcp__context7__*` прямо в `tools` доступа не даёт. Без `ToolSearch` каскад молча деградирует до WebSearch, а тот отдаёт latest-доку - ровно ловушка «сверка против версии проекта». `ToolSearch` обязателен для всякого нового агента с фазой Fact Verification.

**Ответ второй стороны - claim, не истина.** Всё, что приходит ревьюеру/планировщику **извне** в ответ на находку - возражение автора в треде, «так задумано / не баг / иначе нельзя / тест только ручной», вердикт суб-агента-зонда - **утверждение, требующее проверки по коду**, не основание закрыть находку. Снятие (`disputed`, `closed`, `already_fixed`) - только когда возражение подтверждено чтением кода тем же ходом, что сама находка; «звучит обоснованно» / «автор объяснил» / «согласен» критерием **не является** - лазейка, через которую ложное обоснование тихо закрывает реальный дефект. Возражение о **невозможности** (нельзя протестировать/иначе) проверяется **конструктивно**: найти в коде seam/изоляцию/альтернативу, опровергающую «невозможно»; не нашёл - снятие, нашёл - находка `open` + в реплай почему возражение её не снимает. Не подтверждается и не опровергается - `clarify`/`unverifiable`, не молчаливое снятие. Механизм тот же, что у Fact Verification: генеративная сторона выдаёт правдоподобное, разваливающееся при сверке.

**Report** - verdict (APPROVE / REQUEST_CHANGES / NEEDS_DISCUSSION) + сгруппированные находки из Cross-Linking с метками из Output Labeling. Не плоский список.

**Почему Reviewer - отдельный рецепт, не под Analyst:** реальный code review требует доменного контекста, систематического покрытия non-code артефактов, cross-linking (Analyst выдаёт находки изолированно), калибровки под зрелость проекта, явных меток действий. Эти фазы - защиты от failure mode'ов code-review-агентов на реальных PR: расфокусированный список, ложная паника по best-practice, тихий accept без подсветки, находка с выдуманным API или перевёрнутой причинностью.

# Справочное

## Пример: полный агент

Diagnostician-агент (`bug-hunter`) по правилам фреймворка: композиция фаз, hard gate Isolate -> Fix, mandatory-фазы, императивная загрузка skills, fallback, explicit confirmation. Демонстрационный - реальные агенты могут быть больше.

```markdown
---
name: bug-hunter
description: Поиск и исправление багов в .NET - root cause analysis, отладка exceptions, deadlock, N+1, memory leak. Триггеры - find bug, debug, error, exception, NullReferenceException, stack trace, crash
tools: Read, Edit, Bash, Grep, Glob, Skill
model: sonnet
---

# Bug Hunter

Специалист по поиску и исправлению багов в .NET. Каждая диагностика проходит фиксированный набор фаз с hard gate между изоляцией и исправлением - защита от фикса по гипотезе.

## Phases

Reproduce -> Isolate -> Fix -> Verify. Reproduce и Verify обязательны. Fix блокируется hard gate от Isolate.

## Phase 1: Reproduce
**Goal:** Убедиться, что баг воспроизводим - диагностика опирается на факты.
**Output:** Воспроизводящий тест ИЛИ последовательность шагов + минимальные входные данные + environment. ИЛИ явное «не воспроизводится».
**Exit criteria:** Баг воспроизведён (есть артефакт) либо квалифицирован как невоспроизводимый.
**Mandatory:** yes - без reproduce диагностика превращается в угадывание по stack trace.
**Fallback:** не удалось воспроизвести -> остановиться, запросить логи/env/последовательность. НЕ переходить к Isolate.

## Phase 2: Isolate
**Goal:** Определить root cause - конкретное место в коде и причинно-следственная связь.
**Output:** Файл и строка root cause + почему приводит к симптому + related patterns вокруг.
**Exit criteria:** Гипотеза «если X в файл.cs:42, то Y, потому что Z» подтверждена чтением кода или repro из Phase 1.

Загружай релевантные skills императивно через Skill tool:
- Всегда `dex-skill-dotnet-di:dotnet-di` - captive dependency, Service Locator
- Всегда `dex-skill-dotnet-resources:dotnet-resources` - IDisposable, утечки
- Если deadlock/fire-and-forget - `dex-skill-dotnet-async-patterns:dotnet-async-patterns`
- Если баг с данными/EF - `dex-skill-dotnet-ef-core:dotnet-ef-core`

## Phase 3: Fix
**Goal:** Минимальное изменение, закрывающее root cause.
**Gate from Phase 2 (hard):** root cause подтверждён - файл, строка, причинно-следственная связь. Без этого - вернуться в Phase 2, не писать фикс по интуиции.
**Gate (explicit confirmation):** план изменения показан пользователю и одобрен.
**Output:** Диф кода + failing test из Phase 1, который теперь проходит.
**Exit criteria:** Failing test перешёл в passing без изменений самого теста.

Фикс минимальный. Не рефакторить попутно. Соседние проблемы - в отчёт, не трогать.

## Phase 4: Verify
**Goal:** Подтвердить отсутствие регрессий и локализованность изменения.
**Output:** Результат прогона всей suite + подтверждение локализации.
**Exit criteria:** Все тесты проходят. Новые failing -> вернуться в Phase 3.
**Mandatory:** yes - без verify риск скрытой регрессии.

## Boundaries
- Один баг = один фикс. Рефакторинг смежного кода - отдельной задачей.
- Нужны внешние инструменты (debugger, dump, profiler) - эскалировать, не заменять grep'ом.
- Не менять тесты, чтобы проходили. Это анти-фикс.
```

В теле фаз **нет пошаговых процедур** - цель, output, exit criteria, gate'ы, граничные условия, ссылки на skills. Как воспроизводить, каким grep'ом искать, как писать failing test - Claude решает сам.

## Self-check перед коммитом

То же проверяет `node tools/validate-agent.js <path>`, в человекочитаемой форме.

- [ ] Frontmatter: `name`, `description` (роль + области + триггеры-симптомы; > 500 - warning, > 750 - error), `tools` (с `Skill`)
- [ ] Frontmatter: `skills:` - только безусловный process-skill (узел: `dex-skill-node-contract`); условные trap-skill грузятся императивно в фазах, не сюда
- [ ] Узел handoff: безусловный `node-contract` в `skills:` (pre-load), не ручной Skill-вызов в Input
- [ ] Frontmatter: **нет** `allowed-tools:` (у агентов `tools:` / `disallowedTools:`)
- [ ] Frontmatter: явный `model` (`opus` / `sonnet` / `haiku`) по характеру суждения, не `inherit`
- [ ] Frontmatter: **нет** `permissionMode: default` (избыточно)
- [ ] Хотя бы одна `## Phase N: <Имя>`
- [ ] Каждая фаза - `**Goal:**` и `**Exit criteria:**`
- [ ] Фаза входа - `**Input (handoff):**` (половина сигнатуры, основа сверки)
- [ ] Сигнатура объявлена: вход/выход в `description` + `Input`/`Output` в теле
- [ ] Exit criteria observable (не «агент понял», а артефакт)
- [ ] Mandatory-фазы обоснованы - после `**Mandatory:** yes` есть «зачем»
- [ ] Hard gate'ы формулируют условие, не «Phase N завершена»
- [ ] Skills указаны условно (`если X - загрузи Y`), не «загрузи все»
- [ ] Нет нумерованных списков-процедур в теле фаз
- [ ] Нет документации API, boilerplate, объяснений базовых концепций
- [ ] Нет дублирования содержимого skills
