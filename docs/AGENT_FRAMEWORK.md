# Agent Framework

Методичка для создания агентов в Claude Code marketplace. Используется человеком или Claude при работе над плагинами — не загружается в runtime при обычной работе агента.

Документ — не каталог готовых шаблонов, а **конструктор**. Автор агента собирает своего агента из фаз под конкретную задачу.

## Quick Start

Создать агент за 5 минут:

1. Определи тип: Analyst / Diagnostician / Creator / Designer / Operator / Planner / Reviewer
2. Возьми рецепт из раздела [Рецепты](#рецепты-для-типовых-ролей) — стартовый набор фаз
3. Для каждой фазы напиши: **Goal** (1 фраза) + **Output** (что производит) + **Exit criteria** (когда готово)
4. Пометь mandatory-фазы с обоснованием «зачем»
5. Добавь hard gate'ы между критичными переходами (например Isolate → Fix)
6. В `tools:` добавь `Skill`, в теле фаз укажи какие skills загружать условно
7. Запусти `node tools/validate-agent.js <path>` и поправь ошибки

Детали каждого шага — в разделах ниже.

## Глоссарий

Термины, которые используются в документе. Определены здесь один раз, дальше применяются без повторных объяснений.

| Термин                  | Определение                                                                                                                                                             |
| ----------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Агент**               | Специализированный .md-файл в `plugins/specialists/*/agents/`, который Claude Code загружает как sub-agent. Описывает роль, контекст и workflow для конкретного класса задач. |
| **Фаза**                | Самодостаточный этап работы агента с собственной целью, наблюдаемым результатом и критерием завершения. Базовый строительный блок фреймворка.                           |
| **Goal**                | Одна фраза, описывающая зачем фаза существует. Не «что делать», а «что должно быть достигнуто».                                                                         |
| **Output**              | Наблюдаемый артефакт, который производит фаза: файл, тест, отчёт, гипотеза в явной форме. Observable, не внутреннее состояние Claude.                                   |
| **Exit criteria**       | Условие, при котором фаза считается завершённой. Должно быть проверяемо извне.                                                                                          |
| **Gate**                | Дополнительное условие перехода от одной фазы к следующей, сверх простого завершения предыдущей. Защищает от перескакивания.                                            |
| **Hard gate**           | Gate, нарушение которого блокирует переход полностью. Агент возвращается или эскалирует.                                                                                |
| **Soft gate**           | Gate, нарушение которого допускается с явной пометкой в отчёте.                                                                                                         |
| **Explicit confirmation** | Gate, требующий явного подтверждения пользователя перед продолжением. Используется для необратимых действий.                                                          |
| **Mandatory**           | Фаза, которую нельзя пропустить ни при каких условиях. Помечается явно, обязательно требует обоснования «зачем».                                                        |
| **Optional**            | Фаза, которую можно пропустить по решению агента без объяснений.                                                                                                        |
| **Conditional**         | Фаза, пропуск которой зависит от явного условия `skip_if`.                                                                                                              |
| **Skip_if**             | Условие, при котором conditional-фаза пропускается. Формулируется как проверяемое утверждение.                                                                          |
| **Fallback**            | Что делать, если exit criteria фазы не достигнуты. Обычно — эскалация пользователю или возврат в предыдущую фазу.                                                       |
| **Композиция фаз**      | Способ соединения фаз в агента: последовательность, условный пропуск, ветвление.                                                                                        |
| **Рецепт**              | Типовая композиция фаз для часто встречающейся роли (Analyst, Diagnostician, Creator, ...). Не контракт, а стартовая точка.                                             |
| **Императивная загрузка skill** | Вызов Skill tool в runtime из фазы агента, в отличие от pre-loaded skills через frontmatter. Позволяет подгружать только релевантные контексту skills.                |
| **Observable**          | Наблюдаемое извне свойство: артефакт создан, текст записан в отчёт, утверждение зафиксировано. Противоположность — внутреннее состояние Claude, которое нельзя проверить. |

## Почему вообще нужен фреймворк

Без фреймворка каждый автор агента изобретает структуру заново. Результат — 39 агентов с 39 разными workflow, разной степени проработки, с разным уровнем защиты от типичных failure mode'ов LLM (перескакивание к решению без диагностики, отсутствие проверки результата, размытые границы ответственности).

Фреймворк фиксирует **инструмент — фазу** — и правила его использования. Автор получает проверенную механику, а не придумывает с нуля.

## Ключевой принцип: фаза описывает цель и результат, не процедуру

Claude **знает, как** делать большинство вещей: читать stack trace, писать LINQ-запросы, анализировать Dockerfile, проектировать REST API. Всё это есть в его training data. Объяснять Claude'у процедуру — значит дублировать то, что он и так умеет, и при этом делать агента негибким.

Фаза описывает:

- **Goal** — зачем эта фаза существует
- **Output** — какой наблюдаемый артефакт она производит
- **Exit criteria** — когда можно считать фазу завершённой
- **Gate** (опционально) — условия перехода к следующей фазе

Фаза **не описывает**:

- Пошаговые инструкции (Claude сам выберет стратегию)
- Примеры кода (Claude сгенерирует под контекст)
- Синтаксис команд (Claude помнит)
- Как именно анализировать / писать / проверять (это его работа)

Это та же логика, что в проектном гайдлайне для skills: «skill — ловушки и грабли, не документация». Для фреймворка агентов аналогично: **фаза — контракт, не инструкция**.

### Анти-паттерн vs правильно

**Анти-паттерн (процедура):**

```markdown
### Phase: Reproduce

1. Прочитай stack trace из логов
2. Найди первое место в нашем коде
3. Открой файл через Read tool
4. Проверь null safety вокруг строки
5. Запусти grep .Result в этой области
```

Проблемы: Claude и так это умеет; пошаговый скрипт ломается при отклонениях; дублирует training data; делает агента негибким.

**Правильно (контракт):**

```markdown
### Phase: Reproduce

**Goal:** Убедиться, что баг воспроизводим — понятны шаги, входные данные, environment.

**Output:** Воспроизводящий тест ИЛИ фиксированная последовательность действий + минимальный набор входных данных + зафиксированное environment. ИЛИ явное заключение "не воспроизводится, нужна дополнительная информация".

**Exit criteria:** Баг воспроизведён (есть артефакт) либо однозначно квалифицирован как невоспроизводимый.

**Mandatory:** без reproduce диагностика превращается в угадывание.

**Fallback:** если воспроизвести не удалось — остановиться и запросить у пользователя: логи, env, последовательность действий.
```

Ничего о «как» — только «что, зачем, когда готово». Claude сам решит, какие инструменты применить.

## Атрибуты фазы

Фаза состоит из декларативных атрибутов. Минимальный набор — `name`, `goal`, `exit criteria`. Остальные — по мере необходимости.

| Атрибут                                  | Обязательность | Описание                                                         |
| ---------------------------------------- | -------------- | ---------------------------------------------------------------- |
| `name`                                   | обязательно    | Короткое имя фазы (Reproduce, Isolate, Generate, Verify, ...)    |
| `goal`                                   | обязательно    | Одна фраза: зачем фаза существует                                |
| `output`                                 | рекомендуется  | Какой наблюдаемый артефакт производится                          |
| `exit criteria`                          | обязательно    | Когда фаза считается завершённой                                 |
| `mandatory` / `optional` / `conditional` | рекомендуется  | Можно ли пропустить фазу                                         |
| `skip_if`                                | опционально    | Условие, при котором фаза пропускается                           |
| `gate_to_next`                           | опционально    | Hard/soft условие перехода к следующей фазе                      |
| `fallback`                               | опционально    | Что делать, если exit criteria не достигнуты                     |

**Output должен быть observable** — то есть наблюдаемым извне, не внутренним состоянием агента.

Хорошо:

- «Артефакт `bug-repro-test.cs` создан и падает»
- «Отчёт содержит список гипотез с файлами и строками»
- «Решение зафиксировано в ADR-042»

Плохо:

- «Агент понял задачу»
- «Анализ завершён»
- «Гипотеза сформулирована в голове»

**Output одной фазы — вход следующей.** Если output фазы A нигде дальше не используется, фаза либо лишняя, либо её output сформулирован неправильно.

## Композиция фаз

Фазы соединяются в агента по следующим правилам.

### Последовательность (базовая)

Фазы идут по порядку. Каждая следующая начинается после завершения предыдущей (её exit criteria).

```text
Phase 1 → Phase 2 → Phase 3 → ...
```

Это дефолтный паттерн для большинства агентов.

### Условный пропуск

Фаза может быть пропущена по условию `skip_if`. Это не «фаза выполнилась», это «фаза не применима в данном контексте».

```text
Phase 1 → [Phase 2 skip_if context_already_known] → Phase 3
```

Используется для фаз типа Context Gathering — если контекст уже известен из промпта, сбор не нужен.

### Ветвление

После фазы может идти выбор из нескольких следующих фаз. Выбор делается в самой фазе (её output включает решение, какую ветку брать).

```text
Phase 1 → Phase 2 (Branch) → {Phase 3a | Phase 3b | Phase 3c} → Phase 4
```

Используется для Operator-агентов (docker-specialist: после Diagnose либо Troubleshoot, либо Optimize, либо Setup) и других сценариев с явным выбором.

### Что не включено

- **Циклы** (фаза повторяется до удовлетворения условия) — в первой версии фреймворка не поддерживаются. LLM плохо держит состояние цикла, риск бесконечного повтора. Если в будущем появятся реальные кейсы — добавим.
- **Параллельные фазы** — не нужны. LLM-агент всё равно выполняет действия последовательно; параллельность добавляет сложность без пользы.

## Gate'ы

Gate — критерий перехода от одной фазы к следующей. Gate существует для **защиты от перескакивания**, которое LLM делает охотно (особенно перескакивания от гипотезы к фиксу без проверки).

### Типы gate'ов

**Hard gate** — переход невозможен, пока условие не выполнено. Если условие не выполняется, агент возвращается в текущую фазу или эскалирует пользователю.

Пример: в Diagnostician'е hard gate между Isolate и Fix: «root cause подтверждён — есть файл, строка, и объяснение причинно-следственной связи». Без этого Fix запрещён.

**Soft gate** — предупреждение, но переход возможен с явной пометкой в отчёте. Используется, когда экспертиза Claude может оправдать пропуск условия.

Пример: в Analyst'е soft gate перед Skill-Based Scan: «если Direct Analysis не нашёл проблем, Skill-Based Scan можно пропустить, указав это в отчёте».

**Explicit confirmation** — требует явного подтверждения пользователя. Используется перед действиями, которые нельзя откатить.

Пример: в Operator'е explicit confirmation перед Execute фазой, когда команды меняют инфраструктуру.

### Формат записи gate'а в markdown

```markdown
### Phase 3: Fix

**Goal:** Минимальное изменение кода, закрывающее root cause.

**Gate from Phase 2 (hard):** root cause подтверждён в Phase 2 — есть файл, строка, и объяснение причинно-следственной связи. Без этого — вернуться в Phase 2.

**Gate (explicit confirmation):** пользователь одобрил план изменения.

**Output:** Изменённый код + failing test, который теперь проходит.

**Exit criteria:** Test, воспроизводящий баг, перешёл из failing в passing.
```

## Правила пропуска

Фазы делятся по обязательности:

| Тип           | Смысл                                       | Когда применяется                                                                        |
| ------------- | ------------------------------------------- | ---------------------------------------------------------------------------------------- |
| `mandatory`   | Нельзя пропустить ни при каких условиях     | Критичные для корректности фазы: Reproduce в Diagnostician, Verify в Operator            |
| `optional`    | Можно пропустить всегда, по решению агента  | Фазы, добавляющие ценность, но не критичные: Document в Designer                         |
| `conditional` | Пропуск по явному условию `skip_if`         | Context Gathering — пропустить, если контекст уже в промпте                              |

Mandatory-фазы — это **защита от failure mode'ов**. Они помечаются явно и нарушение должно быть заметно.

## Формат разметки в markdown

Разметка фаз делается **в теле агента**, не во frontmatter.

**Почему не frontmatter:**

- Кастомные поля во frontmatter рискованны для Claude Code (неизвестные поля могут ломать загрузку агента)
- LLM лучше следует инструкциям в теле markdown, чем держит состояние из YAML в голове
- Тело markdown проще редактировать и читать

**Frontmatter остаётся** для стандартных полей: `name`, `description`, `tools`, `permissionMode`.

### Стандартная структура агента

```markdown
---
name: agent-name
description: ...
tools: Read, Edit, Bash, Grep, Glob, Skill
---

# Agent Name

Краткое описание роли и области применения. 1-2 абзаца.

## Phases

Перечисление фаз с goal — быстрый обзор workflow.

## Phase 1: <Имя>

**Goal:** ...
**Output:** ...
**Exit criteria:** ...
**Mandatory:** yes / optional / conditional (skip_if: ...)

[тело фазы — что важно знать именно про эту фазу в контексте именно этого агента:
граничные условия, запреты, ссылки на релевантные skills, специфика предметной области]

## Phase 2: <Имя>

**Goal:** ...
**Gate from Phase 1 (hard):** ...
**Output:** ...
**Exit criteria:** ...

[тело фазы]

...

## Boundaries

Глобальные границы агента: что он НЕ делает, в каких случаях эскалирует, какие действия требуют подтверждения.
```

Тело фазы — это место для **граничных условий и запретов, специфичных для этого агента**, не для процедур. Если в теле фазы появляются нумерованные шаги «сделай то, потом то» — это сигнал, что фаза описана как процедура и её нужно переписать.

## Агент и skills

Агент и skill — две разные сущности с разной ответственностью. Их нельзя путать и нельзя дублировать содержимое одного в другом.

### Разграничение ответственности

**Агент отвечает за:**

- Последовательность фаз (workflow) — что, в каком порядке
- Цели и gate'ы каждой фазы
- Специфику роли — bug-hunter ищет баги, code-reviewer ревьюит, architect проектирует
- Координацию — когда загружать какой skill

**Skill отвечает за:**

- Специализированные знания — ловушки, грабли, anti-patterns
- Стек-специфичные чек-листы (EF Core, Redis, OWASP)
- Детализацию, которая не помещается в агент и должна быть переиспользуемой

**Границы:**

- Skill не знает, в какой фазе он будет использован — он самодостаточен
- Агент не дублирует содержимое skill — только ссылается на него
- Один skill может использоваться разными агентами: `dex-skill-owasp-security` нужен и code-reviewer'у, и api-designer'у

### Два способа подключения skills в Claude Code

**Способ A — Pre-loaded через `skills:` во frontmatter.**

Skills инжектятся в контекст субагента при старте, все заявленные skills занимают контекст с первой секунды работы агента. Работает механически, не требует действий от агента.

**Способ B — Императивная загрузка через Skill tool в runtime.**

Skills не в контексте до момента вызова. Агент сам вызывает Skill tool в нужной фазе с точным именем `dex-skill-X:X`. Требует `Skill` в `tools:` frontmatter'а.

**Оба способа поддерживаются Claude Code.** Поле `skills:` — валидное для субагентов ([документация](https://code.claude.com/docs/en/sub-agents)). Однако для маркетплейса мы осознанно выбираем способ B — императивную загрузку. Причины:

1. **Разделение контекста между фазами.** Phase 1 (Direct Analysis) работает без skill-контента; Phase 2 (Skill-Based Scan) подгружает только то, что нужно. Это даёт честное разделение «agent knowledge» и «skill knowledge» вместо смешанного.

2. **Контекст подгружается только под задачу.** Агент может быть связан с 6 skills, но конкретная задача требует только 2. Pre-load затащит все 6 в контекст впустую — императивная загрузка тянет только релевантные.

3. **Условная загрузка по стеку.** В performance-analyst: если у пользователя Postgres, грузим `dex-skill-ef-core`; если Mongo — `dex-skill-mongodb`. С pre-loaded skills это невозможно — либо всё, либо ничего.

4. **Согласованность с принципом «цель и результат».** Агент в фазе заявляет цель «проверь по чек-листу» и указывает какой именно чек-лист нужен здесь. Декларативно, без дублирования содержимого.

### Где в фазе указываются skills

В теле фазы, после атрибутов Goal/Output/Exit criteria/Gate. Формат:

```markdown
## Phase 2: Isolate

**Goal:** ...
**Output:** ...
**Exit criteria:** ...

В этой фазе загружай релевантные skills императивно через Skill tool:

- Всегда `dex-skill-dotnet-patterns:dotnet-patterns` — captive dependency, async void, IDisposable
- Всегда `dex-skill-async-patterns:async-patterns` — deadlock, fire-and-forget
- Если баг с данными/EF — `dex-skill-ef-core:ef-core`
- Если баг в коллекциях/LINQ — `dex-skill-linq-optimization:linq-optimization`
```

Правила:

- **Точное имя** в формате `{plugin-name}:{skill-name}`, без сокращений
- **Условия загрузки** формулируются декларативно («всегда» / «если X»)
- **Одно-двухсловная характеристика** зачем этот skill — чтобы LLM понимал релевантность
- **Без копии содержимого skill** — только ссылка

### Что делать, если skill недоступен

**Ситуация 1 — Skill tool недоступен в subagent.** Маловероятно (подтверждено экспериментально, что доступен), но возможно в других платформах или будущих версиях. Фаза должна иметь fallback: «если Skill tool недоступен — пропусти загрузку и укажи в отчёте, что фаза выполнена без skill-проверок».

**Ситуация 2 — Skill установлен, но контекст не требует его.** Это нормальная работа условной загрузки. Фаза пропускает этот конкретный skill без ошибки, выполняется с теми, условия которых выполнены.

**Ситуация 3 — Skill упомянут в агенте, но не установлен в маркетплейсе.** Ошибка автора агента. Ловится валидатором: проверка, что все упомянутые skill-имена существуют в `marketplace.json`.

## Что писать в теле фазы (и что нет)

**Писать можно:**

- Граничные условия и запреты («не переходи к Fix без подтверждённого reproduce»)
- Ссылки на skills, которые имеет смысл вызвать в этой фазе (императивно через Skill tool)
- Специфику предметной области, которая важна именно здесь
- Формат output, если он нестандартный
- Примеры того, что считается «достаточным» для exit criteria

**Писать нельзя:**

- Пошаговые процедуры («сначала прочитай файл, потом запусти grep, потом...»)
- Объяснения, как работает технология (Claude знает)
- Примеры кода (Claude сгенерирует под контекст)
- Документацию API и синтаксиса команд
- Всё, что можно найти в официальной документации

## Библиотека типовых фаз

Референсный список часто встречающихся фаз. Автор агента может брать их как есть или формулировать свои. Описания — декларативные (цель/результат), без процедур.

### Context Gathering

**Goal:** Убедиться, что у агента достаточно контекста для принятия решений: стек, scope, constraints.
**Output:** Зафиксированные факты о среде пользователя, влияющие на последующие фазы.
**Typical skip_if:** контекст явно передан в промпте.

### Reproduce

**Goal:** Подтвердить, что проблема воспроизводима, чтобы последующая диагностика опиралась на факты, а не предположения.
**Output:** Воспроизводящий тест или последовательность действий + env + ожидаемое vs фактическое.
**Typical mandatory:** yes для Diagnostician.

### Classify

**Goal:** Определить категорию проблемы или задачи, от которой зависит дальнейшая стратегия (какие signals собирать, какие skills загружать, какую ветку исполнения брать).
**Output:** Явная классификация в одну из предопределённых категорий + обоснование отсылкой к данным предыдущей фазы.
**Typical position:** после Reproduce или Diagnose, до Isolate / Execute.
**Когда нужна:** если агент покрывает несколько принципиально разных сценариев (ml-debugger: loss_not_decreasing / overfitting / memory_issue; operator: troubleshoot / optimize / build).

### Project Context Gathering

**Goal:** Понять стиль, паттерны и соглашения существующего проекта, в который попадёт результат работы агента. Частный случай Context Gathering для Creator'ов.
**Output:** Зафиксированные факты о проекте: архитектурный стиль, используемые паттерны, naming conventions, DI registration, похожие компоненты-образцы.
**Отличие от Context Gathering:** общий Context Gathering собирает стек и constraints (что снаружи), Project Context Gathering — стилевые и паттерновые соглашения (что внутри кода).
**Typical skip_if:** standalone-утилита, пустой проект, явное разрешение писать в своём стиле.

### Direct Analysis

**Goal:** Применить общие знания Claude для формирования первичных гипотез/находок.
**Output:** Список находок/гипотез с привязкой к коду (файл:строка).

### Skill-Based Deep Scan

**Goal:** Проверить код против специализированных чек-листов из skills.
**Output:** Дополнительные находки + дедупликация с Direct Analysis.
**Typical gate:** after Direct Analysis.

### Domain Priming

**Goal:** Понять словарь и контекст проекта до того, как агент начнёт давать оценки или рекомендации, чтобы найти доменные конфликты (`Project` как Entity vs как репозиторий) и опираться на проектные гайдлайны при оценке кода.
**Output:** Зафиксированный список ключевых сущностей домена (имена + смысл), используемых терминов bounded context, релевантных гайдлайнов проекта — любые документы проекта с конвенциями (CLAUDE.md, README, `CONVENTIONS.md` / `*guidelines*.md`, glossary, ADR по модели домена).
**Typical position:** перед Direct Analysis в Reviewer-агентах. Для Diagnostician — опционально, если баг связан с доменной семантикой.
**Typical mandatory:** yes для Reviewer — без понимания словаря пропускаются ubiquitous language нарушения.
**Когда нужна:** если агент даёт content-level оценки (нейминг, уместность абстракций, соответствие домену), а не только structure-level (синтаксис, форматирование).

### Non-Code Artifacts Audit

**Goal:** Систематически проверить изменения в non-code файлах: проектные манифесты сборки (`.csproj`, `package.json`, `pyproject.toml`, `go.mod`, `Cargo.toml`), centralized package management (`Directory.Packages.props`, lockfiles), MSBuild-инфраструктура (`.props`, `.targets`, `Directory.Build.*`), конфиги (`appsettings*.json`, `application.yml`), CI/CD pipelines.
**Output:** Список находок по non-code артефактам, отдельной категорией от находок по коду, с привязкой файл:строка.
**Typical position:** параллельно или после Skill-Based Deep Scan, до Cross-Linking.
**Typical mandatory:** yes для Reviewer — иначе пропускается целый класс ляпов (транзитивные зависимости, нарушение CPM, дубли инфраструктуры).

### Cross-Linking

**Goal:** Связать изолированные находки между собой, чтобы вместо `N` отдельных мелких пунктов выдать дерево «root cause → symptoms» с единым решением, покрывающим связанную группу.
**Output:** Граф/дерево связанных находок: для каждой группы — общий root cause + единое решение, для каждой stand-alone находки — явная пометка «не связана с другими».
**Typical position:** после всех scan-фаз (Direct Analysis + Skill-Based Deep Scan + Non-Code Artifacts Audit), до Synthesis / Report.
**Typical mandatory:** yes для Reviewer — без cross-linking агент выдаёт расфокусированный список из 20 мелких пунктов вместо 5 значимых проблем.
**Когда нужна:** если scan-фазы производят >5 находок, между которыми возможны связи (например: «flaky внешний клиент» + «god-handler с >2 внешними вызовами» + «отсутствие resilience» = одно решение, вынести в отдельный subscriber с retry).

### Content-Level Pass

**Goal:** Проверить уместность и семантику решений, не только их формат / синтаксис: для каждого `LogInformation` — обоснование уровня («это нужно оператору в 3 ночи или только разработчику?»), для каждого имени переменной / типа — соответствие bounded context, для каждого `try/catch` — оправданность типа исключения.
**Output:** Список content-level находок с обоснованием «почему это смущает» (не «формат нарушен», а «уровень / семантика не та»).
**Typical position:** после Skill-Based Deep Scan, в той же итерации с Non-Code Artifacts Audit.
**Typical mandatory:** optional — добавляется, когда structure-level проверки уже прошли и нужна более глубокая оценка.
**Отличие от Skill-Based Deep Scan:** scan ищет ловушки по чек-листу (formal pattern matching), content-level задаёт качественные вопросы (semantic judgement).

### Severity Calibration

**Goal:** Подстроить severity и формулировку каждой находки под фактический контекст проекта: project stage (pre-alpha / MVP / production), audience (internal / beta / public), data sensitivity (demo / non-PII / PII / financial), compliance scope (нет / SOC2 / GDPR / HIPAA), наличие auth и multi-tenancy. Одна и та же находка (IDOR, отсутствие resilience, отсутствие тестов) имеет разную severity в разных контекстах.
**Output:** Каждая находка снабжена калиброванной severity + двумя формулировками: minimum fix (что делать сейчас в текущем контексте) и ideal fix (как было бы в зрелом проекте).
**Typical position:** после всех scan-фаз и Cross-Linking, до Tech Debt Classification и Output Labeling.
**Typical mandatory:** yes для Reviewer — без калибровки отчёт либо ложно паникует («всё горит»), либо ложно расслабляет.
**Важно:** калибровка меняет уровень и формулировку, **не вычёркивает** находку. IDOR в pre-auth проекте остаётся в отчёте как correctness invariant + architectural preparation, не исчезает.
**Fallback:** если контекст не известен (нет в CLAUDE.md / README) — либо запросить у пользователя, либо сделать explicit assumption и пометить в отчёте: `[Assumption: production стадия. Если pre-alpha — severity C1 переформулируется как ...]`.

### Tech Debt Classification

**Goal:** Для каждой находки, выглядящей как отклонение от best practice, определить статус: документированный tech debt с планом, тихий tech debt без документации, или архитектурная ошибка. От этого зависит формулировка рекомендации (требовать TODO+Jira / создать тикет / фикс сейчас).
**Output:** Для каждой находки — категория `documented-tech-debt` / `silent-tech-debt` / `error` + обоснование выбора (какие маркеры найдены или не найдены).
**Typical position:** после Severity Calibration, до Output Labeling.
**Typical mandatory:** yes для Reviewer.
**Дефолт — подсвечивать.** Если в проектной документации (CLAUDE.md, README, ADR) или в самом MR / коммите автор **явно не указал**, что решение временное или известный техдолг, находка попадает в категорию `silent-tech-debt` или `error` и **подсвечивается**. Не предполагать «возможно это сознательное решение» при отсутствии доказательств.
**Маркеры accepted tech debt** (для перевода в `documented-tech-debt`): TODO в коде с Jira-ссылкой на тикет миграции; ADR с обоснованием временного решения и планом замены; явная пометка в `CLAUDE.md` / `README`; атрибут `[Obsolete]` / `@deprecated` / эквивалент; явная фраза в описании MR или коммите.
**Формулировка для silent tech debt:** «Решение не задокументировано как временное → требуется либо фикс, либо явная фиксация статуса (TODO + Jira, ADR)». Не «возможно это так задумано».

### Systemic vs Specific Triage

**Goal:** Отделить находки, специфичные для текущего MR, от системных проблем команды/проекта (отсутствие тестов, плохой нейминг, log flooding, повторяющиеся в каждом MR). Системную проблему бесполезно подавать как блокер конкретного MR — она не закроется, автор ответит «у нас так везде».
**Output:** Каждая находка помечена `mr-specific` или `systemic`. Для systemic — рекомендация на уровне процесса (DoD, CI-gate, ADR, тренинг команды, обновление CLAUDE.md), не как блокер MR.
**Typical position:** после Tech Debt Classification, до Output Labeling.
**Typical mandatory:** optional — добавляется, когда у агента есть контекст последних 3-5 MR проекта или явные указания на повторяющиеся проблемы.
**Red flags systemic:** проблема встречается в множестве MR, в репо нет инфраструктуры обнаружения (нет линтера / теста / CI-gate), реакция автора «да, так у нас везде», отсутствие правила в CLAUDE.md.
**Формулировка systemic находки:** «Системная проблема (встречается в N MR). Этот MR не блокируется. Рекомендуется на уровне процесса: [DoD / CI-gate / ADR / тренинг]».

### Output Labeling

**Goal:** Каждой находке присвоить явную метку действия, чтобы пользователь понимал что блокирует мёрдж, а что нет — без неё отчёт превращается в плоский «список замечаний», по которому непонятно как принимать решение о мёрдже.
**Output:** Каждая находка снабжена одной из меток по цветной шкале усиления риска + одно предложение обоснования:

| Метка | Значение | Действие |
| ----- | -------- | -------- |
| 🟢 | Accepted / OK | Осознанное решение с маркером (TODO+Jira, ADR, `[Obsolete]`, пометка в проекте). Не требует действий. |
| 🟡 | Minor — TODO in code | Замёрджить с TODO-комментарием + Jira-ссылкой. |
| 🟠 | Follow-up ticket | Замёрджить, но создать тикет в backlog. |
| 🔴 | Block merge | Не мёрджить, пока не исправлено. |
| 🟣 | Needs discussion | Требует обсуждения с командой / stakeholder'ом, решение не в компетенции ревьюера. |

Шкала 🟢→🟡→🟠→🔴 — усиление риска от accepted к блокеру. 🟣 стоит вне шкалы — это запрос на решение, а не уровень риска.

**Typical position:** последняя фаза перед Report.
**Typical mandatory:** yes для Reviewer.

**Соответствие меток и категорий Tech Debt Classification:**

- `error` без obvious workaround → 🔴
- `silent-tech-debt` с понятным минимальным фиксом → 🟡 или 🟠 (🟡 если фикс = добавить TODO+Jira прямо сейчас, 🟠 если нужен отдельный тикет в backlog)
- `documented-tech-debt`, соответствующий правилам проекта → 🟢
- Неоднозначное архитектурное решение, где ревьюер не имеет достаточного контекста → 🟣

Метка 🟢 ставится **только** при наличии маркеров accepted tech debt из Tech Debt Classification, не «по умолчанию». Если маркеров нет — минимум 🟡.

### Hypothesize

**Goal:** Сформулировать проверяемое утверждение о причине/следствии.
**Output:** Гипотеза в явной форме: «если X, то Y, потому что Z».

### Test Hypothesis

**Goal:** Проверить гипотезу экспериментом (тест, запрос к системе, инспекция).
**Output:** Подтверждение или опровержение + доказательство.

### Understand Requirements

**Goal:** Убедиться, что задача/требования понятны однозначно.
**Output:** Переформулированные требования + выявленные неоднозначности/пропуски.

### Generate

**Goal:** Создать запрошенный артефакт (код, тест, документ, схему).
**Output:** Артефакт.

### Validate

**Goal:** Проверить, что сгенерированный артефакт корректен (компилируется, тесты проходят, соответствует требованиям).
**Output:** Подтверждение корректности или список проблем.
**Typical mandatory:** yes для Creator.

### Propose Alternatives

**Goal:** Представить несколько вариантов решения для выбора.
**Output:** N альтернатив с trade-off'ами.

### Decide

**Goal:** Выбрать вариант из альтернатив с обоснованием.
**Output:** Принятое решение + обоснование + отвергнутые альтернативы.

### Document

**Goal:** Зафиксировать решение или артефакт в форме, пригодной для долговременного хранения (ADR, спецификация, readme).
**Output:** Документ.

### Diagnose (Operator)

**Goal:** Понять текущее состояние инфраструктуры/среды перед любыми действиями.
**Output:** Снимок состояния, релевантный задаче.

### Branch

**Goal:** На основе диагностики выбрать сценарий дальнейших действий.
**Output:** Выбранный сценарий + обоснование выбора.

### Execute

**Goal:** Применить изменения к системе.
**Output:** Результат выполнения команд + возврат значений/статусов.
**Typical gate:** explicit confirmation.

### Verify

**Goal:** Подтвердить, что применённые изменения дали ожидаемый эффект и не создали регрессий.
**Output:** Подтверждение работоспособности или список новых проблем.
**Typical mandatory:** yes для Operator и Diagnostician.

### Fix

**Goal:** Применить минимальное изменение, закрывающее установленный root cause.
**Output:** Диф + failing test → passing test.
**Typical gate from Isolate:** hard — root cause подтверждён.

### Prioritize

**Goal:** Упорядочить элементы (задачи, варианты, находки) по явному критерию.
**Output:** Упорядоченный список + критерий сортировки.

### Report / Present

**Goal:** Представить результаты работы в форме, удобной для пользователя.
**Output:** Финальный отчёт в заданном формате.

### Gap Analysis

**Goal:** Выявить, что покрыто и что не покрыто относительно ожидаемого набора.
**Output:** Таблица покрытия + список пробелов.

## Рецепты для типовых ролей

Типы из CLAUDE.md (Analyst, Diagnostician, Creator, Designer, Operator, Planner) — это **рецепты композиции**, не жёсткие контракты. Автор агента может использовать их как стартовую точку или собрать свою композицию.

### Analyst

```text
[Context Gathering?] → Direct Analysis → Skill-Based Deep Scan → Report
```

Часто Context Gathering опционален. Skill-Based Scan — обычно mandatory для полноты покрытия.

### Diagnostician

```text
Reproduce → [Classify?] → Direct Analysis → Skill-Based Deep Scan → Hypothesize → Fix → Verify
```

Reproduce и Verify — mandatory. Между Isolate (= Direct + Skill) и Fix — hard gate. Classify добавляется, если агент покрывает несколько принципиально разных категорий проблем (ML debugger с loss / overfitting / memory) — от категории зависит стратегия Isolate и выбор skills.

### Creator

```text
Understand Requirements → [Project Context Gathering?] → Generate → Validate
```

Validate — mandatory. Без неё агент выдаёт непроверенный артефакт. Project Context Gathering добавляется, если результат работы попадает в существующий проект со своими стилевыми соглашениями — пропускается для standalone-утилит.

### Designer

```text
[Context Gathering?] → Analyze Constraints → Propose Alternatives → Decide → [Document?]
```

Document часто опционален — ADR пишется только для значимых решений.

### Operator

```text
Diagnose → Branch → Execute → Verify
```

Diagnose и Verify — mandatory. Execute обычно требует explicit confirmation для destructive операций (read-only команды без подтверждения). Branch практически всегда присутствует — Operator типично покрывает несколько сценариев (troubleshoot / optimize / build / routine operations), и без явного ветвления агент путается или пропускает фазы. Если Operator действительно однозадачный — Branch можно свернуть, но это редкий случай.

### Planner

```text
Gather → Analyze → Prioritize → Present
```

Минимум ветвлений, линейный pipeline.

### Reviewer

```text
Domain Priming → Direct Analysis → Skill-Based Deep Scan → Non-Code Artifacts Audit
  → Content-Level Pass → Cross-Linking → Severity Calibration
  → Tech Debt Classification → Systemic vs Specific Triage → Output Labeling → Report
```

Reviewer — частный случай Analyst, расширенный фазами специально под code review. Поток делится на три блока:

1. **Сбор находок** (Domain Priming → Direct Analysis → Skill-Based Deep Scan → Non-Code Artifacts Audit → Content-Level Pass) — что нашли.
2. **Структурирование** (Cross-Linking) — как находки связаны между собой.
3. **Калибровка и оформление** (Severity Calibration → Tech Debt Classification → Systemic vs Specific Triage → Output Labeling) — как подать находки пользователю в форме, по которой он может принимать решения.

**Mandatory-фазы:** Domain Priming, Non-Code Artifacts Audit, Cross-Linking, Severity Calibration, Tech Debt Classification, Output Labeling. Без них агент либо пропускает класс находок (доменные конфликты, ляпы в `.csproj`), либо даёт расфокусированный отчёт (плоский список без связей и без меток действий). Content-Level Pass и Systemic vs Specific Triage — обычно mandatory, опциональны только когда ревью узко-формальное (например, только compile-check) или контекст последних MR недоступен.

**Skill-Based Deep Scan** загружает skills по стеку проекта (EF Core, async patterns, OWASP, resilience, csproj-hygiene и т.п.) — условно, не все подряд.

**Report** даёт verdict (APPROVE / REQUEST_CHANGES / NEEDS_DISCUSSION) + сгруппированные находки из Cross-Linking с метками из Output Labeling. Не плоский список замечаний.

**Дефолт «подсвечивать»:** Tech Debt Classification работает по презумпции «находка подсвечивается, если автор не задокументировал её как accepted tech debt». Не предполагать «возможно это сознательное решение» — это размывает результат ревью и даёт автору лазейку игнорировать замечания.

**Почему Reviewer выделен в отдельный рецепт, а не оставлен под Analyst:** реальный code review требует доменного контекста (без него ubiquitous language нарушения не видны), систематического покрытия non-code артефактов, cross-linking (Analyst выдаёт находки изолированно), калибровки под зрелость проекта и явных меток действий. Эти фазы — критичные защиты от типичных failure mode'ов code-review-агентов, выявленных на реальных PR: расфокусированный список без приоритетов, ложная паника по best-practice-нарушениям, не блокирующим production, или наоборот тихий accept без подсветки.

### Parallel Reviewer Orchestration

Когда несколько Reviewer-агентов запускаются параллельно (разные фокусы — code quality / security / performance / architecture), результаты нужно сводить через **Calibration Pass** оркестратора, не просто конкатенировать:

```text
[Agent A] ─┐
[Agent B] ─┤→ Cross-Agent Deduplication → Calibration Pass → Final Report
[Agent C] ─┤
[Agent D] ─┘
```

**Cross-Agent Deduplication** — одну и ту же находку, поднятую несколькими агентами, объединить в один пункт.

**Calibration Pass** — отдельная фаза, которая для каждой `CRITICAL` / `HIGH` находки задаёт три вопроса: (1) это actual blocker в этом проекте или универсальный best practice? (2) есть ли локальные причины (pre-auth, временная интеграция, MVP)? (3) какой минимальный фикс vs идеальный?

**Зачем отдельный pass:** параллельные агенты создают **видимость консенсуса** (если 3 из 4 подтвердили — кажется, что важно), но все 4 могут иметь общий слепой зон (например, все предполагают production-стадию). Калибровка ловит это.

## Анти-паттерны

Чего делать нельзя при описании фаз.

### 1. Процедурное описание фазы

```markdown
### Phase: Analyze

1. Запусти grep X
2. Запусти grep Y
3. Открой файл Z
```

Неправильно. Claude сам решит, какие инструменты применять.

### 2. Exit criteria через внутреннее состояние

```markdown
**Exit criteria:** Агент понял проблему.
```

Неправильно. «Понял» — не наблюдаемо. Надо: «гипотеза сформулирована в отчёте с привязкой к файлу:строке».

### 3. Output, который нигде не используется

Если output одной фазы не читается следующими — либо фаза лишняя, либо output сформулирован как для галочки.

### 4. Mandatory-фаза без обоснования

Помечать mandatory = заявлять, что без этой фазы агент не работает корректно. Если обоснования нет — скорее всего, фаза не mandatory.

### 5. Gate без условия

```markdown
**Gate from Phase 1:** Phase 1 завершена.
```

Это не gate, это тривиальное следование. Gate — это **дополнительное условие** сверх простого завершения предыдущей фазы.

### 6. Документация API в теле фазы

Всё, что есть в официальной документации технологии, — не место в агенте. Claude знает. Если какая-то грабля неочевидна — это anti-pattern для skill, не для агента.

### 7. Дублирование содержимого skill в агенте

Если фаза описывает то же, что уже есть в skill (копирует раздел, пересказывает чек-лист) — удалить из агента, оставить только ссылку. Содержимое skill актуализируется отдельно и переиспользуется разными агентами — копия в агенте разойдётся с оригиналом.

### 8. Загрузка всех skills без условий

```markdown
## Phase 2: Skill-Based Scan

Загрузи все skills: dex-skill-ef-core, dex-skill-redis, dex-skill-logging, dex-skill-observability, dex-skill-mongodb.
```

Неправильно. Безусловная загрузка всех skills = то же самое что pre-load через frontmatter, только дороже по токенам и контексту. Условия загрузки должны быть конкретными: «если в стеке X», «если баг класса Y», «всегда для этой категории задач».

### 9. Использование skills вместо базовых знаний Claude

```markdown
## Phase 1: Direct Analysis

Загрузи skill dex-skill-dotnet-patterns, чтобы проверить SOLID.
```

Неправильно по двум причинам. Во-первых, Phase 1 (Direct Analysis) — это явно фаза **без** skills, её смысл в применении общих знаний. Во-вторых, SOLID Claude знает из training data — skill не нужен для базовых концепций. Skill подключается только там, где грабля **не очевидна** и требует специализированной проверки.

## Пример: полный агент в парадигме фреймворка

Небольшой Diagnostician-агент (`bug-hunter`), написанный по правилам фреймворка. Показывает композицию из фаз, hard gate между Isolate и Fix, mandatory-фазы, императивную загрузку skills в теле фазы, fallback и explicit confirmation. Это демонстрационный пример — реальные агенты могут быть больше.

```markdown
---
name: bug-hunter
description: Поиск и исправление багов в .NET — root cause analysis, отладка exceptions, deadlock, N+1, memory leak. Триггеры — find bug, debug, error, exception, NullReferenceException, stack trace, crash
tools: Read, Edit, Bash, Grep, Glob, Skill
---

# Bug Hunter

Специалист по поиску и исправлению багов в .NET. Каждая диагностика проходит фиксированный набор фаз с hard gate между изоляцией и исправлением — защита от фикса по гипотезе.

## Phases

Reproduce → Isolate → Fix → Verify. Reproduce и Verify обязательны. Fix блокируется hard gate от Isolate.

## Phase 1: Reproduce

**Goal:** Убедиться, что баг воспроизводим — последующая диагностика опирается на факты, а не предположения.

**Output:** Воспроизводящий тест ИЛИ фиксированная последовательность шагов + минимальный набор входных данных + зафиксированное environment. ИЛИ явное заключение «не воспроизводится».

**Exit criteria:** Баг воспроизведён (есть артефакт) либо однозначно квалифицирован как невоспроизводимый.

**Mandatory:** yes — без reproduce диагностика превращается в угадывание по stack trace.

**Fallback:** если воспроизвести не удалось — остановиться и запросить у пользователя логи, env, последовательность действий. НЕ переходить к Isolate.

## Phase 2: Isolate

**Goal:** Определить root cause — конкретное место в коде и причинно-следственная связь.

**Output:** Файл и строка root cause + объяснение почему это приводит к наблюдаемому симптому + список related patterns, найденных в коде вокруг этого места.

**Exit criteria:** Гипотеза root cause сформулирована в явной форме «если X в файл.cs:42, то проявляется Y, потому что Z» и подтверждена либо чтением кода, либо выполнением repro из Phase 1.

В этой фазе загружай релевантные skills императивно через Skill tool:

- Всегда `dex-skill-dotnet-patterns:dotnet-patterns` — captive dependency, async void, IDisposable, double fault
- Всегда `dex-skill-async-patterns:async-patterns` — deadlock, fire-and-forget, ConfigureAwait
- Если баг с данными/EF — `dex-skill-ef-core:ef-core`
- Если баг в коллекциях/LINQ — `dex-skill-linq-optimization:linq-optimization`

## Phase 3: Fix

**Goal:** Применить минимальное изменение, закрывающее root cause.

**Gate from Phase 2 (hard):** root cause подтверждён — файл, строка, причинно-следственная связь зафиксированы. Без этого — вернуться в Phase 2, не писать фикс «по интуиции».

**Gate (explicit confirmation):** план изменения показан пользователю и одобрен.

**Output:** Диф кода + failing test из Phase 1, который теперь проходит.

**Exit criteria:** Failing test перешёл в passing без изменений самого теста.

Фикс — минимальный. Не рефакторить попутно. Если видишь соседние проблемы — выписать в отчёт, но не трогать.

## Phase 4: Verify

**Goal:** Подтвердить, что фикс не создал регрессий и не изменил поведение за пределами бага.

**Output:** Результат прогона всей тестовой suite + явное подтверждение, что изменение локализовано.

**Exit criteria:** Все тесты проходят. Если появились новые failing тесты — вернуться в Phase 3.

**Mandatory:** yes — без verify риск скрытой регрессии.

## Boundaries

- Один баг = один фикс. Рефакторинг смежного кода — отдельной задачей.
- Если нужны внешние инструменты (debugger attached, memory dump, profiler) — эскалировать пользователю, не пытаться заменить их grep'ом.
- Не менять тесты, чтобы они начали проходить. Это анти-фикс.
```

Обрати внимание: в теле фазы **нет пошаговых процедур**. Есть цель, output, exit criteria, gate'ы, граничные условия и ссылки на skills. Claude сам решит, как именно воспроизводить баг, каким grep'ом искать паттерны и как писать failing test — у него это в training data.

## Соответствие текущим проектным правилам

Фреймворк совместим с существующими правилами из CLAUDE.md:

- **Skills как ловушки, не документация** — фаза аналогично описывает контракт, не процедуру
- **Two-Pass Architecture** — частный случай фреймворка: Analyst с фазами Direct Analysis + Skill-Based Scan
- **Типизация агентов** — становится каталогом рецептов композиции, не жёстким контрактом
- **Императивная загрузка skills** — фаза ссылается на relevant skills, Claude загружает через Skill tool в процессе работы

## Self-check перед коммитом

Чек-лист для автора агента. То же самое проверяет `node tools/validate-agent.js <path>`, но в человекочитаемой форме.

- [ ] Frontmatter: есть `name`, `description` (с триггерами), `tools` (с `Skill`)
- [ ] Frontmatter: **нет** `skills:` (проектное решение маркетплейса — императивная загрузка экономит контекст)
- [ ] Frontmatter: **нет** `allowed-tools:` (для агентов используется `tools:` / `disallowedTools:`)
- [ ] Есть хотя бы одна секция `## Phase N: <Имя>`
- [ ] Каждая фаза имеет `**Goal:**` и `**Exit criteria:**`
- [ ] Exit criteria — observable (не «агент понял», а конкретный артефакт)
- [ ] Mandatory-фазы обоснованы — после `**Mandatory:** yes` есть «зачем»
- [ ] Hard gate'ы формулируют условие, а не «Phase N завершена»
- [ ] Skills указаны условно (`если X — загрузи Y`), не «загрузи все»
- [ ] Нет нумерованных списков-процедур в теле фаз
- [ ] Нет документации API, boilerplate-кода, объяснений базовых концепций
- [ ] Нет дублирования содержимого skills
