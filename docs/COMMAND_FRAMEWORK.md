# Command Framework

Методичка для создания команд (slash-commands) в Claude Code marketplace. Используется человеком или Claude при работе над плагинами.

Команда — **точечное действие**, вызываемое пользователем через `/command-name`. В отличие от агента (многофазный workflow) и skill (справочник ловушек), команда решает **одну задачу за один вызов** в текущем контексте разговора.

## Когда нужна команда, а когда агент

| Признак | Команда | Агент |
| ------- | ------- | ----- |
| Вызывается | Пользователем явно: `/name` | Claude Code автоматически по semantic match |
| Контекст | Текущий разговор | Отдельный sub-agent |
| Масштаб | Одно действие или короткий сценарий | Многофазный workflow с gate'ами |
| Состояние | Не держит между вызовами | Живёт пока работает |
| Примеры | `/build`, `/test`, `/metrics` | bug-hunter, architect, code-reviewer |

**Правило:** если задача требует больше одной фазы с gate'ами — это агент, не команда. Если команда разрастается до 200+ строк с несколькими сценариями и развёрнутым output — переосмыслить: возможно, это должен быть агент.

## Ключевой принцип: команда = цель + триггер, не скрипт

Тот же принцип, что в [AGENT_FRAMEWORK.md](AGENT_FRAMEWORK.md): Claude знает КАК делать. Команда задаёт **что** и **в каком формате** вернуть результат. Не нужно расписывать каждый bash-шаг — Claude сам вызовет `dotnet build`, `dotnet test`, `docker ps` когда поймёт цель.

### Анти-паттерн vs правильно

**Анти-паттерн (скрипт):**

```markdown
# /build

1. Найти solution:
   sln=$(find . -maxdepth 2 -name "*.sln" | head -n 1)
2. Запустить сборку:
   dotnet build "$sln" --configuration Release
3. Если ошибка CS0246 — проверить using
4. Если ошибка CS0103 — проверить опечатки
5. Если ошибка CS8600 — добавить null-check
```

Проблемы: Claude знает как найти sln, как запустить dotnet build, как анализировать ошибки. Скрипт дублирует training data и ломается при отклонениях (monorepo с несколькими sln, Directory.Build.props, нестандартная структура).

**Правильно (цель):**

```markdown
# /build

Собрать .NET проект и проанализировать результат.

**Goal:** Успешная сборка проекта. Если сборка падает — анализ ошибок с конкретными предложениями по исправлению.

**Output:** Результат сборки + для каждой ошибки/warning: файл, строка, причина, предложение fix.
```

Claude сам найдёт sln/csproj, выберет конфигурацию, запустит сборку, проанализирует output. Ему не нужен bash-скрипт для этого.

## Структура команды

### Путь в репозитории

```text
plugins/specialists/<category>/<plugin-name>/
└── commands/
    └── <command-name>.md
```

Пользователь вызывает: `/command-name` или `/command-name аргументы`.

### Frontmatter

```yaml
---
description: Что команда делает (одно предложение)
allowed-tools: Bash, Read, Grep, Glob
argument-hint: [описание аргументов] (опционально)
---
```

**Обязательно:**

- `description` — короткое описание. Claude Code показывает его в списке доступных команд.

**Опционально:**

- `allowed-tools` — ограничение tools (если команда не должна, например, писать файлы)
- `argument-hint` — подсказка что передать: `[stack-trace]`, `[migration-name]`, `[story-id]`

### Тело команды

Минимальная структура:

```markdown
# /command-name

Одно предложение — что команда делает.

**Goal:** Что должно быть достигнуто.

**Output:** В каком формате результат.
```

Расширенная структура (для команд со сценариями):

```markdown
# /command-name

Одно предложение — что команда делает.

**Goal:** Что должно быть достигнуто.

**Scenarios:**
- Если X — сделать Y
- Если Z — сделать W

**Output:** В каком формате результат.

**Constraints:** Ограничения и запреты.
```

## Что писать (и что нет)

**Писать:**

- **Goal** — что команда должна достичь (одна фраза)
- **Output format** — как выглядит результат (таблица, отчёт, файл)
- **Scenarios** — если команда покрывает несколько случаев, перечислить их
- **Constraints** — что команда НЕ должна делать, когда остановиться
- **argument-hint** — что пользователь передаёт как аргумент

**Не писать:**

- Bash-скрипты с конкретными командами (Claude знает `dotnet build`, `docker ps`, `git diff`)
- Справочники error codes (`CS0246 — тип не найден`) — Claude это знает
- Справочники PromQL / KQL / SQL запросов — Claude сгенерирует под стек
- Примеры вывода на 50+ строк с ASCII-таблицами — достаточно описать формат
- Туториалы «как работает технология» — Claude знает

## Размер

- **Цель:** 20-50 строк
- **Допустимо:** до ~80 строк
- **Сигнал тревоги:** >100 строк — команда скорее всего содержит справочник или процедуру, которую нужно вырезать. Или это не команда, а агент.

## Когда команда вырастает в агент

Признаки того, что команда должна быть агентом:

- Больше одного сценария с принципиально разной логикой
- Нужны gate'ы — «не делай X пока не подтвердил Y»
- Нужна mandatory verify фаза после действия
- Нужна загрузка skills для deep check
- Размер > 100 строк даже после вычистки справочников

Пример: `/analyze-story` в текущем виде (285 строк, полноценный workflow с EP/BVA/Decision Table, генерация файлов, RTM) — это по факту **агент**, а не команда. Его стоит мигрировать в `agents/test-analyst.md` как набор фаз.

## Связь с агентами и skills

- **Команда** может **вызвать агента**: `/design` → запускает architect sub-agent
- **Команда** не загружает skills — это механизм агентов (sub-agent контекст)
- Если команде нужны skill-проверки — это сигнал что она должна быть агентом
- Команда может **дополнять агента**: агент `coding-assistant` реализует workflow, а команда `/build` — отдельное быстрое действие из того же плагина

## Анти-паттерны

### 1. Bash-скрипт вместо цели

```markdown
sln=$(find . -maxdepth 2 -name "*.sln" | head -n 1)
dotnet build "$sln" --configuration Release
```

Неправильно. Claude найдёт и соберёт проект сам. Опиши цель, не процедуру.

### 2. Справочник error codes

```markdown
- CS0246 — тип не найден, проверить using
- CS0103 — имя не существует, проверить опечатки
- CS8600 — nullable warning, добавить null-check
```

Неправильно. Claude знает все error codes C# из training data. Если есть специфичная для проекта ловушка (например, «CS0246 на наших internal пакетах = сломался NuGet feed») — это skill, не команда.

### 3. Справочник запросов

```markdown
# Request rate
rate(http_requests_total[5m])
# Latency P95
histogram_quantile(0.95, rate(http_request_duration_seconds_bucket[5m]))
```

Неправильно. Claude знает PromQL. Опиши какие метрики нужны и в каком формате, Claude сгенерирует запросы.

### 4. Развёрнутый output template на 50+ строк

Таблицы ASCII-art, дашборды с рамками — это не нужно расписывать построчно. Достаточно: «Output: таблица метрик (endpoint, P95, calls/min), топ-3 медленных, alerts, рекомендации».

### 5. Команда-агент

Команда на 200+ строк с несколькими сценариями, генерацией файлов и RTM-матрицей — это агент, притворяющийся командой. Мигрировать в `agents/`.

## Пример: /build в парадигме фреймворка

```markdown
---
description: Сборка .NET проекта с анализом ошибок
allowed-tools: Bash, Read, Grep, Glob
---

# /build

Собрать .NET проект и проанализировать результат.

**Goal:** Успешная сборка. При ошибках — анализ каждой с предложением fix.

**Output:** Результат сборки + для каждой ошибки/warning: файл, строка, причина, конкретное предложение исправления. Для warnings — приоритизация (какие критичны, какие можно отложить).

**Constraints:**

- Не исправлять ошибки автоматически — только предлагать
- Если sln/csproj не найден — спросить пользователя, не угадывать
- Для ошибок NuGet restore — предложить `dotnet restore` перед повторной сборкой
```

20 строк вместо 52. Claude знает всё остальное.

## Self-check перед коммитом

- [ ] Frontmatter: есть `description`
- [ ] Есть **Goal** (что команда достигает)
- [ ] Есть **Output** (формат результата)
- [ ] Нет bash-скриптов с конкретными командами (Claude знает)
- [ ] Нет справочников error codes / PromQL / SQL
- [ ] Нет развёрнутых output templates (>20 строк)
- [ ] Размер < 80 строк
- [ ] Если > 100 строк — переосмыслить: не агент ли это?
- [ ] `argument-hint` указан, если команда принимает аргументы
