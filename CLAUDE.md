# Claude Code Marketplace

> **Нормативный дом правил разработки**: как писать плагины, ревьюить, проверять перед коммитом - не обзор. Обзор проекта (концепция, 3 уровня, структура папок, тех-стек, MCP по ролям) - [README.md](README.md); устройство фреймворков - `docs/*_FRAMEWORK.md`.

Источники истины: `.claude-plugin/marketplace.json` - каталог всех плагинов; `tools/validate-{agent,skill,command,bundle}.js` - валидаторы; `docs/` - фреймворки и гайды.

## Каталог плагинов

Полный перечень плагинов с описаниями и версиями - **единственный источник истины: `.claude-plugin/marketplace.json`**. Структуру уровней (Skills / Utilities / Specialists / Bundles) и обзор - см. [README.md](README.md); поштучный список здесь не дублируем (растёт и устаревает молча). Нормативные исключения, которые не выводятся из README:

- **Тип skill - валидируемая ось** (`trap-skill` / `process-skill`): валидатор проверяет файл по-разному. По умолчанию trap-skill (триады «Плохо/Правильно/Почему»). Process-skill (правило-оркестрация, не каталог граблей) освобождён от trap-эвристик, но не от содержания (`process-empty` - нужна таблица-реестр или >=2 H2) и **требует регистрации в allowlist `PROCESS_SKILLS` в `tools/validate-skill.js`** - маркер `<!-- skill-type: process -->` в теле для читателя, валидатор его не парсит. Типология и калибровка - [docs/SKILL_FRAMEWORK.md](docs/SKILL_FRAMEWORK.md) («Два типа skill»).
- **delivery/ и review/ специалисты - языко-агностичны**: глубину даёт by-stack загрузка `dex-skill-<стек>-*`, добавление стека их не меняет ([docs/AGENT_SPECIALIZATION.md](docs/AGENT_SPECIALIZATION.md)).
- **Состав bundle живёт в `bundle.json` `includes[]`, НЕ в `plugin.json`**: Claude Code строго валидирует `plugin.json` и тихо ломает плагины с неизвестными полями.

## Структура плагина

Все плагины: `.claude-plugin/plugin.json` (манифест) + содержимое по типу - Skill: `skills/<name>/SKILL.md`; Specialist: `agents/<name>.md` (+ опц. `commands/<name>.md`); Bundle: `bundle.json` (`{"includes": [...]}`) + `README.md`; Utility: `hooks/` и/или `commands/` (нотификаторы, CLI-обёртки).

## Гайдлайны: как писать Skills / Specialists / Commands

Устройство (формат, анти-паттерны, рецепты, валидатор) - в фреймворках, читать перед созданием/правкой артефакта:

- **Skill** - [SKILL_FRAMEWORK.md](docs/SKILL_FRAMEWORK.md). Суть: skill - ловушки и anti-patterns, не документация API; формат «Плохо / Правильно / Почему», 3-5 строк на ловушку, 80-120 строк на skill.
- **Specialist (агент)** - [AGENT_FRAMEWORK.md](docs/AGENT_FRAMEWORK.md) (+ выбор «общий vs стек-специфичный» - [AGENT_SPECIALIZATION.md](docs/AGENT_SPECIALIZATION.md), карта «слот процесса -> агент» - [DEV_PROCESS_COVERAGE.md](docs/DEV_PROCESS_COVERAGE.md)). Суть: агент = workflow через фазы (декларативный контракт goal/output/exit/gate, не процедура); skills грузятся императивно через Skill tool в фазах, не pre-loaded.
- **Command** - [COMMAND_FRAMEWORK.md](docs/COMMAND_FRAMEWORK.md). Суть: команда = точечное действие (`/build`, `/test`); Goal + Output format, не bash-скрипт; 20-50 строк.

## Конвенции

### Именование
- **Skills**: `dex-skill-{name}`. Для стек-специфичных skills `{name}` = `{стек}-{тема}` (`dotnet-ef-core`, `ts-patterns`, `python-pytorch`) - единый стековый префикс обязателен для by-stack loading (см. ниже). Язык-нейтральные skills (docker, solid, ddd) префикса стека не несут.
- **Specialists**: `dex-{domain}-{role}` или `dex-{name}-specialist`
- **Bundles**: `dex-bundle-{role}`
- **Utilities**: `dex-{name}`

#### Имя агента (frontmatter `name`): стек в имени - только у стек-специфичных

Имя плагина-специалиста (директория `dex-*`) и имя **агента** в его frontmatter (`name`) - разные вещи. Правило для `name`:

- **Стек-специфичный агент** (специфика стека живёт в теле артефакта - идиомы языка, библиотеки экосистемы) -> стек в имени: `dotnet-coder`, `dotnet-test-writer`, `dotnet-ef-specialist`, `ts-test-writer`, `ts-fullstack-assistant`. Generic-имя (`coding-assistant`, `test-writer`) для стекового агента запрещено - оно не отличает агента от общего и даёт коллизию при росте стеков (два `test-writer`'а).
- **Общий (языко-агностичный) агент** (стек грузится через skills по реестру, тело нейтрально) -> чистая роль без стека: `debugger`, `feature-implementer`, `security-reviewer`, `mr-reviewer`, `self-reviewer`.

Имя файла агента совпадает с `name`. Rename агента = major bump (директория плагина и `plugin.json` `name` при этом не меняются - `bundle.json` ссылается на директорию, а не на агента). Критерий «общий vs стек-специфичный» - [docs/AGENT_SPECIALIZATION.md](docs/AGENT_SPECIALIZATION.md).

#### By-stack loading: общий агент грузит профильные skills по реестру

Общие агенты не хранят зашитый список «если .NET -> skill X». Они загружают process-skill `dex-skill-stack-registry` (реестр стек->манифест->префикс + правило «стек × тема»), определяют стек проекта по манифесту и фильтруют видимый список available-skills по префиксу `dex-skill-{стек}-*`. Две оси отбора: префикс имени = жёсткая ось стека, `description` = мягкая ось темы. Новый стек = `dex-skill-{стек}-*` + строка реестра; **агенты не правятся**. Детали и эмпирическое обоснование - [docs/AGENT_FRAMEWORK.md](docs/AGENT_FRAMEWORK.md) (раздел «By-stack loading»).

### Frontmatter агентов

Поля и правила (`skills:` запрещён - императивная загрузка через Skill tool; `allowed-tools:` не поддерживается; явный `model` по характеру суждения, не `inherit`; выбор opus/sonnet/haiku) - нормативный дом [AGENT_FRAMEWORK.md](docs/AGENT_FRAMEWORK.md#выбор-модели-model) (чеклист frontmatter + раздел «Выбор модели»).

### Frontmatter skills
```yaml
---
name: skill-name
description: Ключевые слова для автоматической активации
---
```
> **Важно:** `keywords` не поддерживается в skills - ключевые слова включать в `description`. Валидные поля: name, description, disable-model-invocation, user-invocable, argument-hint, allowed-tools, model, effort, context, agent, hooks, paths, shell.

## Механизмы композиции в Claude Code

Проверено по официальной документации (code.claude.com/docs/en/skills + tools-reference.md):

| Путь | Работает | Механизм |
| ---- | -------- | -------- |
| command -> skill | ✓ | `Skill` в `allowed-tools` команды, вызов в теле |
| agent -> skill | ✓ | `Skill` tool в frontmatter агента, императивный вызов в фазе (указывается имя `{plugin}:{skill}`) |
| agent -> agent | ✓ | `Agent` tool - спавн субагента. Глубина рекурсии не задокументирована |
| skill -> skill | ✗ | API нет. Ссылка `см. dex-skill-X` в теле SKILL.md - только сноска для человека |
| skill -> agent | не задокументировано | считать, что нет |
| command -> agent | ✓ на практике | команда-делегат: тело команды поручает работу агенту. `Agent` в `allowed-tools` команде не давать - оркестрация живёт в агенте (конвенция репо, ревью PR#51) |

**Практические правила:**

- **Skill не загружает другой skill.** Если в теле SKILL.md встречается `dex-skill-X` - только как справочная сноска (`> см. dex-skill-X`). Формулировка «загружает / активирует / подключает» другой skill - ложь, переписать.
- **Дублирование ловушек между skills - зло.** При пересечении тем оставить содержимое в одном skill, из другого - сноска.
- **Композиция skills - через агента.** Один агент в своей фазе вызывает несколько Skill tool с точными именами `{plugin}:{skill}`.
- **Связь между специалистами в bundle** - это установка вместе, не runtime-вызов.

## Чек-лист при изменении плагинов

Уроки, накопленные в работе над маркетплейсом. Проходить по списку **до коммита** любых правок в `plugins/` или `docs/`.

### Принципы содержания

- **Skill = обобщённое правило, не конкретика одного проекта.** Читая критику конкретного MR, извлекать универсальный принцип. Имена Entity / классов / доменных терминов из реального проекта в ловушке - сигнал, что правило не обобщено. Принцип проверки: «убери имена проекта - правило всё ещё читается и применяется?». См. анти-паттерн №10 в `SKILL_FRAMEWORK.md`.
- **Skill - ловушки и anti-patterns, не документация API.** Формат «Плохо / Правильно / Почему», 3-5 строк на ловушку. Claude знает синтаксис - skill нужен только для неочевидного поведения и граблей.
- **Агент описывает workflow через фазы, не процедуры.** Goal / Output / Exit criteria / Gate. Ни «сделай grep», ни примеров кода. Mandatory-фазы обязательно с обоснованием «зачем».
- **Создаёшь / правишь агента -> сверься с его рецептом в `AGENT_FRAMEWORK.md`** (Analyst / Diagnostician / Creator / Designer / Operator / Planner / Reviewer): состав фаз, mandatory-список и обоснование берутся оттуда. Состав фаз рецепта в CLAUDE.md не дублировать - нормативный дом один; полные контракты фаз живут в референс-агенте рецепта.
- **Формулировки фаз / гейтов / правил не должны допускать тихий пропуск этапа или проверки.** Описывая обязательный шаг в агенте, команде или skill - закрыть все лазейки, через которые исполнитель может «легально» его не выполнить. Сигналы лазейки: «обычно / часто опирается на ...» (даёт решить, что в этом случае не опирается), «если не нашёл / недоступно» без указания, что делать дальше (даёт закрыть проверку самим фактом отсутствия), «по усмотрению / при необходимости» (делает обязательное опциональным), ветвление или шаг без терминала на ветке - «если X -> Y» без «иначе -> что», шаг с проверкой без названного исхода провала (откат / возврат наверх / статус) - оставляет поведение на ветке неопределённым. Правило: любой уход от проверки фиксируется **явным статусом с обоснованием** (`n/a` + почему, `unverifiable`, `contradicted`), а не молчанием; недоступность инструмента переключает на запасной, а не отменяет шаг; «очевидно» не освобождает от сверки. Принцип проверки формулировки: «может ли добросовестный исполнитель прочитать это так, что проверку можно не делать и в отчёте не останется следа?» - если да, переписать жёстче.
- **Формулировка правила / ловушки / фазы должна быть однозначной.** Каждая единица читается одним способом и не оставляет лазейки для игнорирования. Двусмысленность - это дефект ревью, не стилистика: если формулировку можно прочитать двумя способами, или «Правильно» допускает толкование, при котором правило не применяется, или соседние единицы в одном skill дают взаимоисключающие указания (один приём осуждён в одной ловушке и рекомендован в соседней) - переписать до однозначности или развести область применения явным условием. Критерий проверки: «может ли добросовестный исполнитель прочитать это так, что правило к его случаю не относится или противоречит соседнему?» - если да, формулировка не готова. Это шире, чем запрет тихого пропуска проверки (см. выше): касается любого нормативного текста, а не только обязательных шагов.
- **Формулировки в артефактах пишутся для LLM, не для человека - сухо и по делу.** Skill, агент, команда читаются моделью; нести механизм, условие, исход - не пересказ. Не путать с тихим пропуском проверки (см. выше): сжимаем подачу, а не нормативную силу - лазеек, статусов и обоснований не сокращаем. Лаконичность - обязательная ось ревью (см. ниже). Операционализация принципа (что резать, как писать, информационная архитектура, поле срабатывания `description`, проверка результата) - skill `optimize-for-llm`, нормативный дом операционных правил. Дисциплину формы (атомарность, терминал ветки, узкий escape, реестр статусов) держать как критерий письма и ревью, но НЕ выносить метками-операторами (`GATE:`, `ESCAPE:`) в текст артефакта: прогон `optimize-for-llm` на собственной размеченной версии показал - видимые метки режутся как накладные токены уровня 3, нормативную силу несёт проза правила + реестр статусов.
- **Пунктуация в артефактах - ASCII, не Unicode.** Дефолт: `-` вместо `—`/`–`, `->` вместо `→`, `<-`/`<->` вместо `←`/`↔`, `<=`/`>=`/`!=` вместо `≤`/`≥`/`≠`, `...` вместо `…`. Причина: для артефакта-входа модели типографика на качество не влияет, но ASCII токенизируется стабильнее (1 байт = 1 токен; `—` - UTF-8 3 байта, у части токенизаторов 2-3 токена) и даёт единый стиль. Решает консистентность, не экономия. НЕ трогать: кавычки «», box-drawing диаграмм (`│├└─`), эмодзи-статусы, `№`, надстрочные индексы, `×`, Unicode-примеры внутри `inline code` (там символ - предмет показа, не текст). Миграция **пофайловая**: не сквозняком по репо за раз, но тронул файл по любой причине - приводишь его к ASCII целиком (не только свои строки). Единица миграции = файл. Нетронутые файлы на Unicode до своей правки - не дефект.

### Ревью артефактов (skill / агент / команда)

Ревью = прогон по 4 осям ниже, **каждая обязательна**. «Всё валидно по фреймворку» ревью не закрывает - формальная валидность не отвечает на «стоит ли это держать в каталоге».

1. **Соответствие фреймворку** - формат, обобщённость, версии, композиция по `SKILL_FRAMEWORK.md` / `AGENT_FRAMEWORK.md` / `COMMAND_FRAMEWORK.md`.
2. **Fact-check** - каждый техфакт в «Правильно» / «Почему» сверить с текущей версией API (анти-паттерн №11 `SKILL_FRAMEWORK.md`): context7 -> WebSearch fallback. Сверять КАЖДЫЙ факт, не только целевой; симметрично дописанная half (write-факт к read-факту) - красный флаг. Уход от сверки - явный статус (`unverifiable` / `contradicted`), не молчание.
3. **Полезность** - вывод: держать / убрать / переформулировать. Сигналы слабой единицы: тривиальна (Claude уже знает), косметика без последствий, тематически выбивается, дублирует соседа или конфликтует с другим skill. Качество выше количества - слабую убрать, не тащить ради числа.
4. **Лаконичность** - вода / повтор мысли / пересказ вместо механизма (ось «пишутся для LLM»). Сигналы: «Почему» растёкся в абзац; одно следствие сказано дважды; приписка «это проверяется при ...» вместо механизма; словесное описание дублирует код-блок ловушки.

**Пересечение ловушек != дубль.** Разный ракурс (вход vs выход, разные контексты одного API) - разные грабли, дедуп не нужен. Реальный дубль (одна грабля в двух skills) - чинить по `SKILL_FRAMEWORK.md` «Дублирование между skills» (вынос в базовый skill предпочтительнее сноски: `skill->skill` загрузки нет).

**Особо строго - к авто-ловушкам `/mr-analyze` -> `/mr-apply`:** формально валидны, но часто слабы по сути или с перевёрнутой причинностью.

**Критерий завершённости:** по каждой оси × каждой единице вынесен вывод. Непройденная ось или единица без вывода - ревью не завершено.

### Проверка обобщённости ловушки перед коммитом

- Нет ли в тексте имён конкретного проекта, где была впервые замечена проблема? (`UpCore`, `MergeRequest`, `EyeLineData`, `ArchValue` и т.п.)
- Нет ли привязки к IDE-specific-конвенциям (`.junie/`, `.cursorrules`, и т.д.)? Вместо перечисления - обобщённое «любые документы проекта с конвенциями» + примеры.
- Если скилл .NET-specific - имена API .NET допустимы, имена из конкретного стартапа - нет.

### Самоконтроль при правках фреймворков и документации (docs/, CLAUDE.md)

Правила выведены из реальных рассинхронов (Reviewer-фазы разошлись между библиотекой и рецептом; лимит «250 символов» не подтверждён и нарушался собственными агентами).

- **Один факт - один нормативный дом.** Перед добавлением правила - `rg` по docs/ и CLAUDE.md: уже описано в другом месте -> ссылка, не копия. CLAUDE.md держит строку-суть + ссылку, нормативный текст живёт в одном файле.
- **Фреймворк не впитывает содержимое артефактов.** Детали одной роли (таблицы меток, перечни маркеров, готовые формулировки) живут в референс-артефакте (агенте/skill), фреймворк - контракт + ссылка. Сигнал нарушения: запись в «библиотеке типовых фаз» длиннее 5-6 строк атрибутов.
- **Утверждение о поведении платформы Claude Code** (лимиты, обрезка, поддержка полей) - только со сверкой по официальной документации или живым экспериментом; источник фиксируется рядом. Недокументированное помечается «не задокументировано», цифры не выдумываются. Это анти-паттерн №11 SKILL_FRAMEWORK, применённый к docs.
- **Заявление «это проверяет валидатор» сверяется с кодом валидатора.** Лимит/правило в self-check без реальной проверки в `tools/validate-*.js` помечается как ручное (👁), не выдаётся за автоматическое.
- **Числа состояния каталога** (количество агентов/плагинов/строк) в нормативный текст не хардкодить - устаревают молча. Либо без числа («десятки»), либо там, где число и так бампается (marketplace.json).
- **Roadmap / «Что дальше» / «Статус миграции»** - не в нормативных доках; место - PR description и git history. Исключение - с датой и условием удаления.
- **После правки правила - кросс-сверка дублей:** `rg` по ключевым терминам изменённого правила по docs/ + CLAUDE.md; каждая найденная копия либо синхронизирована, либо сведена к ссылке.

### Версионирование при изменениях (semver)

| Тип изменения | Bump | Пример |
| ------------- | ---- | ------ |
| Новый плагин | `1.0.0` | Создан новый skill / специалист |
| Новый плагин -> версия каталога | minor каталога | `marketplace.json` `version` `5.6.0` -> `5.7.0` при добавлении плагина |
| Изменён состав bundle (`includes[]`) | minor bundle | Добавлен/убран компонент - bump версии самого bundle |
| Rename плагина | major (`2.0.0`) | Переименование имени в frontmatter / директории |
| Breaking change в workflow / API агента | major | Новый рецепт, изменение output format |
| **Добавление новых ловушек в skill** | minor (`+0.1.0`) | Обогащение существующего skill обратно-совместимо |
| Новая категория ловушек | minor | Добавлен раздел, keywords расширены |
| Dedup с другим skill | minor | Перенос общих ловушек в базовый skill |
| Fix description / keywords / опечатка | patch (`+0.0.1`) | Уточнение без содержательных правок |

**Версии обновляются в двух местах одновременно:** `.claude-plugin/plugin.json` **и** `.claude-plugin/marketplace.json`. Несогласованность - частая ошибка.

**Версия самого каталога** (`marketplace.json` верхнеуровневое `version`) бампается при добавлении/удалении плагина - это отдельный счётчик, не выводится из версий плагинов и легко забывается. Новый плагин = minor каталога.

### Обязательные синхронизации

- **Создал новый skill** -> добавить запись в `.claude-plugin/marketplace.json` (иначе плагин не виден каталогу)
- **Бандл должен быть замкнут по скиллам своих агентов.** Каждый не-by-stack скилл, который грузит агент из `includes[]` бандла, обязан быть в этом `includes[]` - установка плоская, каскада «специалист -> его скиллы» нет, недостающий скилл не приедет и агент молча деградирует. Это проверяет `npm run validate:bundles` (правило `bundle-not-closed`). Триггеры: добавил агента в bundle, добавил агенту новую Skill-ссылку, создал skill для агента из bundle. Исключение - by-stack профильные скиллы (`dex-skill-{стек}-*`): грузятся условно по стеку проекта, в каждый бандл не тащатся (см. `dex-skill-stack-registry`). Но как только бандл везёт хотя бы один скилл этого стека (стековый бандл, dotnet-developer и т.п.), он обязан быть замкнут и по этому стеку: иначе стек-специфичный агент (`dex-dotnet-coder`) останется без скилла, который всегда грузит.
- **Создал новый skill, на который ссылается агент в фазе Skill-Based Scan** -> проверить, что точное имя `{plugin}:{skill}` совпадает с фактической директорией
- **Поменял description / keywords skill** -> длина description <= 750 символов (проектный потолок, иначе валидатор `npm run validate:skills` упадёт; 1536 - хард-лимит Claude Code). Свыше 500 - warning, PR не блокируется
- **Поменял агента** -> mandatory-фазы требуют явного обоснования «почему mandatory», иначе валидатор упадёт

### Перед коммитом

`npm run validate` (или точечно `validate:agents` / `validate:skills` / `validate:commands` / `validate:bundles`) - 0 ошибок. Проверки строгие, мягкого режима нет; CI блокирует PR с ошибками. Дополнительно проверить вручную:

- Размер skill: 80-120 строк цель, до 250 допустимо
- Длина `description`: <= 750 символов проектный потолок (error), <= 500 рекомендация (warning); 1536 - хард-лимит Claude Code
- Все Skill-ссылки в агентах указывают на существующие плагины
