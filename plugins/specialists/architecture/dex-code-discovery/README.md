# dex-code-discovery

Обзорный (breadth-first) ревьюер существующего кода. Инвентаризирует проблемы проекта по топикам **вширь** и сохраняет результат в `docs/discover/` — мастер-док + по файлу на топик. Универсален: backend / frontend / mobile, выбор подмножества топиков по детектированному стеку.

«Обзорный» = инвентаризация проблем вширь, НЕ глубокая верификация-под-рефактор и НЕ ADR-планы. Discover отвечает «что в проекте не так и куда смотреть в первую очередь», а не «как именно рефакторить».

## Команда

`/discover [path]` — обзорное ревью проекта (по умолчанию текущая директория). Идёт вширь по всем релевантным стеку топикам, без фильтров: углубление в конкретную область — отдельным инструментом по результату.

## Архитектура

Команда `/discover` тонкая — делегирует агенту `discover-orchestrator`. Оркестратор детектит стек/масштаб и спавнит read-only ревьюеров `discover-reviewer` по топикам (agent->agent). Разделение owner'ов:

- **`/discover`** (команда) — точка входа, делегирует оркестратору. Без `Agent`/`Write` в allowed-tools.
- **`discover-orchestrator`** (агент, Write + Agent) — детект стека/масштаба, спавн ревьюеров, связывание находок между топиками (Cross-Linking), калибровка severity под масштаб, запись `docs/discover/`.
- **`discover-reviewer`** (агент, read-only) — читает код одного топика, возвращает находки с маркерами доказанности ✓/?. На диск не пишет.

Кросс-топиковые шаги живут в оркестраторе, потому что один ревьюер видит только свой топик. Оркестрация в агенте (а не в команде) использует документированный путь agent->agent.

## Каталог топиков

1. Архитектура и слои · 2. Надёжность и устойчивость ⭐ · 3. Безопасность ⭐ · 4. Данные и доступ · 5. Интеграции и контракты · 6. Конфигурация и зависимости · 7. Наблюдаемость · 8. Состояние и жизненный цикл · 9. Производительность · 10. Качество кода и тестируемость · 11. Ключевая бизнес-логика · 12. UX-сквозное (опц.)

Легенда: ⭐ severity весит всегда (не понижается калибровкой) · остальные калибруются по масштабу (на мелких/незрелых проектах чисто-архитектурные находки понижаются).

## Skills

Агент работает в два прохода (Two-Pass): сначала **Direct Analysis** — анализ знаниями Claude без единого skill (первичный взгляд не сужен чек-листом), затем **Skill-Based Scan** — углубление skill'ами поверх первого прохода с дедупликацией. Между проходами hard gate: skill не загружается, пока первичный анализ не дал готовый список. Так skill расширяет находки, а не заменяет свежий взгляд.

Чек-листы топиков — это карта «топик → какие skills грузить», а не зашитые в агента списки. Агент `discover-reviewer` во втором проходе императивно загружает существующие skills маркетплейса по детектированному стеку (`dex-skill-owasp-security`, `dex-skill-dotnet-ef-core`, `dex-skill-react`, `dex-skill-observability` и т.д.).

**Graceful degradation:** если под топик/стек нет skill (например Mobile — Flutter/Android) или skill не установлен — агент работает знаниями Claude и помечает в выводе `skills: none (Claude knowledge)`. Это не ошибка.

Полнее всего discover работает с установленными skills. Рекомендуется ставить вместе с `dex-bundle-architect` (architecture skills) или нужными skill-плагинами по стеку проекта.

## Формат вывода

- `docs/discover/README.md` — мастер-док: карта стека и масштаба, таблица выбранных топиков, сводная карта приоритетов P0–P3 (кросс-топиковая), executive summary, рекомендуемый порядок разбора.
- `docs/discover/NN-<топик>.md` — на каждый топик: «что смотрим», таблица находок (`severity · находка · якорь файл:строка · маркер ✓/? · рекомендация`).

Severity — **P0–P3** (приоритет разбора, не гейт на мёрдж), калиброванный под масштаб. Маркеры доказанности обязательны: `✓` = код прочитан, `?` = по grep без полного аудита.

## Отличие от `/review-arch`

| | `/discover` | `/review-arch` |
|---|---|---|
| Глубина | обзорная инвентаризация вширь | глубокая верификация чтением + рефактор |
| Результат | список проблем по топикам в `docs/discover/` | per-axis рефактор-планы + ADR |
| Стек | универсальный (BE/FE/Mobile), выбор по детекту | заточен под backend |
| Процесс | один проход | по одному пункту через апрув |
| Связь | стадия разведки перед глубоким ревью | потребляет результат discover |

## Связанные плагины

- `dex-architect` / `dex-architect-dotnet` — глубокое архитектурное ревью и проектирование нового (`/review-arch`, `/design`)
- `dex-codebase-analyzer` — utility для быстрого обзора стека репо до запуска discover
- `dex-mr-reviewer` — глубокий per-MR ревьюер (грузит профильные skills по стеку; discover — обзор всего проекта, не diff'а)
