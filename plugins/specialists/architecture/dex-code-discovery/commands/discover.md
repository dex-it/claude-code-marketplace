# /discover

Обзорное (breadth-first) ревью существующего кода: инвентаризация проблем по топикам, результат сохраняется в `docs/discover/`. Не глубокая верификация-под-рефактор и не ADR-планы — это `/review-arch`.

## Goal

Дать карту проблем существующего проекта вширь: по каждому релевантному топику — список находок с приоритетом разбора (P0–P3), калиброванным под масштаб проекта, и сводный мастер-док с порядком разбора.

## Input

Аргумент — путь к проекту (по умолчанию текущая директория). Без фильтров: discover всегда идёт вширь по всем релевантным стеку топикам. Углубление в конкретную область — отдельным инструментом по результату.

## Architecture

Команда — оркестратор с разделением ответственности по двум слоям.

**Команда** (Write): детектит стек, масштаб и сигналы зрелости (auth, multi-tenancy, тесты, CI, прод-конфиги) → выбирает топики; строит обзорную карту проекта; спавнит ревьюеров (MVP — последовательно, по одному топику); связывает их находки в группы «root cause → symptoms» поверх всех топиков; калибрует severity под масштаб; пишет результат в `docs/discover/`. Кросс-топиковые шаги и запись — только у команды, потому что только она видит находки всех топиков и имеет Write.

**Агент `discover-reviewer`** (read-only): по назначенному топику читает код и возвращает находки с маркерами ✓/?. На диск не пишет.

Калибровка под масштаб: надёжность и безопасность (⭐) весят всегда; чисто-архитектурные находки (⚖️) на мелких/незрелых проектах понижаются.

## Topic Catalog

Выбор топиков по детектированному стеку. Backend: 1-7, 9-11 (+ конкурентность под 8). Frontend SPA: 1-3, 5-10, 12. Mobile: 1-10, 12.

1. Архитектура и слои · 2. Надёжность и устойчивость ⭐ · 3. Безопасность ⭐ · 4. Данные и доступ · 5. Интеграции и контракты · 6. Конфигурация и зависимости (секреты ⭐) · 7. Наблюдаемость · 8. Состояние и жизненный цикл · 9. Производительность · 10. Качество кода и тестируемость · 11. Ключевая бизнес-логика · 12. UX-сквозное (опц., UI)

Карта «топик → какие skills грузить» живёт в агенте `discover-reviewer`. Нет skill под топик/стек (например Mobile) → агент работает знаниями Claude.

## Output

`docs/discover/README.md` — мастер-док: карта стека и масштаба, таблица выбранных топиков, сводная карта приоритетов P0–P3 (кросс-топиковая), executive summary, рекомендуемый порядок разбора.

`docs/discover/NN-<топик>.md` — на каждый топик: «что смотрим», таблица находок (`severity · находка · якорь файл:строка · маркер ✓/? · рекомендация`), краткие рекомендации.

Severity — **P0–P3** (приоритет разбора, не гейт на мёрдж; P0 — разбирать первым), калиброванный под масштаб. Шкала действий с цветными метками (🟢🟡🟠🔴) намеренно НЕ используется — discover смотрит существующий код, а не PR, блокировать мёрдж нечего. Маркеры доказанности обязательны: `✓` = код прочитан, `?` = по grep без полного аудита.

## Constraints

- Калибровка меняет уровень находки, не вычёркивает её. Надёжность и безопасность (⭐) не понижаются по масштабу.
- Дефолт подсветки: нет явного маркера accepted tech debt (TODO+Jira, ADR, пометка в проекте) → находка подсвечивается, не молча принимается.
- Не предлагать рефакторинг-планы и ADR — это обзор, не глубокое ревью.
- Если стек не определён — спросить пользователя. Если и после этого стек не ясен (нет манифестов, README, понятной структуры) — пройти универсальные топики 1 (Архитектура и слои) и 10 (Качество кода и тестируемость), пометив в мастер-доке, что стек не детектирован.

Делегировать чтение кода по топику агенту `discover-reviewer`.
