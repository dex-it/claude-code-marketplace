# Контракт стыка между агентами-узлами

Узел конвейера = агент с сигнатурой (вход + выход). Этот skill - **общий словарь полей
и правило стыка**, чтобы выход узла N принимался входом узла N+1 без потери данных.
Сигнатуру **каждого** узла держит сам агент: сжато в `description` (оркестратор видит до
спавна), полно в фазах тела. Здесь - что поля **значат** и как реагировать на неполный
вход. Маршрут (кто за кем) решает оркестратор, не узел.

> Дом маршрутизации - оркестратор (внешний `autonomous-task`). Здесь только форма стыка;
> «следующий узел» узлом не задаётся.

## Правило стыка (для всех рёбер)

Стык **автоматический**: выход узла N читает не человек, а узел N+1 (или оркестратор),
сам, без надзора. Ошибка на стыке не замечается на месте - течёт по цепочке и всплывает
дорого. Поэтому выход = протокол с проверяемой целостностью, не свободный отчёт.

### A. Исход узла - машинно-различим первым полем

1. **Первое поле выхода = `status`: `complete` | `blocked` | `partial`.** Оркестратор
   маршрутизирует по нему, не по содержанию.
   - `complete` - все обязательные поля ребра заполнены, узел закончил.
   - `blocked` - halt: не хватило обязательного бизнес-поля (см. реакцию на нехватку) ИЛИ
     отказал инструмент самой работы. Несёт: что именно не хватило/отказало + кому это
     пополнить. Результата нет.
   - `partial` - часть готова, часть нет. Несёт: что готово (поля) + что не закрыто (+
     статус из словаря на каждое незакрытое). Оркестратор решает - принять часть или
     дослать.
   `blocked`/`partial` НЕ маскируются под `complete`: молчаливый возврат отказа = оркестратор
   едет дальше на пустом месте. Это запрещено в обоих режимах.

### B. Состав и качество выхода

2. **Выход = именованные обязательные поля** ребра, каждое помечено своим именем (словарь
   полей ниже). Поля нет -> явный статус из словаря статусов, не молчание. Приёмник тащит
   поле по имени; «найди нужное в абзаце» - источник ошибки в автостыке.
3. **Лаконичен и однозначен.** Раскладка свободна (список/таблица - как удобнее приёмнику),
   строгость - нет: каждое поле читается одним способом, без воды. Выход = вход контекста
   следующего узла-LLM; вода и двусмысленность тиражируются по цепочке. Несёшь поле/статус/
   решение - не пересказ.
4. **Неуверенность - часть поля, не опускается.** Поле, добытое с неполной уверенностью
   (гипотеза root cause, не подтверждённый фактом вывод, спорное допущение), несёт признак
   доказанности (`confidence`/`по гипотезе`/`unverifiable`). Без него приёмник примет догадку
   за факт - в автоцепочке это не откатывается.
5. **Все решения узла - в выход.** Любой выбор узла (восполнение инженерной нехватки,
   трактовка неоднозначности, округление, паттерн, имя, структура, дефолт по `[default-ok]`)
   -> явным пунктом, не молча в артефакте (коде/дизайне/диаграмме/доке). Решение без следа =
   тихое допущение, запрещено и в `autonomous`.
6. **Сквозные поля передаются дальше, даже если узел их не трогал.** Поле, нужное узлам
   ниже по цепочке (`constraints`, `non-goals`, `key decisions`), но не относящееся к работе
   твоего узла, переноси в выход как есть. Потеря на промежуточном узле = поле исчезает из
   цепочки молча, всплывает нарушением на 3 шага позже.

### C. Приём входа

7. **Приёмник проверяет вход** - сверяет пришедшее с обязательными полями своего входа.
   Недостающее обязательное поле НЕ домысливается молча: реакция - по природе нехватки
   (раздел ниже), инженерное восполнение всегда с пометкой в Output (п.5). Поле пришло, но
   его смысл/имя узлу непонятны -> трактовать как нехватку (halt безопаснее догадки), не
   угадывать.

### D. Сквозной инвариант

8. **`mode: interactive | autonomous`** в каждом прямом handoff. Источник проставляет; нет
   поля -> `autonomous` (спавн узлом = канала к юзеру нет; `interactive` включается только
   явной передачей от команды-обёртки главного цикла). Узел режим не угадывает - читает из
   handoff. **Режим меняет только адресата возврата, НЕ право пропустить бизнес-решение.**

**Словарь статусов** (поле/проверка не закрыты - один из, не молчание): `n/a` + причина
(неприменимо к узлу/задаче); `unverifiable` (сверить не удалось - инструмент/источник
недоступен); `contradicted` (источник опроверг); `<phase> без проверки <skill>` (профильный
skill не загрузился, см. graceful degradation).

## Реакция приёмника на нехватку обязательного поля

Критерий реакции - **природа недостающего решения**, не имя поля и не режим:

| Род нехватки | Что это | Реакция (оба режима) |
|---|---|---|
| **Инженерное** | узел вправе выбрать сам: округление, тип возврата, паттерн, имя, структура | принять выбор, продолжай (`interactive` - можно вместо этого вернуть). Выбор -> явным пунктом в Output (п.5), молча нельзя ни в одном режиме |
| **Бизнес** | узел знать не может: `success criteria`/«готово», бизнес-правило, приёмка, намерение | **halt: верни `status: blocked` наверх, оба режима.** Нехватка = неполная постановка; пополнить, не угадать |
| **`[default-ok]`** | non-goals, ADR, остатки | дефолт + пометка, оба режима |

`success criteria`, `requirements` - бизнес-ось, halt и в `autonomous` (нечем мерить
«готово»). `diff-scope`, публичные контракты - блокеры, но чаще инженерно восполнимы.
Сомнение «инженерное или бизнес» -> считать бизнес (halt безопаснее тихого допущения).

**halt = возврат оркестратору, НЕ вопрос юзеру.** Узел-субагент канала к юзеру не имеет:
`status: blocked` идёт источнику вызова, оба режима. Оркестратор сам решает - эскалировать
юзеру, добрать из другого узла, дополнить сам. `autonomous` != «спросить некого, поэтому
угадай»: для бизнес-нехватки = «верни, пусть пополнят постановку».

## Outward-facing действие = санкция оркестратора, не дефолт режима

Действие, видимое вне прогона (запись в чужой MR/PR, деплой, отправка сообщения, мутация
внешней системы), узел по своему режиму НЕ инициирует - право даёт **явное поле входа**
от оркестратора (`publish`/`apply`/`deploy` = `true`/`false`): он знает, авторизован ли
прогон на запись. Поля нет / `false` -> не выполнять: собери результат, верни в Output,
а в нём явным полем «outward-действие не выполнено (нет санкции)» - чтобы оркестратор не
счёл, что запись прошла. Здесь `autonomous` тоже значит «верни оркестратору», не «делай
молча». Внутренние действия (правка рабочего дерева, локальные коммиты) этим не
ограничены - это работа узла.

## Словарь полей

Поле сверяется **по смыслу, не по имени**: источник может назвать иначе - засчитывать
синоним, если несёт тот же смысл.

| Поле | Смысл | Синонимы источника |
|---|---|---|
| `status` | исход узла, первым полем выхода | `complete`/`blocked`/`partial` (см. правило A) |
| `mode` | канал узла | `interactive`/`autonomous`; нет -> `autonomous` |
| `requirements R/I` | что реализовать: явные (R) + неявные из проекта (I) | scope инкремента + Deep Dive (architect); ТЗ |
| `success criteria` | проверяемый, измеримый критерий «готово» | DoD; Success metric; acceptance criteria; для bug-fixer - исходные баги red->green |
| `non-goals` | что НЕ делать | scope «что не входит» |
| `key decisions` | принятые решения + релевантные `Accepted` ADR | Deep Dive (storage/API/cache/sharding) |
| `constraints/risks` | ограничения, риски | Risks инкремента |
| `diff-scope` | **пути** изменённых файлов + ветка/база (не тела; артефакт читается с диска, см. «Транспорт артефакта») | список путей; changeset |
| `intent` | источник намерения для сверки кода с требованием (план/задача) | success criteria; ссылка на ТЗ |
| `run-status` | результат build/test/lint | зелёный/красный + что |
| `anchor` | привязка находки | `file:line` |
| `closure` | когда находка считается закрытой | критерий исправления |
| `severity` | критичность | CRITICAL/HIGH/MEDIUM/LOW |

## Правила передачи (общие, по типам стыка)

Какие **поля** требует конкретный узел - в `Input/Output` самого агента (дом сигнатуры
- агент, не этот skill). Здесь - **правила формы передачи**, общие для всех узлов
данного типа стыка.

### Оракул проверки = критерии приёмки, не реализация
Узел, проверяющий чужую работу (тест, ревью), держится за `success criteria`/намерение
(«что должно быть»), НЕ за реализацию проверяемого. Вывод оракула из самой реализации
зеркалит её и цементирует её баги.

### Транспорт артефакта = репозиторий/диск, не текст handoff
Тела файлов, diff, патчи через handoff **не передаются** - handoff несёт **указатели**
(пути, ветка/база), приёмник читает артефакт сам из двух источников:
1. **репозиторий (дефолт):** состояние из VCS (для кода - committed + staged + worktree).
2. **пути из handoff (fallback):** репозиторий пуст ИЛИ handoff явно дал `diff-scope` ->
   читать перечисленные файлы с диска.

Тело в тексте избыточно и провоцирует работу по присланной копии вместо реальной на
диске. Оба источника пусты -> `status: blocked` («обрабатывать нечего») оркестратору, не
выдумывать содержимое.

### Форма возврата на доработку
Находка возврата = `anchor` + `closure` + `severity`; без `anchor` или `closure` не
передаётся (закрывает «голую находку»). Указывай: что невыполнимо/противоречиво (пункт),
почему (контракт/стек/недостающее решение), чего не хватает - вернуть, не домысливать.

## Mid-flight: вопрос возник в ходе работы

Тот же критерий «природа решения»: бизнес-вопрос (намерение, приёмка, бизнес-правило)
-> halt (`status: blocked`) оркестратору, оба режима; инженерный -> `autonomous` допущение
+ громкая пометка, `interactive` можно вернуть. Не-блокер -> дефолт + пометка. Возврат
всегда оркестратору, НЕ юзеру.

## Graceful degradation при недоступности skill

Загрузка профильного skill через Skill tool не удалась (не установлен / Skill tool
недоступен / неверное имя) -> узел **не останавливается**: skill-проверка - усиление, не
блокер постановки. Действие:

1. Выполни фазу по своим знаниям без этой проверки.
2. Пометь в Output: «фаза N без проверки `<skill>` - <что не свёрено>» (словарь статусов).
3. В финале собери пропущенные проверки одним блоком - вызывающий видит зону непокрытого.

**Граница skill != инструмент работы:** skill даёт *проверку/грабли*, его потеря
деградирует качество -> продолжай с пометкой. Инструмент даёт *возможность выполнить шаг*,
его потеря делает шаг невыполнимым -> `status: blocked` оркестратору.
