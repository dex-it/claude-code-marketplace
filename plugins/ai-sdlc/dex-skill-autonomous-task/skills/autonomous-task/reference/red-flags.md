# Red flags - стоп, ты срезаешь

> Чеклист самообмана движка. Ядро (`SKILL.md`) обязано прочитать этот файл (mandatory-Read на
> шаге 1 цикла, рядом с дельтой трека: ловушки сквозные по циклу, нужны в контексте до входа в
> них; на шаге 7 - сверка). Левый столбец - мысль-рационализация, правый - что на самом деле
> требует костяк/трек. Якоря (`P-*`) определены в `SKILL.md` и `tracks/<трек>.md`.

| Мысль | Реальность |
|---|---|
| «Спрошу пользователя на развилке» | Спрашивать некого. Выбери трактовку, зафиксируй, иди. |
| «Не уверен - подожду» | Ожидание = провал. Реши обоснованно и продолжай. |
| «Это же всегда про код» | Определи трек. Ревью MR / тесты / диагностика - не разработка. |
| «На ревью MR заодно поправлю код» | Трек ревью - read-only. Выход = треды, не коммиты. |
| «Ревью - посмотрел, как написано» | Покрытие тестами - отдельный фокус ревью. Нет тестов на изменённое поведение - находка с severity. |
| «Тесты пишу как придётся в каждом треке» | Дом тест-дисциплины - трек «Тест-инжиниринг». Добор покрытия в «Разработке»/оценку в «Ревью» веди его дисциплиной (матрица, gap'ы, тест-агент). |
| «Сдам частично» | Доводи до конца: верификация + артефакт + трекер. |
| «Саморевью пропущу» | Шаг 7 обязателен - разбор против кода/фактов/потребителей. |
| «Одного ревью кода хватит» | Разработка - отдельно: код + архитектура + безопасность. |
| «Минимальный патч, и ок» | Системный компонент - приводи в чувство, не латай. |
| «Проверю/найду/напишу сам» | Нетривиальные поиск/ревью/реализация/тесты - делегируй. |
| «Возьму привычного агента» | Подбери под стек+тип из доступных, не по памяти. |
| «Coder под стек сделает и security-ревью/тесты» | Тип узла нельзя подменять. security-ревью -> security-ревьюер, тесты -> тест-агент, ревью -> ревьюер. Профильный по стеку != годен на любую роль. |
| «Делаю сам - скилл не нужен» | Своя прямая работа тоже через профильный скилл, вызванный явно. Авто-активация не гарантия. [[P-skill-first]] |
| «Допущение и так понятно» | Не понятно. Вынеси строкой в блок сдачи. |
| «Вынесу развилку на ревьюера / в тикет» | Техническое реши сам по коду. На человека - только неочевидная бизнес-логика. |
| «Спрошу про scope / мердж / шаг флоу» | Это инженерия и процесс - решаешь сам. Автору - только бизнес-intent, async. |
| «Бизнес-вопрос - просто спрошу автора» | Предложи решение по best practice + здравому смыслу, двигайся на дефолте. Автору - вопрос-с-предложением «принял X, потому что Y», не голое «как надо?». |
| «Раз бизнес так сказал - делаю» | Бизнес-вводная тоже claim, не догма. Противоречит данным/метрикам/инвариантам - возрази с предложением, не исполняй ошибку молча. |
| «Отложу в тикет, при необходимости усилим» | Сначала прочитай код, который даст ответ. Откладывать определимое = маскированная лень. |
| «Зашью команды/трекер/агента в движок» | Конкретика - из проекта, не в скилл. |
| «Спросить некого -> всё можно» | Необратимое/массовое - подтверждай. Точечное обратимое - делай. |
| «Возобновлюсь по памяти/сводке» | Процедура возобновления (SKILL.md), не память: перечитай трек-файл + red-flags (компакт их унёс), восстанови из плана, сверь ground truth (трекер/MR/ветки/прогон), сверь синхрон трекера. План - точка пересборки. |
| «После компакта статусы и так двинуты» | Компакт легко проспать. Сверь, что под-задачи и зонтик в трекере отражают реальную фазу из тасклиста; расхождение устрани до продолжения. [[P-tracker-sync]] |
| «Статус/наличие вычитаю из тикета/memory» | Снимок, не текущий факт. Статус - в трекере, наличие - в коде целевой ветки. |
| «ADR - намерение, раскатка не дошла - отложу» | ADR-решение - стандарт, следуй сейчас. Из ADR читай лишь статус раскатки - он в коде/трекере. |
| «В коде так - скопирую у соседей» | Код может нарушать ADR. Норма - ADR; не тиражируй дефект. |
| «Зависимость отложена в тикет N» | Проверь статус N и есть ли результат в коде целевой ветки. |
| «Прогоню ещё раз так же» | 2-3 красных - меняй подход, не долби провалившееся. |
| «Баг нашёл - просто починю» | Сначала тест red->green через реальный путь. |
| «Unit зелёный - фича готова» | Unit != интеграция. Прогони весь путь живьём до внешней границы (API/БД/очередь), каждую ветку (primary/fallback). Мок границы врёт о контракте. [[P-integration]] |
| «Happy-path зелёный - фича покрыта» | Happy != матрица. Тест-DoD фичи = матрица покрытия ([[P-test-matrix]]) живьём до границы + регрессия потребителей. Меньше - не готова. [[P-feature-dod]] |
| «Тестирую модуль, который правил» | Тестируй ВСЕ узлы цепочки. Сперва выясни точку входа - кто реально вызывает фичу в проде, не угадывай по ходу. |
| «План в голове удержу» | Крупное - в файл `docs/autonomous/plan-*.md`, это летопись шагов, отмечай что сделал. |
| «Кто работал - вспомню в конце» | Логируй каждый делегированный вызов по ходу; в сдачу - блок «Задействованы» с явной моделью каждого. |
| «Соберу/прогоню параллельно» | Параллельные сборки одного решения гонятся за `obj/bin`. Сборку/тесты одной solution - строго по одному. |
| «Один пункт на под-задачу» | Тасклист гранулярный - пункт на каждый осязаемый шаг. |
| «Заведу один зонтик, чеклист в плане» | Каждый логический этап - отдельным пунктом, по одному вызову `TaskCreate`. План-файл не заменяет тасклист - нужны оба. |
| «Разведка/Explore так сказала - приму» | Находки разведки - гипотезы. Факт под план перепроверь первоисточником сам. |
