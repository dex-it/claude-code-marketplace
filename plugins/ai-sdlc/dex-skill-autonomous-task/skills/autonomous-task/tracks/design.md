# Трек «Проектирование» - дельта сверх таблицы

> Дополнение к строке «Проектирование» таблицы треков в `SKILL.md`. Ядро читает этот файл,
> определив трек (mandatory-Read из шага 1) - до плана. Якоря `I-decide`/`P-recon`/`P-fact` - в `SKILL.md`.

*Цель = инженерный дизайн будущей реализации (КАК делать), код ещё не пишем; верификация =
дизайн сверен с кодом + обязательные разделы есть. Апрув автора НЕ нужен.*

**Граница с соседними треками - не спутать:**
- **Спецификация** - про **идею/постановку** (ЧТО делаем, бизнес-цель, критерии приёмки, scope);
  бизнес-вопросы -> автору, гейт передачи = апрув автора. **Проектирование** - про **реализацию**
  (КАК: контракты, слои, границы, паттерны); это инженерия, решаешь **сам** ([[I-decide]]), апрува
  автора нет. Спека пришла с критериями -> Проектирование берёт их как вход и проектирует под них.
- **Разработка** - **пишет код**. Проектирование - **не пишет** (артефакт = дизайн-док/ADR/диаграммы).
  Дизайн готов -> передаётся в «Разработку» как DoR. Мелкий дизайн «по ходу фичи» остаётся внутри
  «Разработки» (под-вид фича, дизайн до плана); отдельный трек - когда дизайн **сам по себе цель**
  (спроектировать сервис/модуль/API/интеграцию до реализации).

**Конвейер:**
1. **Контекст и ограничения.** NFR (нагрузка, latency, доступность, консистентность), существующие
   контракты/слои/ADR затронутых реп. Факты о текущем коде - первоисточником сам ([[P-fact]],
   [[P-recon]]), не по памяти/тикету.
2. **Capacity (если применимо).** Back-of-envelope: объёмы, RPS, рост, хот-пути. Цифры - вход для
   решений (шардирование, кэш, очередь), не украшение.
3. **Дизайн.** Контракты (API/события/схемы), границы модулей/слоёв/транзакций, размещение,
   reference-паттерн под задачу. **Инж. развилки решаешь сам с обоснованием** ([[I-decide]]) -
   trade-off каждой явной строкой, не «решим на ревью».
4. **ADR при смене правила.** В корне выбор «применить существующее правило» vs «ввести новое
   правило/паттерн»: ввести новое -> **ADR в той же ветке** (контекст -> решение -> последствия ->
   альтернативы), применить существующее -> ссылка на действующий ADR. Не ритуал на каждый дизайн -
   по сути.

**Обязательные разделы дизайн-дока** (полнота = часть DoD): контракты; компоненты и их границы;
поток данных; обработка ошибок/граничных случаев; NFR и как закрыты; trade-off принятых решений
с обоснованием; затронутые ADR. Раздел пуст «потому что неясно» -> это инж. вопрос, **закрой сам**
(дочитай код, взвесь), не оставляй дыру.

**Верификация (как закрыть «готово» без кода и без апрува):** (1) дизайн **сверен с существующим
кодом/контрактами** - реализуем, не противоречит ADR (нашёл противоречие -> дизайн правится или
ADR оспаривается явно, не молча); (2) **все обязательные разделы** заполнены; (3) каждая инж.
развилка **решена с обоснованием**, не отложена «на потом/ревью». Бизнес-неочевидность (если
всплыла) - вопрос-с-предложением автору async, дизайн держит предложенный дефолт, **не блокируется**.

**Саморевью:** реализуемость (можно ли это закодить в текущей архитектуре), полнота разделов,
не протащена ли инженерия в «бизнес-вопрос автору», синхрон с кодом, security дизайна (границы
доверия, поверхность атаки заложены на уровне дизайна, не «потом прикрутим»).

**Исполнители:** дизайн/архитектура -> architect под стек (`architect`, `architect-dotnet`); API/
контракты -> `api-designer`; ADR -> `adr-writer`; диаграммы -> `diagram-creator`. Тип узла -
проектировщик, **не coder** (см. карту «Делегирование» в `SKILL.md`).
