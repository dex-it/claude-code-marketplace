# Трек «Согласование/Спецификация» - дельта сверх таблицы

> Дополнение к строке «Согласование/Спецификация» таблицы треков в `SKILL.md`. Ядро читает этот
> файл, определив трек (mandatory-Read из шага 1) - до плана. Якоря `P-recon`/`I-decide` - в `SKILL.md`.

*Цель = согласованная спецификация будущей разработки, кода пока нет; верификация = инж. вопросы
закрыты + бизнес-вопросы приняты автором.* Вход - **присланный план/постановка**, его изучаешь,
а не сочиняешь с нуля. Конвейер:

1. **Бизнес-анализ.** Продуктовая цель плана; неочевидную бизнес-логику - вопросом автору async.
2. **Системный анализ.** Сверка с кодом по всем затронутым репам (факты - первоисточником
   сам, [[P-recon]]). **Классифицируй каждый пункт плана:** *уже в коде* -> в верификацию;
   *реальный остаток* -> в спеку; *невозможно/против ADR* -> переформулировать с обоснованием.
   Здесь **решаешь все инженерные/процессные вопросы**. **A - pre-check «нужен ли ADR»**
   встроен в эту фазу: если в корне лежит выбор «применить существующее правило» vs.
   «ввести новое правило/паттерн» - это сигнал, что нужен ADR в той же ветке до плана.
   Применить существующее -> план + ссылки на существующие ADR. Ввести новое -> ADR + план.
   Не ритуал на каждый тикет - включать по сути.
3. **Спецификация.** Контракт/дизайн, пути правок по репам, затронутые контракты/ADR.
4. **Ожидаемые выходные артефакты** - **обязательный раздел**: всё, что разработка произведёт
   (ADR, тесты по приоритету, конфиги/миграции, доки/RELEASE_NOTES, MR по репам).
   **D - fallback прикрепления плана:** стандарт - файл плана вложением в задачу трекера
   через API прикрепления файлов (конкретный синтаксис - от трекера: Jira `attachments`,
   GitLab `POST /projects/:id/uploads` и т.п.; у GitHub REST API вложений в issue нет - только
   веб-UI, поэтому инлайн плана или внешний хостинг); если API/MCP не видит локальный путь (типично для MCP, крутящегося в `/app/`
   или другом контейнере) - инлайнить план в сообщение передачи. Правило «план
   прикладывается файлом» не отменяется; inline - компромисс при невозможности,
   сохраняющий суть для исполнителя.
5. **Критерии приёмки** - **обязательный раздел**: проверяемый чеклист наблюдаемых фактов.
6. **Вне scope** - **обязательный раздел**: что НЕ входит (защита от scope-creep).
   Найденный по ходу техдолг/баги - сюда + **отдельные задачи после согласования** (не до),
   не тащи молча и не игнорируй.
7. **Финал.** Ждём ответов (не блокируясь) -> ревью спеки -> финализируем -> передаём в работу.
   **B - артефакт передачи публикуется ТОЛЬКО после явного апрува автора.** Перед публикацией
   сообщения с планом и открытыми вопросами (handoff в трекере, описание MR, эквивалент в
   другом трекере) - **дождаться явного «ок» на текст**. Показать текст -> получить «ок» ->
   публиковать. Не публиковать «превью» в трекере «чтобы показать как выглядит» - это шум
   для исполнителя и плодит комментарии.

**Режим по умолчанию - интерактивный в чате (оператор в loop).** Если оператор явно не
объявил «полностью автономный» режим, согласование идёт в чате: показываешь материал,
обсуждаешь, итерируешь, финализируешь прямо в чате. В трекер ничего не публикуешь
«превью». Публикация handoff (комментарий/issue-описание в трекере, описание MR, передача
исполнителю) - только по явной команде оператора («опубликуй / запости / передай»),
после чего работает правило B. В «полностью автономном» режиме (явный opt-in) handoff
публикуется по правилу B без отдельной команды на публикацию - исполнитель узнаёт о
задаче через трекер, не через чат. Это не «авто-публикация всего подряд», а снятие
обязательства ждать команды на сам акт публикации (правило B про «показать -> ок ->
опубликовать» сохраняется).

**Под-вид C - интерактивный для тривиальных bug-фиксов.** Критерии: фикс по существующему
ADR/спецификации/call-site'у; нет развилок бизнес-логики; 1-2 файла; DoD помещается в
голове. Если все сходятся - допустимо согласование «по ходу» в чате без формального брифа;
план в `docs/autonomous/plan-<TICKET>.md` (локально, в MR не обязателен), публикация в
трекере - по правилу B. Хотя бы один пункт не сходится - полный цикл выше.

**Жёсткое разделение вопросов:** инженерные/процессные решаешь **САМ** ([[I-decide]]) и
**фиксируешь в спеке как принятое решение с обоснованием**, не откладываешь «решит ревью /
тикет при необходимости»; **на обсуждение - только неочевидная бизнес-логика**, тредом async и
**с предложенным решением** (см. «Решения» в `SKILL.md` - вопрос-с-предложением, не голый), спека
держит предложенный дефолт, а не пустой вопрос.
Структурируй через DoR/DoD; «Ожидаемые артефакты» и «Критерии приёмки» - часть DoD. **Гейт
передачи в разработку: спека полна И бизнес-вопросы приняты автором**; пока ответов нет - не
блокируйся (инж. часть добивай, бизнес-вопросы держи списком). **DoD спеки = DoR разработки.**
**Лазейки протаскивания (RED-проверено, закрывай):** (1) «подтверди X для единообразия / так
ведь?» по вопросу, ответ на который в коде - вопрос-маскировка, фиксируй решением; (2) граница
авторизации - *механика* (паттерн политики, scope, изоляция тенанта) инженерная, решаю сам;
*какая бизнес-роль вправе* - продуктовое, автору; (3) инвариант безопасности/комплаенса (PII/PCI
не утекает) - фиксируешь решением, **не** выносишь «на подтверждение». Саморевью: каждый ли
инж. вопрос реально закрыт (нет «решим при необходимости»); не протащена ли инженерия в
бизнес-вопросы; есть ли 3 обязательных раздела; синхрон с кодом; нет LLM-маркеров.
