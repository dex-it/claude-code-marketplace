# Трек «Разработка» - дельта сверх таблицы

> Дополнение к строке «Разработка» таблицы треков в `SKILL.md`. Ядро читает этот файл, определив
> трек «Разработка» (mandatory-Read из шага 1 цикла) - **до плана**. Слоты (выполнение/верификация/
> артефакт) - в таблице ядра; план/ревью/сдача - в костяке. Якоря `P-feature-dod`, `P-integration`
> определены здесь; `P-test-matrix` - в `test-engineering.md`; `P-root`/`P-recon`/`P-fact`/`I-decide`
> - в `SKILL.md` (файлы читаются вместе через mandatory-Read).

Под-вид определи **до плана** (критерий «готово» и роль тестов спутать нельзя):

- **фича** - новое поведение, тесты под него (TDD). <a id="P-feature-dod"></a>**Тест-DoD
  фичи = матрица покрытия (+) дельта трека (все пункты, не подмножество):** матрица
  happy/негатив/краевые/все-ветки ([[P-test-matrix]]) **через реальный путь**, прогнанная
  **живьём до внешней границы** ([[P-integration]]), плюс **регрессия потребителей** -
  тесты задетых вызывающих, стык не сломан. Покрыто меньше - фича НЕ готова; отрезанный
  пункт - строкой в блок сдачи с обоснованием, не молчанием.
- **баг-фикс** - тесты воспроизводят дефект;
- **рефакторинг** - поведение НЕ меняется; тесты - сейфти-нет, зелёные без правки ассертов.

Смешанное «почини и заодно причеши» -> раздели на под-виды, не в один diff.
**Саморевью - 3 обязательных прохода** (не один): (1) код под стек, (2) архитектура
(слои/контракты/зависимости/NFR), (3) **security** (auth, доступ, данные, ввод, секреты) -
security **отдельный проход даже если «чисто»**, не растворять в ревью кода; + доменные по
нужде (перф, миграции, API-контракт).

<a id="P-integration"></a>**Пирамида: тестируй ВСЕ узлы цепочки, не один лист.** Декомпозиция
на модули -> unit на КАЖДЫЙ модуль -> сборка (tsc/контракты) -> **интеграционный прогон всего
пути живьём**. Unit изолированного модуля **не закрывает фичу**: баг живёт на *стыке* модулей
и на границе с чужой системой. **Граница интеграции = там, где наш код отдаёт управление
внешней системе** (внешний API, БД, очередь, чужой сервис) - мок этой границы **врёт о
контракте** (мок принял `mediaType:"image"` - реальный Google API отверг 400; unit был
зелёным, прод падал). Поэтому: (а) **до** написания тестов **выясни и зафиксируй точку входа**
- кто и как реально вызывает фичу в проде (не угадывай сигнатуру по ходу); (б) для
LLM/внешних-API/IO обязателен **живой прогон реального пути** до самой границы, не только
unit-форма того, что мы отправляем; (в) если у пути есть **развилки** (primary/fallback
провайдер, retry-ветка, разные адаптеры) - прогнать **каждую** ветку: это разные получатели
с разными контрактами (два провайдера у LLM-приложения - частный случай). Зелёный unit при
непокрытой интеграции = **фича не проверена**, даже если «тесты зелены».

- **баг-фикс:** воспроизведи дефект -> **тест red через реальный путь** (не суррогат) ->
  **root-cause до первопричины** (делегируй; лечи корень - [[P-root]]) -> **минимальный**
  фикс -> green -> регресс-проход. Корень не дался за 2-3 подхода -> блокер; обходной путь -
  только с заведённым тикетом.
- **рефакторинг** *(инвариант: внешнее поведение неизменно)*: **сначала сейфти-нет** (нет
  покрытия - допиши тесты на поведение как есть) -> структурные правки **малыми шагами**,
  после каждого тесты зелёные **без изменения ассертов** (поменял ассерт - это уже не
  рефакторинг, стоп) -> «готово»: тот же contract/API/выход, структура и читаемость лучше,
  мёртвый код удалён. Саморевью: арх-проход главный, security если трогались границы. Баг
  по пути - **отдельная цель**, не правь молча.
- **внешнее ревью на свой MR - доделка внутри этого трека, не новый трек** *(цель та же -
  сдать фичу)*: треды + свой diff с момента ревью -> классифицируй и **проверь чтением кода,
  что ещё валидно** (исправленное поздним коммитом отсей; неоднозначное за ревьюера не
  трактуй) -> план правок + черновики ответов -> доделай **теми же профильными** -> ре-ревью
  дельты -> петля до закрытия.

**«Разберись и почини» - это «Разработка» (баг-фикс), не отдельный аналитический трек.**
Автономный разбор «что и почему» - **фаза контекста/root-cause внутри баг-фикса**: делегируй
аналитическим/диагностическим агентам, забери вывод как гипотезу, **перепроверь первоисточником
сам** ([[P-recon]]), веди фикс по треку. Отдельного аналитического артефакта нет - выход баг-фикса
(MR + тест red->green).

**Сбор тестов переиспользует дисциплину трека «Тест-инжиниринг», не изобретает свою.**
TDD red->green (тесты ведут код) - встроенная механика «Разработки». Но **добор покрытия** (матрица
осей [[P-test-matrix]], отчёт о gap'ах) ведётся дисциплиной и исполнителем «Тест-инжиниринга»:
делегируй тест-агенту под стек, веди матрицу, назови оставшиеся gap'ы без тихого усечения.
Тестовую механику в треке заново не выдумывай (см. `test-engineering.md`).
