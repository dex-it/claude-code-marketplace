# /mr-analyze

Читает файл, созданный `/mr-collect`, и предлагает предметные правки маркетплейса: новые ловушки в существующие skills и / или точечные правки существующих агентов. Имена конкретного проекта обобщаются по принципу SKILL_FRAMEWORK («убери имена проекта — правило должно работать»).

**Goal:** Файл с конкретными предложениями или явное сообщение «ничего не найдено». Stdout -- только абсолютный путь либо сообщение о пустом результате.

**Input:** абсолютный путь к файлу `/tmp/mr-collect-*.md`. Читается секция `## Findings` -- готовый плоский список атомарных находок (`### F<n>` с полями severity / origin / file / correlation / text). Опциональный `--task <TASK-KEY>` -- для имени выходного файла.

**Real finding =** finding с непустым полем `correlation` (есть коммит-фикс, тронувший упомянутый файл после ревью). Остальные -- в bucket «отброшено» с причиной (discussion-only / accepted / disagreed / не нашёлся фикс).

**Trusted weighting:** finding с тегом `[TRUSTED]` (из комментария ревьюера в списке `--trusted` команды `/mr-collect`) обрабатывается с двумя послаблениями: (1) попадает в Real finding даже при пустом `correlation` — авторитет компенсирует отсутствие явного fix; (2) при ничьей в dedup (похожая ловушка существует, но формулировка отличается) выбирается версия trusted-находки. Если `[TRUSTED]` не используется (флаг не передан) -- обработка по базовому правилу выше.

**Generalization:** для каждого real finding сформулировать Плохо / Правильно / Почему без имён конкретного проекта, классов, файлов, авторов. Если после обобщения остаётся проектная конкретика -- либо переформулировать, либо отбросить как non-generalizable.

**Fact verification (гейт достоверности, обязателен для каждого предложения до записи output):** **каждое** предложение проходит этот гейт -- пропуск запрещён. Сначала выпиши все технические утверждения ловушки: **имя API** (метод / класс / атрибут / опция), **дефолт** (поведение / значение по умолчанию), **поведение ошибки** (что бросается и когда), **сложность** (O(...)). Если утверждений нет -- ловушка чисто организационная / структурная; это фиксируется в поле `Fact check: n/a` с обоснованием, а не подразумевается молчанием. Каждое выписанное утверждение сверяется с источником истины: генеративная модель закономерно выдумывает имена API и переворачивает причинность, поэтому сверку обязан выполнить анализатор -- ревьювер-человек получает уже проверенный факт.

- **Инструмент -- каскад (выбор источника, не основание пропустить проверку):** сначала `context7` (`resolve-library-id` -> `query-docs`, официальная API-дока версии). Если библиотека в context7 не резолвится **или** факт относится к языку / общему паттерну, а не к конкретной библиотеке -- то же утверждение сверяется через `WebSearch` (факт остаётся фактом и требует сверки, меняется только инструмент). «Не нашёл в context7» само по себе проверку не закрывает: переходи к `WebSearch`, несверенным утверждение не оставляй.
- **Сверять каждое утверждение, без исключений по усмотрению:** имя API сверяется всегда, даже если «очевидно существует»; утверждение «бросает / по умолчанию / всегда / никогда / O(...)» сверяется и в примере, и в `Почему`. **`Почему` сверяется отдельно, даже когда пример заведомо верен:** перевёрнутая причинность коварнее выдуманного имени -- имя грепается, причину ловит только дока.
- **Результат каждого утверждения** сводится в строку `Critical assessment`: `Fact check: verified (context7:<lib> | websearch) | unverifiable | contradicted`. Если у ловушки несколько утверждений -- статус по худшему. `contradicted` (источник опровергает) -> исправить формулировку по источнику; правило держится только на ложном факте -> `Dropped` `unverifiable-api`. `unverifiable` (ни context7, ни WebSearch не подтвердили за разумное число запросов) -> ловушку оставить, но `Recommendation: discuss` + пометка «факт не подтверждён -- сверить вручную»; молча записывать неподтверждённый факт как verified запрещено.
- **CI gap (known):** в headless-раннере context7 пока не поднят (нет `--mcp-config`). Это **не отменяет гейт** -- весь каскад выполняется на `WebSearch` (он всегда доступен), статус становится `verified (websearch)` / `unverifiable`. Поднятие context7 в раннере -- отдельная инфра-задача; до неё проверка обязательна через WebSearch, не пропускается.
- **Концептуальный ярлык сверяется так же, как имя API:** если ловушка приписывает проблему **названному принципу / паттерну / категории** (`нарушение CQS`, `SRP`, `DIP`, `LSP`, `cohesion`, `idempotency`, `тип-сейфти`), это утверждение -- такой же факт, как имя метода, и сверяется тем же каскадом (`context7` для библиотечного термина, `WebSearch` для общего принципа). Проверяется **точность приписывания**: действительно ли описанная проблема -- нарушение именно этого принципа, или ярлык навешен мимо (типичный промах генеративной модели: назвать «нарушением CQS» проблему читаемости имени метода; CQS -- про совмещение мутации и возврата значения, а не про форму имени). Несовпадение ярлыка и сути -> либо снять ярлык и переформулировать через реальную ось проблемы, либо заменить на корректный принцип; результат в `Critical assessment` полем `Label correctness` (см. ниже). Ловушка без концептуального ярлыка этот под-пункт пропускает явно (`Label correctness: n/a`), не молчанием.

**Skill mapping:** источник правды -- `.claude-plugin/marketplace.json` + `plugins/skills/*/skills/*/SKILL.md` (Glob + Read). Если тема не покрыта -- предлагать новый skill с именем `dex-skill-<domain>-<topic>`.

**Dedup:** перед предложением Grep целевой SKILL.md по ключевым словам формулировки. Если похожая ловушка уже есть -- не предлагать, внести в секцию «пропущено». То же для agent-файлов. **Маршрут «уже покрыто» (три исхода, не путать):** (1) дубль в **целевом** SKILL.md/agent на диске -> `Skipped (already covered)`; (2) дубль в **другом** skill (не целевом) -> `Dropped` `already-covered` со ссылкой на перекрывающий skill; (3) покрыто **общими знаниями Claude / линтером** без skill-дома -> это банальность, идёт через Quality gate Non-obviousness в `Dropped` `banal-or-lint-caught`, не `already-covered`.

**Cross-check между предложениями:** dedup выше сверяет находку с тем, что уже на диске. Отдельно -- сверить предложения **этого прогона** между собой: пример «Плохо» / «Правильно» одной ловушки не должен нарушать правило другой предлагаемой ловушки (типичный случай -- одна ловушка демонстрирует в примере побочную конструкцию, которую другая ловушка этого же прогона запрещает; разобранный сценарий см. фикстуру `test-fixtures/analyze-cross-check-conflict.md`). Проверка: каждый drop-in блок прогнать против формулировок всех остальных предложений прогона. При конфликте -- переписать пример так, чтобы он не задевал чужое правило (убрать побочную конструкцию из примера, сузить кейс); если переписать без потери смысла нельзя -- одну ловушку оставить, вторую в `Dropped (not actionable)` с причиной `conflicts-with-sibling-proposal`. Какую оставлять -- по полям `Critical assessment`, детерминированно: (1) выше `generalization confidence` (`high` > `medium` > `low`); (2) при равенстве -- ниже `false positive risk` (`low` > `medium` > `high`); (3) при дальнейшем равенстве -- решение оператора, обоснование фиксируется в `apply-report.md`. Конфликт примеров между skills, уезжающими в один PR, -- то, что ревьювер видит сразу, поэтому ловить до записи output file.

**Quality gate (до записи output, после dedup и cross-check):** dedup отсекает повтор того, что уже на диске; fact-verification ловит ложный факт. Этот гейт отсекает **слабые ловушки, формально валидные**: банальные, размытые, не на своей оси. Прогоняется по **каждому** выжившему предложению; провал любого под-пункта -> либо переписать до прохождения, либо `Dropped` с указанным reason code (не молчаливое отбрасывание и не молчаливый пропуск проверки). Каждый под-пункт фиксируется полем в `Critical assessment`.

- **Non-obviousness (неочевидность).** Ловушка обязана нести знание, которое целевая аудитория skill **не получает даром**. Провал, если знание: (а) ловится компилятором / типов-чекером (`перепутанный аргумент того же типа` -- ок, `вызов несуществующего метода` -- нет, компилятор поймает); (б) ловится стандартным линтером / анализатором по умолчанию (неиспользованная переменная, дубль присваивания, недостижимый код); (в) сводится к общему лозунгу без специфики области (`не дублируй код`, `используй null-safety`, `закрывай ресурсы`). Тест: «убери технический контекст области -- останется ли что-то, кроме лозунга, который и так все знают?» Если нет -- `Dropped` `banal-or-lint-caught`. **Граница:** ловушка про *неочевидное следствие* банального на вид правила (`Dispose в финализаторе без GC.SuppressFinalize удваивает работу GC` -- не банальность, хотя «вызывай Dispose» банально) -- проходит; фиксируй специфику в `Почему`. Поле `Non-obviousness: high | medium | low`; `low` -> кандидат на drop, обоснование обязательно.
- **Axis fit (ловушка на оси целевого skill) -- роутер, не вышибала.** Skill mapping нашёл дом по домену/стеку (EF Core, LINQ, нейминг); этот под-пункт проверяет, что ловушка про **ту же ось, что заявляет skill**, а не просто «в том же стеке». **Ось определяется объективно -- по реальным H2-заголовкам целевого `SKILL.md`** (Grep `^## ` в файле), не по вольной интерпретации: «ось» = тема существующего H2-раздела. Тест: ложится ли ловушка хотя бы в один из существующих H2 skill? Ловушка, релевантная **нескольким осям**, проходит, если **хоть одна** совпадает с осью дома (пример: `AsNoTracking` -- одновременно perf и stale-данные; в perf-skill проходит по perf-оси, даже если вторая ось -- корректность). `mismatch` ставится, только когда тема ловушки **не пересекается ни с одним H2** skill и потребовала бы завести раздел из чужой области (пример: «фильтр не применён -> неверный результат» -- корректность контракта; в `dex-skill-*-linq-optimization`, где H2 про материализацию / аллокации / round-trips, своей оси не находит). **`mismatch` не ведёт в `Dropped`** -- это проблема адреса, не качества: ловушку (а) переназначить на skill с верной осью, если такой есть; (б) если верного дома нет -- `Proposed new skills`. Находка из-за оси не теряется никогда. Поле `Axis fit: <skill-name> (<H2-раздел>) | reassign -> <skill> | new-skill`.
- **Counter-example для FP-risk (активный, не оценка на глаз).** Поле `False positive risk` больше не ставится субъективно: для каждой ловушки **выписать 2-3 легитимных примера** кода, которые подходят под формулировку «Плохо» **по форме**, но являются нормой (канонический паттерн языка/фреймворка). Классические носители ложного срабатывания: `Try*`-паттерн (`TryParse`/`TryGetValue` -- «команда+bool», но норма) для правил про имена; `out`-параметры; `IAsyncEnumerable` для правил про возврат коллекций; партиал-методы; explicit-interface impl. Если хоть один контр-пример правило **ошибочно осуждает** -> сузить формулировку или добавить исключение прямо в `Правильно` (как `Try*` в #74), затем перепроверить. `False positive risk` выставляется по итогу: остались незакрытые контр-примеры -> `high`; закрыты исключением -> `low/medium`. Контр-примеры (или явное `контр-примеров нет: правило по семантике, не по форме`) пишутся в `Critical assessment` строкой `FP counter-examples:`.
- **Root not surface (корень, не поверхность).** Обобщение не должно выхолащивать находку до поверхностного слоя. Провал, если `Почему` описывает симптом, а корневую причину (ради которой ревьюер и оставил замечание) потеряло: находка «метод вернул groupId, используемый как memberId» -> поверхностно «два имени одной переменной» (банально, теряется суть), корнево «один результат под двумя доменными смыслами -- скрытый семантический конфликт либо не та абстракция метода». Тест: «отражает ли `Почему` ту проблему, из-за которой замечание возникло в исходном MR, или только её внешний признак?» Поверхностная формулировка -> переписать через корень; если корень после деобобщения оказывается проектной конкретикой и без неё ловушка пуста -> `Dropped` `surface-not-root` (а не выпуск банальной оболочки).

**Agent gaps:** если находки показывают класс проблем, который существующие агенты не проверяют (не просто «в этом MR пропустили», а систематический gap в workflow), предложить точечную правку конкретного agent-файла: новая фаза / расширение чек-листа / дополнительный Skill tool вызов.

**Output file:** `/tmp/mr-analyze-<task-key>-<YYYYMMDD-HHMM>.md`. Task key из флага `--task`, иначе из Metadata collect-файла или его имени, иначе `no-task`.

**Output sections:** Proposed skill additions / Proposed new skills / Proposed agent changes / Skipped (already covered) / Dropped (not actionable). Каждое предложение -- с drop-in markdown-блоком формата `Плохо:` / `Правильно:` / `Почему:` (триада с двоеточиями, синхронизировано с regex валидатора `tools/validate-skill.js`), категорией H2 в skill, именем skill или agent-файла.

**Nothing-to-propose behaviour:** если после фильтрации и dedup осталось ноль новых ловушек, ноль новых skills, ноль правок агентов -- файл **не создавать**. Stdout: `Ничего не найдено для предложений: все находки либо уже покрыты, либо не actionable.`

**Critical assessment per proposal:** к каждому предложению (skill addition / new skill / agent change) перед drop-in блоком в отчёт добавляется блок самооценки -- ревьюверу PR видны границы уверенности анализатора. Поля (владелец указан у каждого: `Fact check` и `Label correctness` -- из Fact verification; `Non-obviousness`, `Axis fit`, `FP counter-examples` -- из Quality gate; Root-not-surface отдельного поля не имеет -- влияет на формулировку `Почему` и reason code `surface-not-root`; остальные -- базовая самооценка):

- **Generalization confidence: high | medium | low** -- насколько правило обобщилось без потери смысла. `high` обязательно с обоснованием в 1 предложение (`high` без объяснения -- подозрительно)
- **Systemic vs incidental: systemic | incidental** -- ловит класс ошибок или один случай в одном MR. Исход `incidental` детерминирован скрещиванием с `Non-obviousness` (не «на усмотрение»): `incidental` + `Non-obviousness: low` -> `Dropped` `incidental` (один случай И знание не несёт неочевидной ценности -- фиксировать в skill незачем); `incidental` + `Non-obviousness: medium/high` -> ловушка **остаётся** в отчёте с `Recommendation: discuss` и пометкой «единичный, но неочевидный -- решает ревьювер» (редкий, но ценный случай молча не выбрасываем)
- **False positive risk: low | medium | high** -- будет ли правило ругаться в нормальных кейсах. Выставляется **по итогу counter-example-проверки Quality gate**, не на глаз: остались легитимные паттерны, которые правило ошибочно осуждает -> `high`; закрыты исключением в `Правильно` -> `low/medium`. `high` -- кандидат на переформулировку
- **FP counter-examples: <2-3 паттерна> | нет (правило по семантике, не по форме)** -- легитимные конструкции, проверенные на ложное срабатывание (см. Quality gate). Не опускать: пустое поле = проверка не сделана
- **Already covered elsewhere: no | partial | yes** -- пересечение с существующими skills / документацией Claude / здравым смыслом. `partial`/`yes` -- ссылка на конкретный skill. Маршрут при `yes`: дубль в целевом skill -> `Skipped`; в другом skill -> `Dropped already-covered`; только «здравый смысл», без skill -> идёт в Quality gate Non-obviousness (`banal-or-lint-caught`), см. Dedup
- **Non-obviousness: high | medium | low** -- несёт ли ловушка неочевидное знание (не лозунг, не то что ловит компилятор/линтер; см. Quality gate). `low` -- кандидат на drop `banal-or-lint-caught`, обоснование обязательно
- **Axis fit: <skill-name> (<H2-раздел>) | reassign -> <skill> | new-skill** -- ложится ли ловушка хотя бы на один реальный H2-раздел целевого skill, а не «в том же стеке» (см. Quality gate). Промах оси -> `reassign` на skill с верной осью либо `new-skill`; в `Dropped` из-за оси не уходит никогда
- **Label correctness: correct (<принцип>) | mislabeled -> <верный> | n/a** -- точность концептуального ярлыка (имя принципа/паттерна), сверена в Fact verification. `n/a` -- ловушка без ярлыка. `mislabeled` -- ярлык снят/исправлен до записи
- **Fact check: verified (context7:<lib> | websearch) | unverifiable | contradicted | n/a** -- результат Fact verification (см. раздел выше). `n/a` -- только если ловушка не содержит ни одного факта об API / дефолте / поведении ошибки / сложности **и** ни одного концептуального ярлыка (чисто организационная / структурная конвенция), и это явно обосновано одной фразой (не ставится молчанием при наличии факта или ярлыка). `unverifiable` / `contradicted` -- влияют на Recommendation ниже

После полей -- одна строка **Recommendation for reviewer: apply as-is | apply with edits | discuss | reject**. Это **сигнал ревьюверу**, не приказ -- ревьювер PR может перечеркнуть. Логика по умолчанию: `high gen + systemic + low FP + no overlap + Non-obviousness high + Axis fit верна (ok ИЛИ после reassign на правильный дом) + Label correct/n/a + Fact check verified/n/a` → `apply as-is`; любой `low`/`incidental`/`high FP`/`partial`/`Non-obviousness low` → `apply with edits` или `discuss`. **`Axis fit reassign` сам по себе рекомендацию НЕ понижает** -- переназначение на верный skill исправляет адрес, а не качество ловушки; переназначенная ловушка с чистыми остальными полями остаётся `apply as-is` (для правильного skill). **`Fact check: unverifiable` → не выше `discuss`** (факт не подтверждён -- ревьювер обязан сверить). **`Label correctness: mislabeled`, не исправленный до записи, → не выше `discuss`.** Если анализатор сам ставит `reject` -- предложение всё равно остаётся в отчёте (не переносится в `Dropped`), чтобы ревьювер видел ход мысли. Ловушка, проваленная под-пунктом Quality gate **без возможности переписать** (банальность не убирается, корень -- неустранимая проектная конкретика), уходит в `Dropped` с соответствующим reason code, а не остаётся как `reject`. Промах оси (`Axis fit`) исключение -- он не качество, а адрес: всегда `reassign` / `new-skill`, в `Dropped` не уходит.

Формат блока в отчёте -- между названием предложения и drop-in блоком, **отдельно** от drop-in (чтобы `/mr-apply` копировал в SKILL.md только drop-in, без метаданных).

**Validation constraints (synced with `tools/validate-skill.js`):** drop-in блок каждой ловушки должен проходить валидатор без ошибок -- иначе валидатор маркетплейса упадёт на ревьюверском PR, и ловушку придётся переписывать вручную. Решение «принять / переписать / выкинуть» -- за ревьювером PR, наша задача -- не подкладывать ему заведомо не-валидный материал. Применять при генерации, не оставлять на «потом починят»:

- Code fence ≤ 12 строк (`code-fence-too-long`). Это **потолок**, не цель -- стремиться к 1-3 строкам (pointer, not road). Длиннее 12 строк -- ужать (однострочный `if`, объединение цепочек, опустить boilerplate `using` / обёртки `{}`); если без потери смысла не получается -- в `Dropped (not actionable)` с причиной `cannot-fit-fence-limit`
- Триада обязательна: `Плохо:` / `Правильно:` / `Почему:` -- каждая ловушка содержит все три строки (`trap-missing-triad`)
- Подзаголовок ловушки -- `###` внутри существующей H2-категории. Не `####`, не `##`
- Drop-in блок не повторяет дословно формулировки существующих ловушек целевого SKILL.md (dedup-проверка обязательна *до* финального предложения)
- Имена конкретного проекта (классы, файлы, namespaces из MR) в drop-in блоках запрещены. Placeholder-имена: `Service`, `Calculator`, `Options`, `MaxSize`, `_logger`, `Result`
- Размер итогового SKILL.md после добавления ловушки: **ошибка** валидатора на ≥ 250 строк (`size-exceeds-recommended` -- это `ERROR`, не warning: `npm run validate` упадёт), хард-лимит 500 (`size-exceeds-hard-limit`). Если целевой SKILL.md близок к 250 -- фиксировать в Pre-analysis и оставлять только самые ценные ловушки (250 -- сигнал «skill переполнен, новую ловушку в соседний / новый skill», а не растить дальше)
- **Description -- длина и обязательные поля:** description целевого skill (если правится) 50-250 символов; обязательная фраза `Активируется при` с ≥ 10 keywords через запятую после неё (`description-no-activation` / `description-few-keywords`)
- **Description -- выбор активаторов:** активаторы -- **аспекты** (разделы H2 / законченные темы skill: `## Concurrency`, `## Mapping`), не **точки входа** (конкретные API / симптомы внутри одной ловушки: `OwnsOne`, `ExecuteDelete`, `cache miss`) и не широкие слова, пересекающиеся с другими skills (`timeout`, `batch`, `interval` -- ложные срабатывания). Description целевого skill править **только** если новая ловушка вводит новый аспект (новый раздел H2); частный кейс уже покрытой темы -- description не трогать (`SKILL_FRAMEWORK.md`, «Когда менять description»). При правке description приводить набор keywords в соответствие со **всеми** текущими H2-разделами skill, не только с дельтой этого прогона: если ловушка попадает в существующий H2, который до сих пор не отражён в активаторах -- keyword добавить; если новый H2 вытесняет тему, давно оставшуюся без ловушек, -- устаревший активатор убрать. Для **нового** skill description строится по тому же правилу: якорь технологии + аспекты-разделы
- **Description -- язык:** на английском для записи в `marketplace.json` и `plugin.json`, на языке проекта -- во frontmatter `SKILL.md`. Правило применяется к **новым и изменяемым** описаниям; массовая миграция уже существующих локализованных описаний в `marketplace.json` / `plugin.json` -- отдельная задача, `/mr-analyze` не правит язык описания у skill, который сам не трогает
- Минимум ловушек в skill ≥ 5 (`too-few-traps`) -- для *нового* skill: если находок MR на тему < 5, **досыпать штатные ловушки темы** из общих знаний (грабли области, не из MR), помечая их `origin: типовая ловушка темы`. Цель -- осмысленная заготовка, а не огрызок. **Каждая досыпанная ловушка проходит Fact verification** (см. раздел выше) -- именно здесь доля галлюцинаций API максимальна, факт «из общих знаний» проверяется тем же гейтом, что и факт из находки; имя API, не подтверждённое context7 / WebSearch, не записывается (формулировку переписать без него либо ловушку не досыпать). Если и так < 5 -- skill всё равно предлагать (см. «Новый skill» в Constraints), валидатор упадёт на ревьюерском PR -- это сигнал «добери ловушек», решение за оператором, не повод отбрасывать находку

Правило компромисса: лучше короткий пример из 2-3 строк с pointer-комментарием `// + ещё N полей опускаем`, чем 7-строчный «как в жизни» блок, который автомат отбросит. Нюансы выноси в `Почему:`.

**Constraints:**

- Не применять предложения автоматически -- только текст с drop-in блоками
- Skill не может загрузить другой skill. Если ловушка тематически пересекается с другим skill -- оформить как справочную сноску `> см. dex-skill-X` в теле SKILL.md, не как механизм загрузки
- Ловушка, в формулировке которой остаются имена конкретного проекта, -- не предлагается
- Skill, в котором уже есть похожая ловушка, -- не получает дубликат
- **Новый skill** -- легитимный исход наравне с правкой существующего. Предлагается, когда тема не покрыта ни одним skill. Отсутствие целевого skill (skill-дома) -- **не причина** для `Dropped`: обобщаемая находка без дома → `Proposed new skills`, не отбрасывается
- Перед записью output file прогонять mental-self-check по **Validation constraints** выше; ценную ловушку, не проходящую лимит, -- пытаться переписать (короче пример, pointer-комментарий `// + ещё N полей опускаем`, нюансы в `Почему:`); если и после переписывания не лезет -- переносить в `Dropped (not actionable)` с явной причиной (`cannot-fit-fence-limit`, `cannot-deproject-name`, etc.). Не отсекать ради отсечения -- чем больше выживших ловушек, тем больше пользы для ревьювера PR
- `Dropped` -- только по существу находки: `incidental` (**только** в паре с `Non-obviousness: low` -- см. порог в Critical assessment; единичный, но неочевидный случай в `Dropped` не уходит) / `non-generalizable` / высокий FP-risk / уже покрыто здравым смыслом или существующим skill. Организационные причины («нет skill-дома», «на новый skill не набрали 5 ловушек») в `Dropped` **запрещены** -- это триггер `Proposed new skills`, не отбрасывания

**Reason codes (`Dropped (not actionable)`):** причина каждого `Dropped`-предложения -- значение из закрытого списка, не свободный текст; перечень пополняется при изменении правил `/mr-analyze`. Валидные: `incidental` (один случай в одном MR, не класс, **и** `Non-obviousness: low` -- единичный + неочевидный остаётся как `discuss`, не дропается), `non-generalizable` (после обобщения остаётся проектная конкретика), `cannot-fit-fence-limit` (пример не лезет в code fence ≤ 12 строк после ужатия), `cannot-deproject-name` (имя проекта в формулировке неустранимо), `high-false-positive-risk` (правило ругается в нормальных кейсах -- остались незакрытые counter-examples, см. Quality gate), `already-covered` (ловушка по сути совпадает с уже существующей в каком-то skill, но **не в целевом** -- на диске целевого Grep дубля не нашёл, поэтому не `Skipped`, а Drop с указанием перекрывающего skill; «покрыто здравым смыслом / линтером» сюда **не** относится -- это `banal-or-lint-caught`, не плодить дубль причин), `conflicts-with-sibling-proposal` (пример конфликтует с правилом другого предложения прогона, см. Cross-check), `unverifiable-api` (факт об API / поведении, на котором держится ловушка, опровергнут источником при Fact verification, либо не подтверждён и переписать формулировку без него нельзя), `banal-or-lint-caught` (Quality gate: знание очевидно / ловится компилятором или линтером / сводится к лозунгу без специфики области, см. Non-obviousness), `mislabeled-principle` (Quality gate: концептуальный ярлык навешен мимо сути и без него ловушка пуста, см. Label correctness / Fact verification), `surface-not-root` (Quality gate: после обобщения остаётся поверхностный слой, корень -- неустранимая проектная конкретика, см. Root not surface).

**Errors:**

- Входной файл не найден → показать ожидаемый формат пути
- Файл без секции `Findings` → явная ошибка: ожидается формат `/mr-collect`
- Команда запущена вне корня маркетплейса (`plugins/skills/*` пусто) → показать инструкцию
