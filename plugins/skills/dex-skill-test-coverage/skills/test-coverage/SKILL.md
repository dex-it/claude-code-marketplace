# Контракт покрытия - как измерить и обеспечить порог

Дом порога и метрики покрытия для тест-агентов. Загружается в фазе Validate: измерить охват
числом из инструмента, свериться с порогом, при недоборе дописать сценарии; недостижимость и
деградацию метрики - явным статусом в handoff, не молчанием. Покрытие - факт из прогона
coverage-инструмента, не оценка на глаз.

## Порог и scope

- **Метрика:** branch coverage, где тулчейн даёт её нативно. Branch строже line/statement:
  полный line-охват берётся одним входом, branch требует пройти обе стороны каждого условия.
  Где branch недоступен - деградация (таблица + правило ниже).
- **Порог:** `>= 75%`.
- **Scope:** файлы под тестом (`diff-scope`), **per-file** - порог на каждом файле отдельно, не
  усреднённо: среднее по scope маскирует голый файл жирным покрытым.

## Матрица «язык -> метрика -> включение»

Branch coverage поддержан не везде нативно. Целевая метрика - branch там, где тулчейн её даёт;
где нет - деградация по правилу ниже, не отмена порога.

| Язык | Метрика | Как включить |
|------|---------|--------------|
| C# / .NET | branch | coverlet (`--collect:"XPlat Code Coverage"` или `/p:CollectCoverage=true`), порог `/p:ThresholdType=branch /p:Threshold=75` |
| JS / TS | branch | Vitest/Jest, provider `v8` или `istanbul`; отчёт даёт `branches`, порог `coverage.thresholds.branches` |
| Python | branch | coverage.py с флагом `--branch` (или `run.branch = true`); без флага меряет только statement |
| PHP | branch | PHPUnit + Xdebug (branch/path coverage); PCOV даёт только line - для branch нужен Xdebug |
| Go | **statement** | `go test -cover` даёт statement, не branch (cmd/cover branch не поддерживает); branch - только сторонний `gobco` отдельным прогоном |

## Правило деградации метрики

Где branch недоступен в тулчейне (Go нативно), метрика деградирует, порог - нет:

- Мерить доступную метрику (statement), порог `>= 75%` трактовать **по доступной метрике**.
- В handoff `coverage.type` = фактическая метрика (`branch` / `statement`) - потребитель видит,
  чем измерено, и не принимает statement за branch.
- Недоступность branch в тулчейне - **не** повод не измерять и не блокировать: это свойство
  языка, а не недоделка. Тянуть сторонний инструмент (gobco) для branch - по явному решению
  вызывающего, не самовольно.

## Правило недостижимости порога

Порог не добит на файле при доступной метрике - возврат наверх статусом, не тихое занижение и
не жёсткий fail. Причина берётся из **закрытого перечня**; «недостижимо» без причины из него -
не принимается:

- ветки от сгенерированного / framework-кода в scope (партиалы, designer-код, сериализаторы);
- ветки достижимы только через integration / e2e, а узел пишет unit - фиксируется как
  «нужен integration-тест» и выносится наверх, а не молча закрывается;
- принципиально недостижимые defensive-ветки (`default` у exhaustive switch, «невозможная»
  ветка после исчерпывающей проверки).

Реакция: `status = partial` (не `complete`), поле `coverage` с фактическим числом и `type`,
обоснование недобора из перечня выше + что осталось непокрытым. Маршрут решает оркестратор.

*Чем* добивать непокрытые ветки (какие сценарии дописать) - не здесь: техники подбора тест-кейсов
(BVA, equivalence partitioning, negative) держит `dex-skill-test-design`.

## Coverage-gate: запрет тихого пропуска

Результат не идёт наверх, если любой пункт нарушен:

- нет прогона coverage-инструмента (число «на глаз» не считается измерением);
- порог проверен агрегатом по scope, а не per-file;
- недобор порога без `partial` + причины из закрытого перечня (тихое занижение или петля на недостижимом - запрещены);
- метрика в handoff без пометки `type` (statement, выданный за branch);
- инструмент физически отсутствует в среде (не «пропущен по умолчанию»), но `coverage` не помечен `unmeasured` + причина с просьбой прогнать источнику вызова.
