# Исследовательское тестирование - ловушки и anti-patterns

## Оракул и риск

### Тестируется только happy path
Плохо: проверяется заявленный сценарий "как задумано", негативные и граничные пути не трогаются
Правильно: целиться в риск - невалидный ввод, пустые / огромные значения, обрыв на середине, конкуррентный доступ
Почему: баги живут на краях и в необработанных путях, не на happy path; проверка только golden path даёт ложную уверенность

### Нет оракула корректности
Плохо: критерий "не упало" - если нет исключения, считается, что работает
Правильно: для каждой проверки иметь оракул - что именно считается правильным результатом (спека, инвариант, эталон)
Почему: "не упало" не равно "правильно" - молчаливо неверный результат (не та сумма, потерянная запись) проходит мимо без оракула

### Атака только на заявленные требования
Плохо: проверяются только пункты ТЗ, неявные предположения (порядок, идемпотентность, время) не атакуются
Правильно: явно выписать предположения системы и атаковать каждое
Почему: большинство дефектов в том, что не записано в требованиях - в подразумеваемых инвариантах, которые никто не проверял

## Чартеры и эвристики

### Бесцельное блуждание вместо чартера
Плохо: хаотичные клики "посмотрю, что будет" без цели сессии
Правильно: вести по чартеру - "исследовать область X на риск Y инструментами Z" в ограниченное время
Почему: без чартера покрытие случайно, области пропускаются, а время уходит на повтор уже проверенного

### Эвристики не применяются
Плохо: каждая область тестируется с нуля, без набора эвристик-туров
Правильно: применять туры - граничные значения, типы данных, CRUD, прерывание операции, обратный порядок шагов
Почему: эвристики систематизируют атаку и быстро вскрывают типовые классы дефектов, которые ad-hoc проверка пропускает

## Фиксация находок

### Находка без контекста воспроизведения
Плохо: "иногда падает" без шагов, данных и окружения, в которых это случилось
Правильно: фиксировать сессию - что делал, с какими данными, в каком окружении, что наблюдал
Почему: находка без контекста невоспроизводима и теряется; разработчик не может её подтвердить и закрыть

## Чек-лист исследования

- Цель сессии задана чартером, не "посмотрю что будет"
- Для каждой проверки есть оракул правильного результата, не "не упало"
- Атакованы неявные предположения, не только пункты ТЗ
- Применены туры-эвристики (границы, типы данных, прерывание, порядок)
- Каждая находка зафиксирована с шагами, данными и окружением
- Покрыты негативные и граничные пути, не только happy path
