# Дрейф контрактов - ловушки и anti-patterns

## Поля и форматы

### Поле или формат сменили на одной стороне
Плохо: бэк переименовал / переформатировал поле, клиент читает по старому имени / формату
Правильно: менять контракт на обеих сторонах согласованно либо через версионирование, не в одностороннем порядке
Почему: одностороннее изменение поля тихо ломает десериализацию у потребителя - null вместо значения, пустой экран, без явной ошибки на сервере

### Статус-код изменился
Плохо: эндпоинт сменил код (200 на 204, 404 на 200, 400 на 422), клиент ветвится по старому
Правильно: считать статус-код частью контракта; менять его - breaking, согласовывать с потребителями
Почему: клиент принимает решения по коду; смена кода уводит его в неверную ветку - показывает успех на ошибке или наоборот

### Nullability или обязательность поля изменили
Плохо: поле стало nullable / перестало приходить / стало обязательным без уведомления другой стороны
Правильно: изменение nullability и обязательности трактовать как breaking и согласовывать
Почему: клиент, рассчитывавший на присутствие поля, падает на null; сервер, ждущий обязательное, отвергает старые запросы

## Версии и совместимость

### Версии API разъехались
Плохо: на стенде развёрнуты несовместимые версии сторон (бэк v2, клиент под v1)
Правильно: сверять фактические версии контракта на обеих сторонах стенда
Почему: рассинхрон версий даёт симптом без изменения кода одной стороны - ломает то, что в самой стороне корректно

### Расширение enum или набора значений
Плохо: сервер начал отдавать новое значение enum / тип, клиент падает на неизвестном
Правильно: проектировать потребителя устойчивым к неизвестным значениям; расширение набора согласовывать
Почему: жёсткий разбор закрытого набора ломается при расширении - одна новая константа кладёт клиента целиком

## Граница и проверка

### Контракт нигде не зафиксирован
Плохо: между сторонами нет общей схемы (OpenAPI / proto / типы), дрейф невидим до рантайма
Правильно: держать контракт явной общей схемой, проверять обе стороны против неё
Почему: без зафиксированного контракта расхождение всплывает только инцидентом в проде, а не на сборке

## Чек-лист контракта

- Имя и формат поля совпадают на обеих сторонах
- Статус-коды трактуются как часть контракта
- Nullability и обязательность полей согласованы
- Фактические версии контракта на стенде сверены между сторонами
- Потребитель устойчив к неизвестным значениям enum / типов
- Контракт зафиксирован общей схемой, обе стороны проверены против неё
