# .NET Caching — ловушки и anti-patterns

## Stampede

### Ожидание single-flight от IMemoryCache.GetOrCreate
Плохо: полагаться, что `IMemoryCache.GetOrCreateAsync` вызовет фабрику один раз при конкурентных промахах
Правильно: для защиты от stampede взять `HybridCache.GetOrCreateAsync` (коалесцирует конкурентных вызывающих) либо вручную обернуть фабрику в `SemaphoreSlim` per-key
Почему: `IMemoryCache` потокобезопасен для чтения/записи, но НЕ коалесцирует — N конкурентных промахов запускают N фабрик (stampede, N запросов в БД). `HybridCache` гарантирует одну фабрику на ключ внутри инстанса

## Инвалидация

### Ручная инвалидация покрывает меньше тегов, чем авто-инвалидация
Плохо: debug-эндпоинт сбрасывает один тег (`RemoveByTagAsync("A")`), а авто-инвалидация в бизнес-операциях сбрасывает три (`A`, `B`, `C`)
Правильно: служебный эндпоинт сбрасывает то же множество тегов, что и авто-путь (общий метод-хелпер/`/cache/all`), либо параметризуется тегом
Почему: инженер на стенде дёргает «сбросить кеш», видит, что часть данных стейл, и делает ложный вывод «фикс не работает» — хотя не сброшен второй тег. Ручной и авто путь инвалидации должны быть симметричны

### Запись кэшируется без тега, по которому её потом инвалидируют
Плохо: `GetOrCreateAsync(key, factory)` без `tags`, а инвалидация делается `RemoveByTagAsync(tag)` — запись по тегу не сбрасывается
Правильно: задавать `tags` при записи (`GetOrCreateAsync(key, factory, tags: [tag])`), согласованно с тем тегом, по которому будет инвалидация
Почему: `RemoveByTagAsync` сбрасывает только записи, помеченные тегом при создании. Запись без тега переживает инвалидацию и остаётся стейл — рассинхрон между точкой записи и точкой сброса

## Негативное кэширование

### Негативное кэширование вводят спекулятивно
Плохо: кэшировать пустой/ошибочный результат «на всякий случай», добавляя отдельный negative-TTL и инвариант «пусто ⇒ короткий TTL»
Правильно: не кэшировать пустые/ошибочные результаты, пока нет доказанного thundering herd; per-result TTL задавать в фабрике осознанно
Почему: негатив-кеш добавляет инвариант между двумя TTL и класс багов «пустой результат закэширован на полный TTL». Без замеренной нагрузки это сложность без выгоды; ошибки кэшировать тем более опасно — стейл-ошибка живёт до инвалидации

### Невалидный negative-TTL молча отключает негатив-кеш
Плохо: `if (negTtl < fullTtl) { cache empty short }` — при `negTtl >= fullTtl` (опечатка единиц/копипаст) ветка не выполняется, пустой результат кэшируется на полный TTL
Правильно: валидировать `negTtl < fullTtl` явно (бросок в init/Validate), а не тихим `if`; инвариант «negTtl задан ⇒ результат может быть пустым» проверять до использования
Почему: тихий `if` превращает опечатку в TTL в незаметно отключённый негатив-кеш с худшим поведением — пустой результат живёт полный TTL. Это ровно тот баг, против которого затевалась фича
