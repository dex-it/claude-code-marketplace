# Agile — ловушки

## Правила

- Story = ценность для пользователя, не техническая задача
- 13+ points = epic, РАЗБЕЙ на stories <=8
- DoR перед планированием, DoD перед приёмкой
- Velocity — 3-sprint rolling average, НЕ сравнивай между командами

## Анти-паттерны

| Анти-паттерн | Проблема | Решение |
|--------------|----------|---------|
| Water-Scrum-Fall | Все stories написаны заранее на квартал | Just-in-time refinement |
| Техзадача как Story | "Рефакторинг auth-сервиса" | Переформулируй через user value |
| 13+ SP в спринте | Story не влезает — растягивается | Split на <=8 SP |
| Нет AC | "Сделать фичу X" | Given-When-Then сценарии |
| Цепочка зависимостей | A→B→C→D, одна блокирует всё | Минимизируй или объединяй |
| Всё — P0 | Нет приоритизации | Квота: max 15% items = P0 |
| HIPPO | "Директор сказал" без данных | RICE scoring + обоснование |
| Velocity как KPI | "Ускоряемся каждый спринт" | Velocity = прогноз, не цель |

## Story — частые ошибки

```
Плохо: "Внедрить Redis для кэширования" (техзадача, нет value)
Хорошо: "Как пользователь, я хочу получать результаты поиска за <2 сек"

Плохо: "Улучшить поиск" (слишком размыто, нет AC)
Хорошо: "Как пользователь, я хочу фильтровать товары по цене,
         чтобы найти товары в рамках бюджета"
         AC: Given фильтр по цене 1000-5000
             When применяю фильтр
             Then вижу только товары в этом диапазоне

Плохо: "Как пользователь, я хочу чтобы данные кэшировались в Redis"
       (технология в story — пользователю всё равно)
Хорошо: "Как пользователь, я хочу чтобы страницы загружались за <2 сек"
        (Redis — в Technical Notes)
```

## Splitting — когда story слишком большая

| Стратегия | Пример |
|-----------|--------|
| По workflow steps | Регистрация → Подтверждение email → Настройка профиля |
| По бизнес-правилам | Доставка: domestic → international → бесплатная |
| По данным | Импорт: CSV → Excel → JSON API |
| По Simple/Complex | Поиск по имени → + фильтры → + сортировка → + full-text |

## Чек-лист

- [ ] Stories формулируются через user value, не техзадачи
- [ ] AC в формате Given-When-Then
- [ ] Нет stories >8 SP (split!)
- [ ] DoR проверяется перед planning
- [ ] DoD проверяется перед приёмкой
- [ ] Velocity — rolling average, не цель для роста
- [ ] Backlog refinement каждый спринт (just-in-time)
