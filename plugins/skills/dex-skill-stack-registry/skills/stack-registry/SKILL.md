# Реестр стеков — как общий агент грузит профильные skills

Языко-агностичные агенты (`dex-debugger`, `dex-feature-implementer`,
`dex-security-reviewer`, `dex-self-reviewer`, `dex-mr-reviewer`,
`dex-mr-check-reviewer`, `dex-code-discovery`) не несут зашитого списка имён skills.
Способ загрузки **един для всех**: каждый грузит этот реестр, **детектит стек проекта**
и **конструирует имена** профильных skills по нему — без ссылок «грузи как в другом
агенте». Новый стек = новая строка реестра + `dex-skill-<стек>-*` skills; **тела
агентов при этом не правятся**.

## Реестр: детект стека → префикс skills

| Стек | Признак в репозитории (манифест/файлы) | Префикс skills |
|------|----------------------------------------|----------------|
| `dotnet` | `*.csproj`, `*.sln`, `Directory.Build.props` | `dex-skill-dotnet-*` |
| `ts` | `package.json`, `tsconfig.json`, `*.ts`/`*.tsx` | `dex-skill-ts-*` |
| `python` | `pyproject.toml`, `requirements.txt`, `setup.py`, `*.py` | `dex-skill-python-*` |

Моно-репозиторий = разный стек в разных папках: детект **по папке затронутого кода**,
не по корню. Стека нет в реестре → см. «Fallback».

## Правило отбора: стек × тема (не «грузи всё»)

Префикс отбирает **семейство стека**; тема работы отбирает **подмножество внутри
него**. Грузить весь `dex-skill-<стек>-*` запрещено — только релевантное теме.

- **Плохо:** определил стек `dotnet` → загрузил все 11 `dex-skill-dotnet-*`.
- **Правильно:** стек `dotnet` + тема «N+1 / медленный запрос» → `dex-skill-dotnet-ef-core` + `dex-skill-dotnet-linq-optimization`; тема «дедлок async» → `dex-skill-dotnet-async-patterns`.
- **Почему:** нерелевантные skills раздувают контекст и зашумляют чек-лист находками не по теме. Отбор = пересечение двух осей, а не одна ось «весь стек».

**Как конструировать имена.** Из видимого списка available-skills отфильтруй по
префиксу стека, затем сузь по теме (имя/description skill). Полная форма вызова —
`{plugin}:{skill}` (например `dex-skill-dotnet-di:dotnet-di`). Список available-skills
— источник реальных имён; реестр — только маска префикса.

**Префикс и description — две разные оси, не дубль.** `name` и `description` каждого
skill видны агенту **до** загрузки (тело SKILL.md — нет, оно приходит только при
вызове Skill tool). Поэтому:
- **префикс имени** = жёсткая ось **стека** (детерминированно: строка матчится или нет — без интерпретации);
- **description** = мягкая ось **темы** внутри стека (по смыслу: какой из ts-скиллов релевантен утечке памяти).

Отбор только по description (без префикса) — ненадёжен: формулировки описаний
разнятся, агент колеблется на границе стека. Отбор только по префиксу (без description)
— грузит лишнее. Нужны обе оси: префикс берёт семейство, description сужает по теме.
Авто-активация платформы по description **в субагенте не срабатывает надёжно** —
поэтому отбор делает сам агент явным чтением видимого списка, не полагаясь на неё.

## Тематические skills — по теме, не по стеку

Часть skills не привязана к языку backend и грузится **по теме**, в дополнение к
стековым, независимо от детектированного стека:

- безопасность — `dex-skill-owasp-security` (всегда при эндпоинтах/доступе к данным);
- архитектура/принципы — `dex-skill-solid`, `dex-skill-clean-architecture`, `dex-skill-testability`, `dex-skill-nfr`, `dex-skill-microservices`, `dex-skill-distributed-resilience`;
- инфраструктура — `dex-skill-docker`, `dex-skill-kubernetes`, `dex-skill-observability`;
- frontend/E2E — `dex-skill-react`, `dex-skill-playwright` (по теме «фронт/SPA/E2E», не по backend-стеку).

Эти имена **не несут стекового префикса намеренно** — они кросс-стековые. Не пытайся
искать их под `dex-skill-<стек>-*`.

## Fallback: стека нет или skills не установлены

- **Стек не в реестре** (например мобайл/Go/Rust): работай знаниями + тематическими skills, пометь в отчёте «профильных skills под стек нет».
- **Префикс есть, но `dex-skill-<стек>-*` не установлены** в сессии: фильтр по префиксу даёт пусто. Это **не ошибка** — вызов несуществующего имени возвращает мягкую ошибку `Unknown skill`, фаза не падает. Работай на принципах, пометь «частных skills под стек нет».
- **Skill tool недоступен**: пропусти загрузку, укажи в отчёте.

## Чек-лист загрузки

1. Детектировал стек по манифесту папки затронутого кода (не по корню моно-репо).
2. Взял префикс `dex-skill-<стек>-*` из реестра.
3. Отфильтровал видимый available-skills по префиксу, сузил по теме работы.
4. Добавил тематические skills по теме (owasp/solid/...), не по стеку.
5. Вызвал в форме `{plugin}:{skill}`; пустой результат под стек → fallback, не стоп.
