# Оптимизация markdown-инструкции под LLM

<!-- skill-type: process -->

Артефакт читает **модель**, не человек. Цель: донести механизм/условие/исход минимумом токенов без
потери однозначности и нормативной силы. CLAUDE.md и тело агента в контексте на каждом проходе, тело
skill - по срабатыванию, `description` - всегда; лишний токен бьёт там, где артефакт в контексте, а
чем длиннее носитель, тем выше риск, что важное в нём утонет.

Единственное исключение из scope: артефакт читает человек глазами (README, маркетинг, UI-копирайт) -
тогда скилл не применять.

> Принцип «писать для LLM» живёт в CLAUDE.md (правило «Формулировки в артефактах пишутся для LLM …»); здесь его операционализация.
> Текст, который читает человек: dex-skill-output-hygiene.

## Приоритеты при конфликте

Строгий порядок. Нижний приоритет НИКОГДА не жертвует верхним; минимизация токенов уступает всегда.

1. **Нормативная сила и грабли.** Лазейки закрыты, обязательные статусы на месте, escape-условия узкие, границы инструмента сохранены. Непримиримо ни с чем ниже.
2. **Однозначность.** Каждая единица читается одним способом. Жертвовать ради неё можно только токенами, не нормативной силой.
3. **Минимум токенов.** Режется ТОЛЬКО то, что не задевает нормативную силу и однозначность.

Гейт на каждую правку (он же финальная проверка): принять, только если ВСЕ пункты выполнены - смысл сохранён, однозначность не упала, нормативная сила и грабли на месте, escape-условия не шире прежних. Любой пункт не выполнен -> откатить правку.

## Что резать

- **Повествовательную воду и вводные обороты.** «Стоит отметить, что», «как известно», «в целом», «давайте рассмотрим» - выкинуть. Критерий: убирается ли слово/оборот без потери смысла или однозначности? Да -> убрать.
- **Историю и мотивацию установки.** «Раньше тут было», «изначально», команды инсталляции (`uv tool install …`, `dotnet tool install …`) - модель применяет инструмент, а не переустанавливает. Оставить *как вызвать*, не *как поставить*.
- **Дубли.** Одна мысль/факт - один нормативный дом. Повтор в двух местах -> ссылка или слияние. Две строки таблицы про родственные команды (`cfg`/`dfg`/`slice`) -> одна.
- **Развёрнутые bash-примеры**, когда приём держится в однострочнике без потери граблей и флагов. Несколько примеров одного приёма -> один канонический, остальные через `·`-перечень.
- **Пересказ кода словами.** Рядом код-блок/команда -> словесное описание того же не нужно.

## Что нельзя ослаблять (приоритет «нормативная сила и грабли»)

- **Нормативную силу.** Сжимаем подачу, не правило. Лазейки, обязательные статусы (`n/a`+почему, `unverifiable`, `contradicted`), обоснования «почему mandatory» - не сокращать. Лаконичность ≠ ослабление проверки.
- **Escape-условия - сужать.** Размытый escape - дефект: исполнитель сам решает, что «не нужно», и закрывает шаг молчанием. Сузить до конкретного условия + фиксации пропуска явным статусом; недоступность инструмента переключает на запасной, не отменяет шаг. Плохо: «пропусти, если не смог». Правильно: «пропусти ТОЛЬКО если X недоступен (`command -v X` пуст); зафиксируй статусом `n/a` + причина». Это ужесточение, не рез.
- **Грабли и границы инструмента.** «C#: статики не видит -> `LSP incomingCalls`», «`findReferences` per-symbol» - это и есть ценность, ради неё артефакт существует.
- **Однозначность.** Сокращение породило двусмысленность («может ли исполнитель прочитать так, что правило к нему не относится?») -> вернуть слово.
- **Степени свободы - строгость по умолчанию.** По умолчанию закрывать лазейки и сужать escape. Единственное исключение: у задачи реально несколько валидных подходов - тогда не выдумывать единственный обязательный путь, но зафиксировать свободу явно с критерием выбора («выбери по X: A | B»), не «по усмотрению». Свобода молчанием запрещена и в исключении.

## Как писать после реза

- **Механизм + условие + исход**, не пересказ. «Ошибка X = причина Y» вместо абзаца про X.
- **Порядок и приоритет - нумерованным списком, не прозой.** Последовательность шагов / разрешение конфликта выразить нумерацией 1->2->3 с явным правилом («нижний не отменяет верхний», «при конфликте вердиктов приоритет у …»). Проза «сначала…, обычно…, при необходимости…» оставляет порядок на усмотрение исполнителя. Плохо: «проверь типы, можно также прогнать линтер». Правильно: «1. type-check. 2. lint. При конфликте приоритет у type-check».
- **Таблица вместо прозы**, когда это «вход -> инструмент/исход» (выбор инструмента, маппинг ситуация->действие).
- **Сухо.** Императив, не повествование. «Бери LSP», не «рекомендуется обратиться к LSP».
- **Языко/проекто-нейтрально**, если артефакт общий. Имена конкретного проекта (`UpCore`, `MergeRequest`) в общем правиле -> обобщить (`dst = src.Field` вместо `DTO = entity.Prop`).

## Информационная архитектура

Оптимизируется не только проза, но и где живёт текст. Носитель делит окно с историей и другими артефактами: что в нём, то платит токенами, пока он в контексте.

- **Тонкий носитель + деталь по требованию.** Что нужно всегда и для срабатывания - inline; что нужно редко или во взаимоисключающих ветках - в отдельный материал, читаемый по требованию.
- **Ссылки one-level-deep.** На доп-материал ссылаться прямо из носителя, не цепочкой «файл -> файл -> файл»: вложенные читаются частично и теряют контекст.
- **Длинный материал - с оглавлением сверху**, иначе при частичном чтении не виден его объём.
- **Высокоприоритетное нормативное - в начало носителя**, не в хвост за второстепенным.
- **Лимиты размера носителя и `description` не хардкодить** - дрейфуют; нормативный дом цифр - docs/SKILL_FRAMEWORK.md.

## Поле срабатывания (description)

Для артефакта, чьё обнаружение зависит от поля активации (skill, агент, команда): это поле - главный рычаг, по нему артефакт выбирается из множества; грузится всегда, не по триггеру.

- **Что + когда, в 3-м лице.** «Переписывает X, активируется при Y», не «я помогу» и не «ты можешь»: смешанное лицо ломает обнаружение.
- **Anchor + aspects, не entry-points.** Anchor - стек/тема (`ef core`, `DDD`); aspects - темы верхнего уровня (по любой попадает запрос в scope). Конкретный API/симптом - entry-point: матчится буквально и раздувает поле, не давая охвата.
- **Без повествования в зоне триггеров.** Внутри списка активаторов - только активаторы; преамбулу и исключения вынести из него.
- **Бюджет длины и минимум активаторов** - см. docs/SKILL_FRAMEWORK.md (детали anchor+aspects для skill там же, не дублировать).

## Обязательная сверка фактов

Любое утверждение о поведении инструмента/API/платформы (флаги, лимиты, какой бинарь установлен, поддержка поля) - **проверить перед фиксацией**, не писать по «общеизвестному»:

- наличие бинаря/сервера - `command -v` / PATH;
- поведение CLI - `--help` / живой прогон;
- API библиотеки - context7 -> WebSearch.

Нашёл неточность по ходу реза - **исправить** (это часть работы, не отдельная задача). Не смог сверить - пометить явным статусом (`unverifiable`), не выдавать догадку за факт и не молчать.

## Порядок работы

1. Прочитать артефакт **целиком** (текущее состояние, не по памяти - мог измениться).
2. Выписать техфакты к сверке -> проверить (см. выше).
3. Переписать по «что резать / что нельзя ослаблять / как писать / информационная архитектура / поле срабатывания».
4. После реза - `rg` по ключевым терминам удалённых кусков: не осталось ли **висячих ссылок** («см. раздел ниже» на удалённый раздел), битых перекрёстных ссылок.
5. Прогнать гейт по приоритетам на каждую правку (см. «Приоритеты при конфликте»), диф по смыслу, не только по объёму. Любой пункт гейта не выполнен -> откатить эту правку.
6. Проверить результат **эмпирически, не на глаз**:
   - дельта: строки/токены до->после (рез без числа - не рез);
   - re-derive: из переписанного текста выводятся те же обязательные действия, статусы и грабли, что из оригинала? Нет -> рез задел нормативную силу, откат;
   - правка `description`: trigger-check - активаторы срабатывают на реалистичных запросах в scope и НЕ срабатывают на близких вне scope (near-miss)?
