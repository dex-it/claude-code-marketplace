# Оптимизация markdown-инструкции под LLM

<!-- skill-type: process -->

Артефакт читает **модель**, не человек. Цель — донести механизм/условие/исход минимумом токенов,
без потери однозначности и нормативной силы. Грузится в каждую сессию / каждый вызов агента →
лишний токен стоит на каждом проходе.

## Приоритеты при конфликте

Строгий порядок. Нижний приоритет НИКОГДА не жертвует верхним; минимизация токенов уступает всегда.

1. **Нормативная сила и грабли.** Лазейки закрыты, обязательные статусы на месте, escape-условия узкие, границы инструмента сохранены. Непримиримо ни с чем ниже.
2. **Однозначность.** Каждая единица читается одним способом. Жертвовать ради неё можно только токенами, не нормативной силой.
3. **Минимум токенов.** Режется ТОЛЬКО то, что не задевает нормативную силу и однозначность.

Гейт на каждую правку (он же финальная проверка в конце работы): принять, только если ВСЕ пункты выполнены — смысл сохранён, однозначность не упала, нормативная сила и грабли на месте, escape-условия не шире прежних. Любой пункт не выполнен → откатить правку.

## Что резать

- **Повествовательную воду и вводные обороты.** «Стоит отметить, что», «как известно», «в целом»,
  «давайте рассмотрим» — выкинуть. Критерий: можно ли убрать слово/оборот без потери смысла или
  однозначности? Если да — убрать.
- **Историю и мотивацию установки.** «Раньше тут было», «было бы», «изначально», команды
  инсталляции (`uv tool install …`, `dotnet tool install …`) — модель применяет инструмент, а не
  переустанавливает. Оставить *как вызвать*, не *как поставить*.
- **Дубли.** Одна мысль/факт — один нормативный дом. Повтор в двух местах → ссылка или слияние.
  Две строки таблицы про родственные команды (`cfg`/`dfg`/`slice`) → одна.
- **Развёрнутые bash-примеры**, когда приём сохраняется в однострочнике без потери граблей и флагов.
  Несколько примеров одного приёма → один канонический, остальные через `·`-перечень.
- **Пересказ кода словами.** Если рядом код-блок/команда — словесное описание того же не нужно.

## Что нельзя ослаблять (приоритет «нормативная сила и грабли»)

- **Нормативную силу.** Сжимаем подачу, не правило. Лазейки, обязательные статусы (`n/a`+почему,
  `unverifiable`, `contradicted`), обоснования «почему mandatory» — не сокращать. Лаконичность ≠
  ослабление проверки.
- **Escape-условия — сужать.** В любой сущности (агент, команда, skill, CLAUDE.md) размытый escape —
  дефект: исполнитель сам решает, что «не нужно», и закрывает шаг молчанием. Сузить до конкретного
  условия + фиксации пропуска явным статусом; недоступность инструмента переключает на запасной, не
  отменяет шаг. Плохо: «пропусти, если не смог». Правильно: «пропусти ТОЛЬКО если X недоступен
  (`command -v X` пуст); зафиксируй статусом `n/a` + причина». Это не рез, а ужесточение.
- **Грабли и границы инструмента.** «C#: статики не видит → `LSP incomingCalls`», «`findReferences`
  per-symbol» — это и есть ценность, ради неё артефакт существует.
- **Однозначность.** Если сокращение породило двусмысленность
  («может ли исполнитель прочитать так, что правило к нему не относится?») — вернуть слово.

## Как писать после реза

- **Механизм + условие + исход**, не пересказ. «Ошибка X = причина Y» вместо абзаца про X.
- **Порядок и приоритет — нумерованным списком, не прозой.** Если артефакт задаёт
  последовательность шагов или разрешение конфликта, выразить нумерацией 1→2→3 с явным правилом
  («нижний не отменяет верхний», «при конфликте вердиктов приоритет у …»). Проза «сначала…,
  обычно…, при необходимости…» оставляет порядок на усмотрение исполнителя. Плохо: «проверь типы,
  можно также прогнать линтер». Правильно: «1. type-check. 2. lint. При конфликте — приоритет у
  type-check».
- **Таблица вместо прозы**, когда это «вход → инструмент/исход» (выбор инструмента, маппинг
  ситуация→действие).
- **Сухо.** Императив, не повествование. «Бери LSP», не «рекомендуется обратиться к LSP».
- **Языко/проекто-нейтрально**, если артефакт общий. Имена конкретного проекта (`UpCore`,
  `MergeRequest`) в общем правиле → обобщить (`dst = src.Field` вместо `DTO = entity.Prop`).

## Обязательная сверка фактов

Любое утверждение о поведении инструмента/API/платформы (флаги, лимиты, какой бинарь установлен,
поддержка поля) — **проверить перед фиксацией**, не писать по «общеизвестному»:

- наличие бинаря/сервера — `command -v` / PATH;
- поведение CLI — `--help` / живой прогон;
- API библиотеки — context7 → WebSearch.

Нашёл неточность по ходу реза — **исправить** (это часть работы, не отдельная задача). Не смог
сверить — пометить явным статусом (`unverifiable`), не выдавать догадку за факт и не молчать.

## Порядок работы

1. Прочитать артефакт **целиком** (текущее состояние, не по памяти — мог измениться).
2. Выписать техфакты к сверке → проверить (см. выше).
3. Переписать по «что резать / что нельзя ослаблять / как писать».
4. После реза — `rg` по ключевым терминам удалённых кусков: не осталось ли **висячих ссылок**
   («см. раздел ниже» на удалённый раздел), битых перекрёстных ссылок.
5. Прогнать гейт по приоритетам на каждую правку (см. «Приоритеты при конфликте»), диф по смыслу,
   не только по объёму. Любой пункт гейта не выполнен → откатить эту правку.
