# Tech Audit — ловушки и процедура

Стек-агностичный аудит: идея → подход → компонентная база → код → вердикт.

## Когда нужен tech-audit

- PR/RFC с существенными архитектурными изменениями
- Breaking changes в публичном API
- Миграция на новую технологию / библиотеку / паттерн
- Перед интеграцией внешнего компонента (due diligence)
- Запрос: "проведи аудит", "оцени подход", "какие риски"

## Ловушки аудита

| Ловушка | Проблема | Решение |
|---------|----------|---------|
| Аудит только кода | Пропущены проблемы уровня идеи/подхода | Начинай с "ЧТО и ЗАЧЕМ", не с diff |
| Нет матрицы "было→стало" | Изменения семантики невидимы в diff | Таблица сценариев: happy/error/edge |
| Нет альтернатив | Подход принят без сравнения | "Какие варианты отброшены и почему" |
| Компоненты = "используем X" | Лимиты X неизвестны | Проверь changelog, known issues, версию |
| Совместимость "по умолчанию" | Два механизма одного концерна конфликтуют | Проверь конфликты явно (2 TX, 2 кэша) |
| Вердикт без Impact | "Есть риск" без оценки | Вероятность × Impact × Митигация |
| APPROVE с открытыми вопросами | Замечания не блокирующие, но критичные | Используй CONCERN вместо APPROVE |
| Нет Out of Scope | Scope аудита размыт | Явно зафиксируй что не анализировал |

## Процедура: 5 фаз

### Phase 1 — Идея (ЧТО и ЗАЧЕМ)
- Какую проблему решаем? Реальна ли?
- Кто потребитель? Что будет если НЕ делать?
- Масштаб воздействия (сервисы, потребители)
- **Результат:** "Решаем [X] для [Y], потому что [Z]"

### Phase 2 — Подход (КАК и ПОЧЕМУ ТАК)
- Выбранный паттерн/архитектура/технология
- Альтернативы — почему отброшены
- Breaking changes — для кого
- Стратегия миграции / обратной совместимости

**Матрица "было → стало":**

| Сценарий | Было | Стало | Риск |
|----------|------|-------|------|
| Happy path | ... | ... | Нет |
| Ошибка | ... | ... | ? |
| Edge case | ... | ... | ? |

**Результат:** пометки "без изменений" / "улучшение" / "⚠️ изменение семантики" / "❌ регрессия".

### Phase 3 — Компонентная база (НА ЧЁМ СТРОИМ)

Для каждого ключевого компонента:

1. **Актуальная информация:** context7/nia-docs (MCP), changelog, исходники
2. **Лимиты:** поддерживает ли то, что от него требуют; known issues; не deprecated
3. **Совместимость:** конфликтов между компонентами нет

**Чеклист типичных ограничений:**

| Область | Что проверить |
|---------|---------------|
| DB / ORM | Провайдер, isolation levels, retry, savepoints |
| Async | Cancellation, deadlocks, thread safety, shared state |
| Serialization | Polymorphism, nullable, версионность контрактов |
| Network | Timeouts, retry, circuit breaker, idempotency |
| Security | Input validation, injection, auth boundaries |

**Результат:** "✅ подходит" / "⚠️ ограничение" / "❌ не поддерживает".

### Phase 4 — Код

Если есть PR/ветка/код — запустить **deep-audit** skill на изменённых файлах. Результаты включить в Phase 5.

Если кода ещё нет — пропустить.

### Phase 5 — Вердикт

```markdown
## Вердикт: [APPROVE / CONCERN / REJECT]

### Резюме
[1-2 предложения: что делает и главный вывод]

### Что хорошо
- [сильные стороны]

### Критические проблемы
| # | Проблема | Где | Impact | Рекомендация |

### Риски
| # | Риск | Вероятность | Impact | Митигация |

### Рекомендации
1. [Blocking — до мержа]
2. [Non-blocking — желательно]
3. [Задокументировать]
```

| Вердикт | Когда |
|---------|-------|
| **APPROVE** | Подход верный, реализация корректна, риски приемлемы |
| **CONCERN** | Подход верный, но нужны доработки |
| **REJECT** | Фундаментальные проблемы — пересмотреть |

## Адаптация глубины

| Контекст | Фазы |
|----------|------|
| PR с кодом | 1-2-3-4-5 |
| RFC без кода | 1-2-3-5 |
| "Оцени идею" | 1-2-5 |
| "Аудит компонента" | 3-4-5 (используй deep-audit) |

## Что НЕ входит в tech-audit

- Реализация решения
- Написание тестов
- Исправление найденных проблем (только анализ и рекомендации)
