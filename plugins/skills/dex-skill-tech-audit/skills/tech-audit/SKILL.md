# /tech-audit — Технический аудит решения

Полный аудит технического решения: идея → подход → компонентная база → реализация → вердикт. Стек-агностичный.

## Определение цели аудита

Аргумент `$ARGUMENTS` может быть:
- **PR номер/URL:** `#42` или URL — загрузить diff и description через `gh pr view`
- **Ветка:** `feature/new-auth` — `git diff main...$ARGUMENTS` для scope изменений
- **Файл/путь:** `docs/rfc-003.md` — прочитать как RFC/design document
- **Текст:** описание идеи — работать с тем что дал пользователь
- **Пусто:** спроси — "Что анализируем? PR, ветку, RFC или опиши идею."

## When to Use

- "Проанализируй это решение"
- "Оцени подход / архитектуру"
- "Какие риски у этого PR / RFC / proposal?"
- Ревью PR с существенными архитектурными изменениями
- Перед принятием решения о технологии / паттерне / миграции
- Due diligence перед интеграцией внешнего компонента

## Входные данные

Определяются автоматически из `$ARGUMENTS` (см. "Определение цели аудита" выше).

## Workflow

## Phase 1: Анализ идеи

**Цель:** Понять ЧТО и ЗАЧЕМ.

### Вопросы
- Какую проблему решаем? Реальна ли она?
- Кто потребитель? Как используется сейчас?
- Что будет если НЕ делать?
- Каков масштаб воздействия? (сколько сервисов, потребителей)

### Источники
- PR description / RFC
- Задача в трекере
- Уточняющие вопросы пользователю

### Результат
Формулировка: "Решаем проблему [X] для [Y], потому что [Z]."

---

## Phase 2: Анализ подхода

**Цель:** Понять КАК и ПОЧЕМУ ТАК.

### Вопросы
- Какой подход выбран? (паттерн, архитектура, технология)
- Какие альтернативы рассматривались? Почему отброшены?
- Что менялось в семантике? ("было A, стало B")
- Есть ли breaking changes? Для кого?
- Какова стратегия миграции / обратной совместимости?

### Матрица "было → стало"

Для каждого ключевого поведения:

```
| Сценарий              | Было                | Стало               | Риск   |
|-----------------------|---------------------|---------------------|--------|
| Happy path            | ...                 | ...                 | Нет    |
| Ошибка                | ...                 | ...                 | ?      |
| Edge case N           | ...                 | ...                 | ?      |
```

### Результат
Таблица сценариев с пометками: "без изменений" / "улучшение" / "⚠️ изменение семантики" / "❌ регрессия".

---

## Phase 3: Анализ компонентной базы

**Цель:** Понять НА ЧЁМ СТРОИМ и какие есть ограничения.

### Для каждого ключевого компонента / библиотеки / API:

1. **Загрузить актуальную информацию**
   - MCP-инструменты для документации библиотек (context7, nia-docs — если доступны)
   - WebSearch / WebFetch для changelog, known issues (если доступны)
   - Прочитать исходники (если внутренний компонент) — всегда работает

2. **Проверить лимиты и ограничения**
   - Поддерживает ли компонент то, что от него требуют?
   - Есть ли known issues / caveats для нашего use case?
   - Версия актуальна? Не deprecated?

3. **Проверить совместимость**
   - Компоненты совместимы между собой?
   - Нет ли конфликтов (например, два механизма транзакций одновременно)?

### Чеклист типичных ограничений

| Область | Что проверить |
|---------|---------------|
| DB / ORM | Поддержка провайдера, isolation levels, retry, savepoints |
| Async / Concurrency | Cancellation, deadlocks, thread safety, shared state |
| Serialization | Polymorphism, nullable, версионность контрактов |
| Network | Timeouts, retry, circuit breaker, idempotency |
| Security | Input validation, injection, auth boundaries |

### Результат
Список компонентов с "✅ подходит" / "⚠️ ограничение" / "❌ не поддерживает".

---

## Phase 4: Анализ кода

**Цель:** КАК написано, совпадает ли с заявленным подходом.

### Если есть PR / ветка / код — запустить `/deep-audit` на изменённых файлах.

Результаты deep-audit включить в итоговый вердикт (Phase 5).

### Если кода ещё нет — пропустить эту фазу.

---

## Phase 5: Вердикт

### Структура

```markdown
## Вердикт: [APPROVE / CONCERN / REJECT]

### Резюме
[1-2 предложения: что решение делает и главный вывод]

### Что хорошо
- [конкретные сильные стороны]

### Критические проблемы
| # | Проблема | Где | Impact | Рекомендация |
|---|----------|-----|--------|--------------|
| 1 | ...      | ... | ...    | ...          |

### Риски
| # | Риск | Вероятность | Impact | Митигация |
|---|------|-------------|--------|-----------|
| 1 | ...  | Высокая     | ...    | ...       |

### Рекомендации
1. [Что сделать перед мержем — blocking]
2. [Что желательно — non-blocking]
3. [Что задокументировать]
```

### Уровни вердикта

| Вердикт | Когда |
|---------|-------|
| **APPROVE** | Подход верный, реализация корректна, риски приемлемы |
| **CONCERN** | Подход верный, но есть проблемы в реализации или непокрытые риски — нужны доработки |
| **REJECT** | Подход имеет фундаментальные проблемы — нужно пересмотреть |

## Адаптация глубины

| Контекст | Фазы | Что пропустить |
|----------|-------|----------------|
| PR с кодом | 1-2-3-4-5 | Все фазы |
| RFC без кода | 1-2-3-5 | Phase 4 (код) |
| "Оцени идею" | 1-2-5 | Phase 3-4 |
| "Аудит компонента" | 3-4-5 | Phase 1-2 (используй `/deep-audit`) |

## Чего НЕ делает этот скилл

- Не реализует решение (для этого `/build`)
- Не пишет тесты (для этого `/tdd`)
- Не делает поверхностный code review PR (для этого `/review`)
- Не исследует кодовую базу (для этого `/explore`)
