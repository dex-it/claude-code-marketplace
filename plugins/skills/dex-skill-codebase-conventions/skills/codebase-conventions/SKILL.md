# Codebase conventions — ловушки и anti-patterns

Конвенция — произвольный выбор, который ценен только когда применён единообразно. У произвольного выбора нет правильного ответа в вакууме — есть только «как принято здесь». Это отличает её от технических best-practices (там есть объективная правильность).

При добавлении нового кода (класс, метод, файл) — сверься с соседними артефактами того же типа в проекте. 5 осей конвенций ниже + мета-граница «что копировать, а что нет».

## Ось 1: Форма представления одного смысла

### Один смысл — несколько форм; новый код выбрал не ту, что соседи
Плохо: у типа есть и типизированный конструктор `(double? from, double? to)`, и DSL-парсер `("[N:M]")` — выбран DSL, хотя соседние классы того же типа в проекте идут типизированным путём
Правильно: `grep` по имени типа в проекте — какой путь использует большинство; следовать той же форме или явно обосновать отступление в коде / MR
Почему: разные формы одного смысла увеличивают когнитивную нагрузку — читателю выяснять «почему здесь иначе». Строковые / DSL-формы переносят ошибки из compile-time в runtime. Конвенция теряет силу с первой копипастой строкового стиля «как у этого нового места»

### Возврат значения не в той форме, что соседи в слое
Плохо: соседние Service-методы возвращают `Result<T, Error>` для error-path; новый метод возвращает `T` и кидает исключения — или наоборот, везде throw, новый молча возвращает `null`
Правильно: посмотреть 2-3 соседних метода того же слоя — какая форма ошибки выбрана (throw vs Result vs nullable vs Option); следовать единой форме во всём слое
Почему: caller строит обработку ошибок под конвенцию слоя — `try/catch` если throw, pattern-match если Result, null-check если nullable. Один метод в другой форме ломает паттерн вызова, провоцирует пропуск ошибки (вернули `null`, никто не проверил) или дубли try/catch вокруг Result-метода

## Ось 2: Имя через назначение vs через реализацию

### Имя по реализации, хотя соседи именуются по назначению
Плохо: новый Response-DTO с полем `HasImplementation`, при этом соседние Response-DTO используют доменные термины (`HasOnboarding`, `IsActivated`); новый Handler `ProcessDataInLoopHandler`, соседи — `<Domain>CompletedHandler`
Правильно: `grep` по соседям той же роли (Response / Handler / Repository / Service) — на какой оси они именуются (назначение / доменное событие / техническое действие); следовать той же оси
Почему: соседи по слою задают ось нейминга в проекте. Имя на другой оси выпадает из ряда — читатель ожидает один словарь в одном слое. Имя по реализации устаревает при рефакторинге внутренней ветки, имя по назначению переживает изменения

### Тип исключения / возврата вне доменного словаря проекта
Плохо: в проекте есть `OrderNotFoundException`, `OrderInvalidStateException` в Domain.Exceptions — новый код кидает `InvalidOperationException("Order not in Draft state")` со строкой
Правильно: посмотреть существующие доменные исключения / типы ошибок проекта — использовать существующий или добавить рядом, по той же конвенции (наследование, namespace, naming)
Почему: тип исключения / тип ошибки — часть API слоя. Caller `catch (OrderInvalidStateException)` не сработает на `InvalidOperationException` со строкой; обработка ошибок в API-слое теряет специфичность. Доменный словарь работает только пока пронизывает все слои в bounded context

## Ось 3: Сквозная конвенция слоя

### Метод не следует общей конвенции сигнатуры слоя
Плохо: все async-методы Handler'ов слоя принимают `CancellationToken ct` последним параметром; новый Handler-метод объявлен без `CancellationToken` или с другим именем (`token`, `cancel`); соседи везде с суффиксом `Async`, новый — без
Правильно: посмотреть 2-3 соседних метода того же слоя — есть ли общая ось сигнатуры (CancellationToken, ILogger, корреляционный id, `IRequest<TResponse>`, суффикс `Async`); следовать ей
Почему: пайплайны (cancellation, tracing, retry) полагаются на однородность сигнатуры — один метод без `CancellationToken` ломает graceful shutdown цепочки. Разный нейминг параметра / суффикс ломает auto-complete и source generators. Однородность слоя — основа для всего pipeline-кода

### Артефакт не следует общему правилу слоя (структура / атрибуты / базовый класс)
Плохо: все Validator'ы в слое наследуют `AbstractValidator<T>` — новый ручной `IValidator`; все Controller'ы помечены `[ApiController][Authorize]`, новый без атрибутов; все DTO в слое — `record` с init-only, новый — `class` с public setters
Правильно: посмотреть 2-3 соседних артефакта того же типа на предмет общего базового класса, обязательных атрибутов, выбора record/class/struct; следовать единому правилу
Почему: общее правило слоя — это договор для инфраструктуры (auth-middleware ищет `[Authorize]`, validation-pipeline ищет `AbstractValidator`, маппер ищет `record`). Артефакт вне правила выпадает из инфраструктуры — без auth, без валидации, с нежелательной мутабельностью. Один из ряда обесценивает остальной ряд

## Ось 4: Расположение артефакта

### Файл положен не туда, где лежат соседи того же типа
Плохо: новый Handler в `Application/Services/`, при этом все Handler'ы проекта лежат в `Application/Handlers/<Feature>/`; новый Validator в одной папке с DTO, остальные — в `Validators/`
Правильно: перед созданием файла — `find` или дерево по ключевому суффиксу (`*Handler.cs`, `*Validator.cs`, `*Repository.cs`); положить рядом с соседями того же типа
Почему: разработчик ищет файлы по конвенции, не по `Find in files` каждый раз. Файл «не там» невидим при ревью соседних изменений, дублируется при следующем рефакторинге («не нашёл — создал заново»), ломает auto-discovery (DI-сканеры по namespace, MSBuild-globs, namespace = папка)

### Логика положена не в тот слой / класс, где живёт у соседей
Плохо: валидация запроса размазана между Controller, Handler и Validator — у соседей вся валидация в `*Validator` через pipeline behavior; маппинг entity → DTO разбросан в каждом Handler — у соседей через общий `*Profile` / extension `ToDto()`
Правильно: посмотреть, где у соседей живут одинаковые обязанности (валидация, маппинг, авторизация, логирование) — следовать тому же месту
Почему: одинаковая обязанность в одном месте — основа для cross-cutting изменений (поменять формат ошибки валидации, добавить поле в DTO). Размытое расположение требует править N мест, легко забыть одно. Параллельные стили — постоянный спор «куда писать новое»

## Ось 5: Способ интеграции с внешним ресурсом

### Способ работы с внешним ресурсом расходится с соседями
Плохо: HttpClient через `IHttpClientFactory` у соседей, новый код через `new HttpClient()`; соседи читают конфиг через `IOptions<T>`, новый — `IConfiguration["Key"]` напрямую; в слое везде `ILogger<T>`, новый файл — `static Log`
Правильно: посмотреть, как у соседей подключён тот же тип ресурса (HTTP / config / log / cache / persistence / messaging) — следовать тому же подходу
Почему: способ интеграции включает невидимые сбоку контракты — `IHttpClientFactory` даёт DNS-обновление и Polly-обёртку, `IOptions<T>` даёт hot-reload и валидацию, `ILogger<T>` даёт scope с категорией. Уход с конвенции отрезает новый код от этих контрактов: `new HttpClient()` течёт сокетами, прямой `IConfiguration` не валидируется

### Регистрация / конфигурирование сделаны не тем способом, что у соседей
Плохо: соседние options регистрируются `services.Configure<T>(config.GetSection("X"))`; новый — `services.AddSingleton(new T { ... })` с захардкоженными значениями; соседние сервисы регистрируются через extension `services.AddOrderModule()`, новый — отдельный `services.AddScoped<,>()` в `Program.cs`
Правильно: посмотреть, как соседи регистрируют тот же класс артефактов (options, сервисы, hosted services, decorators) — следовать единому стилю композиции
Почему: единый стиль регистрации даёт предсказуемые точки расширения — добавить декоратор / interceptor / валидатор на все нужные сервисы можно правкой одной extension'а. Разный стиль регистрации требует обходить все места поодиночке, легко пропустить новое. Hardcoded options теряют конфигурируемость по средам

## Ось 6: Граница «конвенция vs техническое решение»

### Скопирована «конвенция», которая на деле — техническое решение
Плохо: «здесь у соседа сервис зарегистрирован как Singleton — делаю так же» — без проверки, что Singleton корректен (нет mutable state, нет captive dependency, thread-safe); «у соседа throw, и я throw» — без проверки, нужна ли клиенту обработка через Result
Правильно: разделять две оси. Конвенции (произвольный выбор, валиден пока единый) — копируем у соседей: форма представления, нейминг, структура папок, сигнатура слоя, способ интеграции. Технические решения (валидность зависит от свойств кода) — обосновываем под новый код, не «как у соседей»: lifetime DI, throw vs Result для конкретного контракта, sync vs async, выбор exception type для семантики
Почему: «как у соседей» работает для конвенций — там нет «правильно» вне проекта, единство и есть ценность. Не работает для технических решений — там валидность зависит от свойств конкретного кода. Слепое копирование Singleton наследует captive dependency или race condition; копирование throw наследует утечку технического exception в API. Перед копированием — спросить «это произвольный выбор или решение под требования?»

## Чек-лист перед PR

- [ ] Новый артефакт — есть соседи того же типа? Если да — какие из 5 осей конвенции применимы?
- [ ] Форма представления (возврат / конструирование / иммутабельность) — как у соседей?
- [ ] Имя — на той же оси, что соседи в слое (назначение vs реализация vs техническое действие)?
- [ ] Сигнатура / атрибуты / базовый класс — следуют общему правилу слоя?
- [ ] Файл положен туда, где соседи того же суффикса; обязанности (валидация, маппинг, auth) живут в одном месте, как у соседей?
- [ ] Способ интеграции с внешним ресурсом (HTTP, config, log, cache) — тот же, что у соседей?
- [ ] При копировании технического решения у соседа (lifetime, exception, sync/async, throw vs Result) — проверил, что оно валидно для нового кода, а не только что «у соседа так же»?
- [ ] Если отступаю от конвенции — обоснование явно в коде (комментарий) или в описании MR?
