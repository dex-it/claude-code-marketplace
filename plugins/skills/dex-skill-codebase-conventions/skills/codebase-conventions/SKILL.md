# Codebase conventions — ловушки и anti-patterns

При добавлении нового кода (класс, метод, файл) сверься с соседними реализациями того же назначения. Если в проекте уже есть конвенция — следуй ей или явно обоснуй отступление в коде / MR. Конвенции, в отличие от технических best-practices, валидны только в рамках конкретного проекта.

## Конструирование объектов

### Один из нескольких путей конструирования выбран без оглядки на конвенцию проекта
Плохо: у типа есть и типизированный конструктор `(double? from, double? to)`, и DSL-парсер `("[N:M]")` — выбран DSL, хотя соседние классы того же типа в проекте используют типизированный путь
Правильно: посмотреть `grep` по имени типа в проекте — какой конструктор использует большинство соседних классов; следовать этой форме или явно обосновать отступление в коде / MR
Почему: разные пути конструирования одного типа в одном проекте увеличивают когнитивную нагрузку — читателю приходится выяснять «почему здесь иначе». DSL-формы переносят ошибки из compile-time в runtime (start-up failure вместо ошибки компиляции). Если конвенция проекта — типизированный путь, отступление без причины ломает консистентность и провоцирует копипасту строкового стиля в новые места

## Нейминг

### Класс / поле / метод назван по реализации, хотя соседи именуются по назначению
Плохо: новый Response-DTO имеет поле `HasImplementation`, при этом соседние Response-DTO в проекте используют доменные термины (`HasOnboarding`, `IsActivated`); новый Handler назван `ProcessDataInLoopHandler`, соседи — `<Domain>CompletedHandler`
Правильно: посмотреть `grep` по соседним классам той же роли (Response / Handler / Repository / Service) — по какому принципу они именуются; следовать той же оси (назначение vs реализация vs техническое действие)
Почему: соседи по слою задают ось нейминга в проекте — назначение, доменное событие, техническое действие. Новое имя на другой оси (по реализации, по технической детали) выпадает из ряда: читатель ожидает один словарь в одном слое, а получает смешение. Реализация со временем меняется — имя по реализации устаревает, имя по назначению переживает рефакторинг

## Структура файлов и папок

### Новый файл положен не туда, где лежат соседи того же типа
Плохо: новый Handler положен в `Application/Services/`, при этом все остальные Handler'ы в проекте лежат в `Application/Handlers/<Feature>/`
Правильно: перед созданием файла — `find` или дерево `Application/` по ключевому суффиксу (`*Handler.cs`, `*Repository.cs`, `*Validator.cs`); положить рядом с соседями того же типа
Почему: разработчик ищет файлы по конвенции проекта, не по `Find in files` каждый раз. Файл «не там» — невидим при ревью соседних изменений, дублируется при следующем рефакторинге («не нашёл — создал заново»), ломает auto-discovery (DI-сканеры по namespace, MSBuild-globs)

## Сигнатуры методов в одном слое

### Новый метод не следует общей конвенции сигнатуры слоя
Плохо: все async-методы Handler'ов в проекте принимают `CancellationToken ct` последним параметром; новый Handler-метод объявлен без `CancellationToken` или с другим именем (`token`, `cancel`)
Правильно: посмотреть 2-3 соседних метода того же слоя — есть ли общая ось сигнатуры (CancellationToken, ILogger, корреляционный id, `IRequest<TResponse>`); следовать ей или явно объяснить отступление
Почему: пайплайны (cancellation, tracing, retry) полагаются на однородность сигнатуры — один метод без `CancellationToken` ломает graceful shutdown для всей цепочки. Разный нейминг (`ct` / `token` / `cancellationToken`) увеличивает шум при чтении и мешает auto-complete. Однородные сигнатуры одного слоя — основа для генераторов (T4, source generators, openapi codegen)

## Граница «своя ошибка vs чужая»

### Применил конвенцию соседа без проверки, что сосед сам не ошибается
Плохо: «здесь у соседа сервис зарегистрирован как Singleton, делаю так же» — без проверки, что Singleton корректен (отсутствие mutable state, thread-safety, отсутствие captive dependencies)
Правильно: разделять две оси: технические решения (lifetime DI, throw vs return Result, sync vs async, exception type) — обосновывать по требованиям нового кода, а не «как у соседей». Конвенции (форма конструирования, нейминг, структура папок, сигнатура слоя) — следовать соседям
Почему: «как у соседей» работает для конвенций (произвольный выбор, валидный пока единый), но **не работает** для технических решений (валидность зависит от свойств конкретного кода). Слепое копирование Singleton может унаследовать captive dependency или race condition. Перед копированием технического решения — проверить применимость к новому коду, а не только факт «у соседа так же»

## Чек-лист перед PR

- [ ] Новый класс / тип — есть ли соседние того же назначения? Если да — следую той же форме конструирования / нейминга / структуры?
- [ ] Если отступаю от конвенции — обоснование явно в коде (комментарий) или в описании MR?
- [ ] Перед добавлением нового варианта API того же типа — есть ли уже типизированный путь?
- [ ] Имя нового класса / поля / метода — на той же оси, что соседи в этом слое (назначение vs реализация)?
- [ ] Файл положен туда, где лежат соседи того же суффикса (`*Handler.cs`, `*Repository.cs`)?
- [ ] Сигнатура нового метода однородна с соседями слоя (CancellationToken, logger, корреляционный id)?
- [ ] При копировании технического решения у соседа (lifetime, exception, sync/async) — проверил, что оно валидно для нового кода, а не только что «у соседа так же»?
