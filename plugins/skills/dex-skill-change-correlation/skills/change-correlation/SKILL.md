# Корреляция изменений - ловушки и anti-patterns

## Что и когда менялось

### Образ на стенде не сверен с веткой
Плохо: расследование идёт по коду ветки, хотя на стенде развёрнут другой sha / тег
Правильно: сверить sha / тег запущенного образа с tip ветки до выводов
Почему: расхождение версии стенда и ветки - само по себе зацепка; анализ по коду, которого нет на стенде, ведёт к ложным выводам

### Окно корреляции не привязано к моменту начала
Плохо: смотрятся "последние коммиты" без связи с timestamp первого симптома
Правильно: брать изменения в окне вокруг зафиксированного WHEN (момента начала из спецификации)
Почему: без якоря времени в выборку попадает шум - изменения до и сильно после симптома, причина теряется среди нерелевантных

### Учтён только код, не конфиг и флаги
Плохо: коррелируются только коммиты кода, игнорируются деплои, фиче-флаги, конфиг, версии зависимостей, миграции
Правильно: в окне проверить весь набор изменений - код, конфиг, флаги, образы, миграции, обновления зависимостей
Почему: симптом без изменения кода часто вносится конфигом или флагом; узкий взгляд только на git пропускает реальный триггер

## Тайминг и изоляция

### Изменение после начала симптома объявлено причиной
Плохо: на инцидент повешен мердж, приехавший на стенд позже первого проявления симптома
Правильно: отбрасывать кандидатов, чьё время выката позже момента начала симптома
Почему: причина не может появиться после следствия; изменение, не успевшее на стенд к моменту симптома, исключается по таймингу

### Несколько изменений в окне без изоляции
Плохо: в окне три мерджа, причина повешена на один без отсева остальных
Правильно: изолировать кандидатов по близости к зоне (IS) и таймингу, проверять по одному
Почему: при нескольких одновременных изменениях совпадение по времени не указывает виновника; без изоляции фикс уходит не туда

## Инструменты корреляции

### blame на строку без проверки исполнения
Плохо: `git blame` показал недавнее изменение строки - оно объявлено причиной, хотя строка не на пути симптома
Правильно: подтвердить, что изменённая строка реально исполняется в сценарии симптома
Почему: недавнее изменение строки не значит, что именно оно ломает - строка может быть мёртвой для этого пути

### bisect по всему репо без first-parent
Плохо: `git bisect` без `--first-parent` и без scope блуждает по веткам-кустам и нерелевантным каталогам
Правильно: `git bisect` с `--first-parent` и scope на каталог затронутого проекта
Почему: без first-parent bisect заходит в merge-кусты и шумит; без scope тратит шаги на изменения вне зоны симптома

## Чек-лист корреляции

- Sha / тег образа на стенде сверены с tip ветки
- Окно изменений привязано к моменту начала симптома
- Проверены не только коммиты: конфиг, флаги, миграции, зависимости
- Кандидаты позже момента начала отброшены по таймингу
- При нескольких изменениях проведена изоляция, а не выбор первого
- bisect (если применим) - с first-parent и scope на зону
