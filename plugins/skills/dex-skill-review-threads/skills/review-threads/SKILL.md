# Review Thread Delivery

Ловушки доставки ревью в UI хостинга. Находки полезны только если автор может работать с ними построчно: видеть приоритет, прыгать к коду, отмечать сделанное. Один длинный комментарий со списком «критичные / важные» этого не даёт. Правило: один тред равен одной находке, привязанной к конкретной строке нативным механизмом платформы.

## Гранулярность тредов

### Несколько находок в одном треде

Неправильно: связка разных проблем (security плюс нейминг плюс тест) в одном комментарии.
Правильно: один тред на одну находку; связанные находки на одной строке допустимо в одном треде с явным перечислением пунктов.
Почему: тред это единица, которую отмечают как resolved; смешав несколько проблем, автор не может закрыть одну, не закрыв все, и часть теряется.

### Один гигантский комментарий вместо тредов

Неправильно: всё ревью опубликовано одним общим комментарием с разделами по severity.
Правильно: каждая находка отдельным inline-тредом; общий комментарий допустим только как короткий overview.
Почему: общий комментарий не привязан к строкам и не резолвится по пунктам, поэтому автор не понимает, что из него уже сделано, а что нет.

## Привязка к коду

### Привязка через текст «см. file:line»

Неправильно: тело комментария содержит «смотри File.cs строка 42» без нативной inline-привязки.
Правильно: создать тред нативным API с position на нужную строку (gh api на pulls/comments с path и line, glab api на discussions с position); тело без текстовой ссылки.
Почему: текстовая ссылка не кликабельна и устаревает при сдвиге строк; нативная привязка следует за кодом и открывает дифф в нужном месте.

### Тред привязан к неверной ревизии

Неправильно: комментарий создаётся без актуальных SHA-якорей диффа, попадает на устаревшую версию.
Правильно: взять актуальные diff_refs (base/head/start sha для glab, head commit для gh) и привязать тред к ним.
Почему: привязка к старой ревизии вешает комментарий на строку, которой уже нет, и автор видит замечание не там, где проблема.

## Содержание

### Severity не в первой строке

Неправильно: уровень критичности упомянут где-то в середине текста или отсутствует.
Правильно: первая строка тела треда содержит severity, затем суть в 2-4 предложения и конкретный fix.
Почему: автор сканирует треды быстро; без severity в начале он не отличает блокер от мелочи и тратит время не на то.

### Overview дублирует содержимое тредов

Неправильно: общий комментарий пересказывает каждую находку, которая уже есть в треде.
Правильно: overview только считает (N блокеров, M важных) и отсылает к тредам; детали живут в тредах.
Почему: дублирование заставляет читать одно и то же дважды и расходится при правке тредов, создавая два разных текста об одной находке.

## Жизненный цикл

### Resolve без протокола

Неправильно: тред закрывают по факту коммита, без явного действия resolve и без критерия «что считается закрытым».
Правильно: resolve выполняется отдельным вызовом API после публикации ответа; в ответе указано, чем именно закрыто (sha фикса или причина снятия).
Почему: коммит сам по себе не закрывает тред в UI; без явного resolve тред висит открытым, а без причины в ответе непонятно, исправлено или отклонено.

### Чужой тред тронут без основания

Неправильно: reply, resolve или unresolve в треде, созданном другим ревьюером.
Правильно: трогать только свои треды; чужие не резолвить и не переоткрывать без явной команды.
Почему: чужой resolve стирает контекст обсуждения и решение за другого человека, ломая прозрачность ревью для команды.

### Запись в MR до подтверждения

Неправильно: треды публикуются сразу, без показа плана пользователю.
Правильно: сначала показать план тредов (file:line, severity, заголовок), дождаться подтверждения, затем по одному вызову API на тред.
Почему: публикация в чужой MR необратима в глазах автора и команды; план до записи ловит ошибочные и грубые формулировки до того, как их увидят.
