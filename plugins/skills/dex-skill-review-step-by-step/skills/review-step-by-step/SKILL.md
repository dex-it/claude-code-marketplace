# Review Step-by-Step — пошаговая обработка списков правок

## Когда применять

Есть список правок (ревью MR/PR, security-аудит, чек-лист рефакторинга, бэклог багов), и **по каждому пункту сходу не ясно, что с ним делать** — нужно сначала прочитать код, проверить утверждение ревьюера, обсудить с пользователем (правим / выносим в followup / опровергаем / пропускаем). Цель — пройти пошагово, по каждому пункту принять осознанное решение, а не «прокатить пачкой».

Не применяется при явной команде пользователя «делай батчем» / «без согласования» / «пункты X-Y-Z без апрува».

## Процедура

1. **TaskCreate сразу** на все пункты в порядке источника.
2. **Скоп НЕ фиксируется заранее.** Решение «делаем / followup / не делаем» принимается по каждому пункту в ходе разбора.
3. **Цикл по пункту:** погружение в код (Read, не grep) → формулировка решения → СТОП → команда пользователя → действие → реплай в discussion → следующий пункт.
4. **Ветки решений:** A=делаем (commit + реплай) / B=уже исправлено (реплай без апрува) / C=followup (тикет по команде) / D=пропускаем (реплай «не в этом MR») / E=опровержение (объясняем, ждём согласия).
5. **Финал:** таблица «Адресовано в ревью» в описании MR, push по команде, tracker transition задачи.

## Ловушки

### Скоп зафиксирован заранее

Плохо: разложил все 12 пунктов на «делаем / followup / забей» до разбора
Правильно: классификация по каждому пункту после погружения и команды пользователя
Почему: до чтения кода неизвестно, реальный это баг или ложное срабатывание ревьюера. Преждевременная сортировка → лишняя работа или пропущенные находки

### Батч-обработка похожих пунктов

Плохо: «эти 3 пункта про nullability — поправлю разом, они одинаковые»
Правильно: один пункт = один цикл апрув → действие → следующий
Почему: «похожие» пункты при детальном чтении часто оказываются разными. В батче проходит ошибочная правка, контроль за интерпретацией теряется

### Коммит без явного апрува

Плохо: сформулировал решение → сразу edit + commit
Правильно: после формулировки СТОП, ждать `+` / `делаем` / `А` / выбор ветки
Почему: апрув формулировки = разрешение коммитить. Без апрува код едет в репу мимо контроля, откат = лишний коммит в истории

### Догадка по grep вместо Read

Плохо: `grep -r "MyClass"` → «функция есть, ревьюер прав»
Правильно: Read нужного файла, найти определение, проверить behavior
Почему: grep цепляет комментарии, dead code, namespace alias. Утверждение «X существует / X делает Y» = ✓ VERIFIED только после Read

### Followup-тикет без команды пользователя

Плохо: «пункт большой / тянет в сторону / явно отдельная задача» → создал тикет
Правильно: агент **предлагает** вынести («это тянет в сторону, заводить followup?»), тикет создаётся только по явной команде («создай задачу» / «followup»)
Почему: followup = расширение скопа за пределы MR. Что для агента «явно отдельная задача» — для пользователя может быть «доделай в этом же MR». Решение по скопу принимает пользователь, агент имеет право только предложить

### Молчание принято за апрув

Плохо: сформулировал решение → нет ответа → пошёл коммитить
Правильно: молчание = переспросить «делаем?», не угадывать
Почему: пользователь мог отвлечься или формулирует возражение. Действие без явного сигнала → откат

### Tracker-префикс хардкодом

Плохо: `git commit -m "fix: ... SUP-123"` без проверки префикса в проекте
Правильно: префикс из истории коммитов / веток / CLAUDE.md. Не нашёл — спросить
Почему: проекты используют разные трекеры (Jira, Linear, YouTrack, GitLab issues). Чужой префикс = битая ссылка, ревьюер не найдёт тикет

### Реплай в discussion забыт

Плохо: правка закоммичена, но в обсуждении MR — тишина
Правильно: на каждый пункт реплай: «закрыто коммитом X / вынесено в TICKET-Y / не делаем потому что Z»
Почему: ревьюер не отслеживает коммиты вручную. Без реплая он не знает, что пункт обработан → повторное ревью того же

### Опровержение без согласования

Плохо: «ревьюер не прав» → пропустил пункт, реплай «не баг»
Правильно: объяснить почему конкретным кодом, СТОП, дождаться согласия пользователя или ревьюера
Почему: опровержение = конфликт интерпретаций. Без согласия второй стороны → тлеющее разногласие, всплывёт на повторном ревью

### Комментарии в коде про прошлое

Плохо: `// раньше тут было foo, теперь bar потому что ревьюер сказал`
Правильно: комментарий про WHY текущего состояния, без отсылок к истории
Почему: «раньше было X» — это git blame, не комментарий. Через 3 месяца «раньше» бессмысленно, комментарий устарел

### Push без команды

Плохо: коммит → автоматически `git push`
Правильно: push только по команде пользователя или после закрытия всех пунктов
Почему: push = публичное действие, видно ревьюеру. Промежуточный push с половиной правок = шум, ревьюер начнёт ревьюить недоделанное

### Tracker transition забыт

Плохо: все пункты закрыты, MR обновлён, задача в трекере висит в «In Progress»
Правильно: после финального push двинуть задачу на «На ревью» / следующий статус
Почему: PM/lead отслеживают статус по трекеру, не по MR. Несдвинутая задача = считается заблокированной

## Сигналы пользователя

| Сигнал | Ветка |
|---|---|
| `+` / `делаем` / `А` / апрув формулировки | A — edit + commit + реплай |
| «уже исправлено» (правило сессии) | B — реплай без коммита |
| «создай задачу» / «followup» | C — тикет + реплай со ссылкой |
| «пропускаем» / «забей» / «не в этом MR» | D — реплай «не делаем» + причина |
| опровержение пункта | E — объясняем, СТОП, ждём согласия |
| молчание после формулировки | переспросить, НЕ действовать |
| «делай без согласования пункты X, Y, Z» | батч только указанных |
