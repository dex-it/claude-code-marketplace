# Product Discovery — ловушки

## Интервью

### Наводящий вопрос о фиче
Плохо: "Вам бы понравилась функция X?" — confirmation bias, люди говорят "да"
Правильно: "Расскажите о последнем разе, когда вы [задача]. Что было сложнее всего?"
Почему: гипотетическое одобрение не равно реальному использованию. Люди не хотят обидеть

### Гипотетическое намерение платить
Плохо: "Вы бы заплатили $10/мес за это?" — фантазия, не поведение
Правильно: "Сколько вы сейчас тратите на решение этой проблемы?"
Почему: готовность платить в теории и реальная оплата различаются в 10x. Спрашивай про текущие затраты

### Интервью с друзьями/коллегами
Плохо: опрашивать знакомых — Mom test: они скажут что вы хотите услышать
Правильно: интервьюировать незнакомых из целевого сегмента
Почему: friends & family дают социально желательные ответы, искажение данных гарантировано

### Закрытые вопросы (да/нет)
Плохо: "Вам удобно пользоваться текущим решением?" — ответ да или нет
Правильно: открытые: "Расскажите...", "Как вы...", "Что было самым сложным?"
Почему: закрытые вопросы не раскрывают контекст и мотивацию, упускаешь инсайты

### Мало интервью для выводов
Плохо: 2-3 интервью и "мы поняли проблему"
Правильно: 5-10 интервью для problem discovery, 5-8 для solution testing
Почему: малая выборка = confirmation bias, один яркий респондент искажает картину

## Валидация

### Прыжок от интервью к full build
Плохо: 5 интервью -> "людям нужно" -> 6 месяцев разработки
Правильно: поэтапно: interview -> landing/fake door -> concierge MVP -> beta + оплата
Почему: каждый уровень дешевле предыдущего. Прыжок через уровни = риск построить ненужное

### "Просто построим"
Плохо: skip validation, начать разработку на основе гипотезы без проверки
Правильно: Assumption -> Hypothesis -> Experiment -> Learn, потом build
Почему: 70% фичей не используются после запуска. Validation дешевле разработки в 100x

### Нет формата гипотезы
Плохо: "Нам нужна CRM для менеджеров" — нет проблемы, нет данных
Правильно: "We believe [X] will [Y] because [Z]" — falsifiable hypothesis
Почему: без формата гипотезы невозможно спроектировать эксперимент и определить успех/провал

## Красные флаги

### "Всем это нужно"
Плохо: нет сегментации, нет данных, "все пользователи хотят"
Правильно: конкретный сегмент с quantified problem: "[Junior devs] тратят [50% времени] на [X]"
Почему: продукт для всех = продукт для никого. Без сегмента невозможно приоритизировать

### "У нас нет конкурентов"
Плохо: не искали или игнорируют substitutes (Excel, email, ручной процесс)
Правильно: анализ конкурентов включая substitutes и workarounds
Почему: конкуренция есть всегда. Текущий workaround пользователя = ваш главный конкурент

### Solution-first thinking
Плохо: список фичей без описания проблемы, "давайте добавим AI/blockchain"
Правильно: Problem Statement с quantified impact: кто + сколько + проблема + последствия
Почему: решение без проблемы = технология ради технологии. Problem first, solution second

## Чек-лист

- Проблема validated (10+ интервью, не гипотеза)
- Открытые вопросы, не наводящие
- Интервьюируемые из целевого сегмента (не друзья)
- Problem statement с quantified impact
- Hypothesis format: "We believe [X] will [Y] because [Z]"
- Валидация поэтапная (interview -> fake door -> MVP -> beta)
- Competitors analyzed (включая substitutes)
