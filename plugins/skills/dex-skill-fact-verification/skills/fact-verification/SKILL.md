# Fact Verification

Сверка технического утверждения с источником истины, прежде чем оно попадёт в вывод как факт. Применимо везде, где вердикт держится на поведении фреймворка или библиотеки: находка ревью, корневая причина в диагностике, дефолт в аудите конфигурации. Чтение кода подтверждает, что путь применяется в репозитории, но не истинность утверждения о самом фреймворке. Источник истины — документация версии, используемой в проекте: context7 по библиотеке, при недоступности MCP или для общеязыкового факта — WebSearch/WebFetch первоисточника (docs, changelog релиза). Уход от сверки фиксируется явным статусом (`unverifiable` / пометка-предположение / снятие утверждения), не молчанием.

## Сверка с источником истины

### Имя API или сигнатура взяты по памяти

Неправильно: утверждение называет метод, класс, опцию или атрибут фреймворка, имя которого вспомнилось, но не сверено с документацией версии проекта.
Правильно: сверить имя и сигнатуру с официальным источником — context7 по библиотеке, при недоступности или для общеязыкового факта WebSearch; в вывод попадает только подтверждённое имя.
Почему: генеративная модель закономерно выдумывает правдоподобные, но несуществующие имена (например `EvictByTagAsync` вместо реального `RemoveByTagAsync`); утверждение со ссылкой на несуществующий API справедливо отвергается, и доверие к выводу падает.

### Причинно-следственная связь не проверена

Неправильно: объяснение, почему это баг или причина, опирается на дефолт или поведение фреймворка по памяти («без X молча игнорируется», «по умолчанию выключено», «бросает исключение»), и не сверено.
Правильно: каждое утверждение «по умолчанию / бросает / всегда / никогда / O(...)» сверить с источником; причинность проверять отдельно, даже когда само утверждение верно.
Почему: перевёрнутая причинность коварнее выдуманного имени — имя грепается, причину ловит только дока; неверное обоснование дискредитирует и верный вывод (типичный случай — «лимит без размера молча бесполезен», тогда как на деле это исключение в runtime).

### Технический факт принят на веру из MR, кода или лога

Неправильно: техническое утверждение из описания MR, коммита, комментария в коде или строки лога («это потокобезопасно», «кэш делает single-flight», «опция включена по умолчанию») встроено в вердикт как данность.
Правильно: техническое утверждение, на котором держится вердикт, сверяется с источником независимо от того, кто его произнёс — автор изменения, лог стенда или сам исполнитель.
Почему: вывод наследует чужую ошибку, если принимает технические допущения без проверки; источник истины — документация версии, а не формулировка в MR или сообщение в логе.

### Сверка против latest вместо версии проекта

Неправильно: факт сверён с источником, но с документацией последней версии библиотеки, тогда как в проекте закреплена другая (по lock-файлу/манифесту); context7 без указания версии и верхний результат поиска по умолчанию отдают latest.
Правильно: определить версию из манифеста проекта и сверять именно с её документацией — запросить context7 с версией, в поиске искать changelog нужного релиза; разойдясь во времени, поведение latest к версии проекта не относится.
Почему: дефолты и сигнатуры меняются между версиями (поведение, включённое по умолчанию в новом релизе, в закреплённой версии выключено); сверка с latest даёт ложное «подтверждение» факта, которого в версии проекта нет.

### Отрицательный факт «такого API нет» не доказан

Неправильно: вывод утверждает, что метода, опции или перегрузки не существует, на основании того, что он не вспомнился или не нашёлся беглым взглядом.
Правильно: отрицательное утверждение об API доказывается так же, как положительное — сверкой с документацией версии; «нет в источнике» подтверждается источником, а не отсутствием в памяти.
Почему: модель забывает реально существующие API так же закономерно, как выдумывает несуществующие; недоказанное «этого нет» порождает находку-фантом наравне с выдуманным именем и так же роняет доверие.
